# 概述

本文档仅描述集成测试控制系统与各二级子系统之间的接口设计，即通信协议。

本文档的预期读者是集成测试控制系统与各子系统的软件开发人员以及联调和测试人员。

### 文件目录和文件命名

#### 文件目录

运行控制中心与二级子系统间文件传输目录结构为：基础目录/根目录/主备标识/信源标识_TO_信宿标识/系统状态模式/生成时间。

目录结构中各参数和符号说明及相关要求如下：

1)  基础目录：**基础目录统一暂定为"CSS-IIS-FILE"，与信息交互发送文件就绪通知主题一致。**

2)  根目录:"fileroot"；

3)  主备标识：Primary（主用标识），BackUp（备用标识）

4)  信源、信宿：参见附录中信源信宿地址编码及文件交互标识约定表。

5)  系统状态模式：OP；

6)  生成时间：yyyyMMdd

#### 文件名称

文件名称由文件类型_信源标识_TO_信宿标识_时刻_编号.扩展名组成。

各参数和符号说明及相关要求如下：

1)  文件类型：文件类型标识，参见2.3.1；

2)  信源、信宿：参见附录中信源信宿地址编码及文件交互标识约定表。

3)  时刻：按照24小时制，包括时、分、秒三部分，分别用2位阿伯数字表示。（星网时）

4)  编号:用4字符表示，表示当天生成的某一类文件的序列流水号。用4个字符标识，范围为
    0001-9999。

5)  扩展名：文件上传过程中以tmp文件后缀标识，上传结束后修改为正式的文件扩展名。

6)  文件名称中各参数间用字符"\_" （ASCII码）分隔。

7)  文件名称总长度（含扩展名及所有分隔符）不超过64 个字符。

\[示例\]根据上述命名规则，雄安站的Ka子系统文件交互标识为GSN-XA-KPS，生成信令文件时间为
2022 年10月 10日上午9点15分20秒。目录结构定义为：

基础目录/fileroot/Primary/GSN-XA-KPS_TO_SOCC-CQ-CCS/OP /20221010/
CNSig_GSN-XA-KPS_TO_SOCC-CQ-CCS_091520_0001.json

#### 上传文件内容格式

##### 终端信令文件

> 由二级子系统上报，文件类型为csv。

按照AS/NAS两个层级进行分类。AS层按照物理层、MAC层、RLC层等进行细分。NAS层按照终端接入认证、会话控制、数据、语音、物联网业务等类别进行细分。在信令文件生成时，需对记录的信令进行标记，如提供明确的正常或异常的特征信息。

具体信息包括：时间点(CollectTime)（精确毫秒），终端ID（STId）、载荷ID（PayloadId），
方向（Direction）（入/出），信令类型（SigType）、信令数据标识（SigId），信令数据（SigData），信令状态（SigStat）。

CollectTime: 产生时间, 如20230909090909999，
即2023年9月9日9点9分9秒，999毫秒

STId: 终端标识，基站分配终端的id, 如ST001

STImsi: 终端Imsi， 如460002020045579

PayloadId: 载荷标识，如Sgnb001

Dirction： 信令进出载荷标志，取值DL(Downlink)或UL(Uplink)

SigType:信令类型，如NAS/PDCP/RLC/RRM等

Source：信令发送源，如UE/gNB

Destination：信令发送目的，如UE/gNB

SigID: 信令数据标识，如\" PDU-Session-Establish\"

SigData:信令数据， 如以十六进制标识的码流

SigFData:格式化信令数据， 建议asn.1编码格式提供

SigStat: 信令状态， 例如S001表示信令参数不正确等。

##### 日志文件

由二级子系统系统上报。

运行日志文件主要包括网元运行中产生的运行信息记录数据，一般只用于人员阅读，程序不进行解析，具体内容根据二级子系统需要自定义。
建议内容为：\<时间 等级 日志内容\>。

##### 性能数据文件

由二级子系统系统上报。

性能数据文件主要包括网元性能数据，如KPI等。
建议内容为：\<网元ID，性能项类别、性能项、值，采样时间\>

样例（CSV格式）：

  ------------------------------------------------------------------------
  CollectTime      NEId          UserAccessSuccessRate          \...
  ---------------- ------------- ------------------------------ ----------
  202309090909     AMF001        100                            \...

  ------------------------------------------------------------------------

##### 测试数据文件

测试数据文件主要包括网元测试过程产生的数据。 建议内容为：

\<网元ID，数据项类别、数据项、值，采样时间\>

样例：

{

\"NEId\": \"AMF001\", // 网元标识

\"DatType\": \"CP-Data\", // 例如控制面数据

\"DatItem\":\"D001\", // 数据项

\"DatVal\": :\"678\", // 测试数据取值

\"CollectTime\": \"202308080909\", // 数据采集时间

}

##### 测试报告文件

\<二级子系统类型,二级子系统id，任务id、任务开始时间、任务结束时间、测试任务执行结果状态，测试任务结果信息，每个用例测试结果信息\<测试用例id,卫星id,任务开始时间、任务结束时间、测试用例结果、其他附加信息（可选）\>

# 通用消息

## 消息约定

1)  消息类型，分为请求和响应两种类型，且一一对应，即每个消息下发后对应系统都要返回响应消息，确保消息发送成功；

2)  消息名称，响应方法名称命名规则为在其对应的请求消息名后加Response，如，请求消息为：getDeviceList，响应消息名称为：getDeviceListResponse

3)  二级子系统id，唯一标识一个二级子系统，类型为字符串，**由星网统一规划提供**，建议命名规则："子系统类型_3位数字"，如："ADS_001"，二级子系统所有接口消息中必须携带此id。

4)  设备id，唯一标识一个设备，由各个子系统提供，同一个子系统内不能重复，取值范围：不大于50个字符。建议格式：
    子系统标识_设备类型_编号,如："ADS_UE_01"

5)  设备类型定义参照"附录章节"的"设备类型"章节。

6)  消息结构中的"method"属性必须全子系统唯一，否则中心控制系统无法识别消息。因此，通用消息结构各系统必统一，不能修改。

7)  二级子系统自定义消息的"method"属性为避免重复，命名规则为："二级子系统类型_消息名称"，如："ADS_setPars"

8)  用例id，建议命名规则：二级子系统标志_用例英文名\_
    3位数字，长度不超过50字符，其中"用例英文名"要使用驼峰式命名规则，且不能包含"\_"或"-"等特殊字符，如：Ka子系统的"UDP灌包"用例，用例id为："KPS_UdpFill_001"。

9)  sessionId：标识一个事务，同一事务内所有消息的sessionId取值相同，sessionId由发起方产生。

10) 载荷信息从星座管理系统获取。

11) 在一个测试任务中具体要向集成控制系统上报哪些文件（比如信令、日志等），由二级子系统根据用例决定，不是由集成控制系统决定。

## 用例脚本编码约定

各二级上传给集成控制系统的用例脚本文件必须可以用ＶＳＣｏｄｅ编辑修改，建议使用python或xml。

## 通用接口定义
