# 主机软件遥控遥测帧结构说明

## 1. 文档目的

本文档用于给主机软件人员说明当前工程中遥控下发和遥测接收的公共帧结构、字段位宽、组帧规则和查表方法。

主机软件在配置遥控指令和解析遥测状态时，应同时参考：

- `工程部署/工程实现/主机控制与状态协议层/遥控指令清单/`
- `工程部署/工程实现/主机控制与状态协议层/遥测指令清单/`
- `工程部署/工程实现/控制与状态模块/应用层协议头与消息语义说明.md`
- `工程部署/工程实现/控制与状态模块/应用层模块ID与遥测组说明.md`

本文只说明主机软件对接所需的公共协议结构，不修改现有 RTL，不新增协议字段。

## 2. 总体过程

### 2.1 遥控下发过程

主机软件发送遥控命令时，先生成应用层 `payload`，再套 RS422 固定帧。

整体过程如下：

```text
主机软件生成应用层 payload
-> 套 RS422 固定帧
-> 经 RS422 链路发送给 FPGA
-> FPGA RS422 接收层验帧和校验
-> 合法帧的 payload 进入控制与状态协议层
-> 控制与状态协议层按 module_id 分发给目标模块
-> 目标模块按 group_id 和参数表解析参数
-> 目标模块本地执行配置写入、脉冲触发或其他动作
```

当前协议不提供即时确认帧，也不提供即时结果帧。主机侧应通过后续周期遥测回看命令执行后的配置或状态变化。

### 2.2 遥测接收过程

FPGA 发送遥测时，各模块本地组织应用层 `payload`，再由 RS422 发送链路套固定帧。

整体过程如下：

```text
控制与状态模块周期轮询各功能模块
-> 目标模块锁存本地状态或配置快照
-> 目标模块输出应用层遥测 payload
-> 控制与状态模块汇聚遥测 payload
-> RS422 发送链路套固定帧并发送给主机
-> 主机软件校验固定帧
-> 主机软件解析应用层 header
-> 主机软件按 module_id / group_id 查遥测清单解释数据字
```

## 3. RS422 固定帧结构

RS422 固定帧是外层链路帧，遥控和遥测方向共用同一套固定帧壳。

当前 RTL 中固定帧格式如下：

```text
32-bit frame_header
32-bit payload_length_bytes
N * 32-bit payload_word
32-bit checksum
```

字段定义如下：

| 字段 | 位宽 | 软件填写或解析规则 |
| --- | ---: | --- |
| `frame_header` | 32 bit | 固定为 `0x1ACFFC1D`。 |
| `payload_length_bytes` | 32 bit | 应用层 payload 的字节数，不包含 `frame_header`，不包含本长度字段，不包含 `checksum`。 |
| `payload_word` | `N * 32 bit` | 应用层 payload，必须按 32-bit word 对齐。 |
| `checksum` | 32 bit | `frame_header + payload_length_bytes + 所有 payload_word` 的 32-bit 累加和，溢出自然回卷。 |

长度字段与 payload word 数的关系为：

```text
N = payload_length_bytes / 4
```

当前接收 RTL 要求：

- `payload_length_bytes` 必须是 4 字节对齐。
- `payload_length_bytes` 不能为 0。
- 当前单帧 payload 最大不超过 RTL 参数 `MAX_PAYLOAD_WORDS` 限制，现有默认值为 250 个 32-bit word。

## 4. 字节顺序和串口位序

每个 32-bit word 在 RS422 字节流中按高字节先发送：

```text
word[31:24] -> word[23:16] -> word[15:8] -> word[7:0]
```

例如 `0x1ACFFC1D` 在字节流中依次为：

```text
1A CF FC 1D
```

串口字节本身为普通 8N1 格式：

```text
1 bit 起始位
8 bit 数据位
1 bit 停止位
```

字节内部按串口常规方式 LSB first 发送。当前 RTL 默认波特率参数为 `115200`。

## 5. 应用层 Payload 结构

RS422 固定帧中的 `payload` 是应用层帧内容。遥控和遥测都以同一个 `header word0` 开头。

通用 payload 结构为：

```text
header word0
-> data word0
-> data word1
-> ...
```

其中 `header word0` 固定为 1 个 32-bit word。

## 6. 应用层 Header Word0

`header word0` 位域如下：

```text
[31:28] version
[27:24] msg_type
[23:20] module_id
[19:12] group_id
[11:4]  param_count
[3:0]   reserve
```

字段定义如下：

| 字段 | 位宽 | 遥控方向 | 遥测方向 |
| --- | ---: | --- | --- |
| `version` | 4 bit | 固定填 `0x1` | 固定为 `0x1` |
| `msg_type` | 4 bit | 固定填 `0x1`，表示遥控命令 | 固定为 `0x2`，表示遥测帧 |
| `module_id` | 4 bit | 目标模块编号 | 遥测来源模块编号 |
| `group_id` | 8 bit | 目标模块内的遥控 group 编号 | 遥测组编号，当前公共组为 `0x80` 或 `0x81` |
| `param_count` | 8 bit | 参数个数，不直接等于参数区 word 数 | 遥测数据区 32-bit word 数，不包含 header |
| `reserve` | 4 bit | 固定填 `0x0` | 固定为 `0x0` |

主机软件组 `header word0` 时可使用以下公式：

```text
header_word0 =
    (version     << 28) |
    (msg_type    << 24) |
    (module_id   << 20) |
    (group_id    << 12) |
    (param_count << 4 ) |
    reserve
```

## 7. 消息类型

当前现役协议只保留两类 `msg_type`：

| 数值 | 含义 |
| --- | --- |
| `0x1` | 主机下发遥控命令 |
| `0x2` | FPGA 上报遥测帧 |

## 8. 模块 ID

主机协议侧 `module_id` 是 4-bit 编号。

| `module_id` | 目标模块 | 对应清单目录 |
| --- | --- | --- |
| `0x0` | 时钟管理模块 | `clock_manager_block` |
| `0x1` | ADC 接收模块 | `adc_rx_block` |
| `0x2` | GT 发送模块 | `gt_tx_block` |
| `0x3` | CXP 业务模块 | `cxp_yewu_block` |
| `0x4` | 激光器控制模块 | `laser_ctrl_block` |
| `0x5` | 业务接收模块 | `yewu_rx_block` |
| `0x6` | 业务发送模块 | `yewu_tx_block` |
| `0x7` | 通信发送模块 | `comm_tx_block` |
| `0x8` | 通信接收模块 | `comm_rx_block` |
| `0x9` | 控制与状态模块保留号 | 当前不作为普通遥控目标 |

注意：这里的 `module_id` 是协议侧 4-bit 编号，不是顶层框架中的 8-bit 模块数组索引。

## 9. 遥控 Payload 组帧规则

遥控 payload 结构为：

```text
header word0
-> 参数内容区
```

遥控方向 `header word0` 固定字段为：

```text
version  = 0x1
msg_type = 0x1
reserve  = 0x0
```

其余字段由具体命令决定：

- `module_id`：查 `遥控指令清单/README.md`。
- `group_id`：查对应模块遥控清单和该模块 `*_ctrl_pkg.sv` 中的 `*_CMD_GROUP_*` 常量。
- `param_count`：表示该 group 中的参数个数。
- 参数内容区：按该 group 的参数顺序放置参数数据。

需要特别注意：

- `param_count` 表示参数个数，不表示参数区总 word 数。
- 一个参数可以占用 0 个、1 个或多个 32-bit word。
- 普通数值类参数通常占 1 个 32-bit word。
- 脉冲类参数可以占 0 个应用层 word，此时 payload 只有 `header word0`，没有参数内容区。
- 窄字段通常放在 32-bit word 低位，高位补 0，具体有效位宽以对应遥控清单为准。

## 10. 遥测 Payload 解析规则

遥测 payload 结构为：

```text
header word0
-> telemetry data word0
-> telemetry data word1
-> ...
-> telemetry data word(param_count - 1)
```

遥测方向 `header word0` 固定字段为：

```text
version  = 0x1
msg_type = 0x2
reserve  = 0x0
```

当前共享遥测 `group_id` 为：

| `group_id` | 含义 | 说明 |
| --- | --- | --- |
| `0x80` | `runtime` | 运行态、测量值、计数值、故障观测等。 |
| `0x81` | `cfg` | 当前生效配置回读。 |

遥测方向 `param_count` 表示数据区 32-bit word 数，不包含 `header word0`。

因此，遥测应用层 payload 长度为：

```text
payload_length_bytes = (1 + param_count) * 4
```

主机软件解析遥测时，应使用：

```text
module_id + group_id + 上报字索引
```

去 `遥测指令清单/` 对应模块文件中查每个 `telemetry data word` 的含义。

## 11. 遥控清单使用方法

遥控配置建议按以下步骤执行：

1. 在 `遥控指令清单/README.md` 中找到目标模块的 `module_id`。
2. 打开对应模块 Markdown 文件。
3. 在 `Group 总表` 中找到需要下发的 group。
4. 确认该 group 中的参数数量、参数顺序、参数类型、应用层字数和应用层位宽。
5. 按 `header word0 + 参数内容区` 生成应用层 payload。
6. 用 RS422 固定帧包裹 payload。

遥控清单中的重点列含义如下：

| 列名 | 主机软件关注点 |
| --- | --- |
| `参数名字` | 用于识别参数常量。 |
| `参数中文名` | 用于显示或配置界面命名。 |
| `参数id` | 模块内部参数 ID，主机软件通常不需要单独放入帧中。 |
| `参数类型` | 判断是数值、枚举、脉冲还是数据流注入。 |
| `应用层字数` | 判断该参数在参数内容区占多少个 32-bit word。 |
| `应用层位宽` | 判断主机配置值的有效位宽。 |
| `默认值` | 用于主机软件初始化显示。 |
| `模块最终落地位宽` | 判断配置值最终有效低位宽。 |

## 12. 遥测清单使用方法

遥测解析建议按以下步骤执行：

1. 主机软件先解析 RS422 固定帧并校验 checksum。
2. 取 payload 第一个 32-bit word 作为 `header word0`。
3. 检查 `msg_type` 是否为 `0x2`。
4. 根据 `module_id` 进入 `遥测指令清单/` 对应模块文件。
5. 根据 `group_id` 选择 `TM_GROUP_RUNTIME_C` 或 `TM_GROUP_CFG_C`。
6. 按 `上报字索引` 顺序解释后续 `param_count` 个 32-bit data word。

遥测清单中的重点列含义如下：

| 列名 | 主机软件关注点 |
| --- | --- |
| `上报字索引` | 对应 payload 中数据区的 word 序号。 |
| `关联字段` | 软件内部变量名可参考该字段。 |
| `字段中文名` | 用于显示界面名称。 |
| `来源表达式` | 判断字段在 32-bit word 中的有效位位置和补零方式。 |
| `字段真实位宽` | 判断显示、掩码和单位换算时的有效位宽。 |

## 13. 软件实现建议

主机软件建议把协议拆成两层：

```text
RS422 固定帧层
应用层遥控/遥测 payload 层
```

固定帧层负责：

- 添加或检查 `frame_header`
- 填写或检查 `payload_length_bytes`
- 按 32-bit word 计算 checksum
- 完成 32-bit word 与字节流之间的转换

应用层负责：

- 组装或解析 `header word0`
- 按 `module_id` 选择目标模块
- 按 `group_id` 选择遥控 group 或遥测 group
- 按清单解释参数 word 或遥测 word

主机软件不应把旧版 `object_id`、快遥测、慢遥测等旧口径作为当前协议字段。当前现役公共头字段只使用：

```text
version
msg_type
module_id
group_id
param_count
reserve
```

## 14. 当前对接风险和约束

当前遥控、遥测清单已经按模块整理，但部分单模块遥控 Markdown 文件的 `Group 总表` 主要列出 group 名称，未在表格中直接列出 `group_id` 数值。

正式软件配置时，`group_id` 应以以下来源为准：

1. 对应模块 `import/rtl_source/<module>/vars/*_ctrl_pkg.sv` 中的 `*_CMD_GROUP_*` 常量。
2. 后续补充到 `遥控指令清单/` Markdown 文件中的 group_id 数值。

如果 Markdown 清单与当前 RTL 包定义冲突，应先暂停确认，不应由主机软件自行猜测。

## 15. 后续示例说明入口

后续补充具体示例时，建议按以下类别各给若干例子：

- 遥控数值配置类：例如通信发送速率选择、激光器温度设定。
- 遥控脉冲触发类：例如复位脉冲、计数清零脉冲。
- 遥测 runtime 类：例如接收锁定状态、帧计数、激光器温度。
- 遥测 cfg 类：例如通信发送配置回读、通信接收配置回读。

每个示例建议同时给出：

```text
module_id
group_id
param_count
header word0
参数或遥测 word 列表
payload_length_bytes
checksum 计算过程
最终字节流
```
