# 与甲方沟通工作流规范

> 状态：首轮正文
> 阶段：02-rules
> 作用：统一后续与甲方沟通时的材料分层、提问方式、回复落档和进入实现前的闸门，减少绕路、扩范围和被动返工。

## 1. 范围与边界

- 约束什么：
  - 与甲方进行接口、联调、范围、默认前提相关沟通时的工作流。
  - 对外沟通材料的分层方式。
  - 哪些问题现在必须问、哪些只能留痕、哪些必须后置。
  - 甲方回复后的落档方式和继续推进规则。
- 不约束什么：
  - 具体接口 schema 设计。
  - 具体实现方案、模块拆法和开发计划。
  - 替甲方重写完整协议或直接替甲方拍板最终技术方案。
- 为什么必须先于架构和实现阶段明确：
  - 这类问题一旦没有统一流程，最容易在“内部排雷”“对外提问”“默认理解”“实现承诺”之间来回跳，导致范围反复放大。

## 2. 规则条款

### 2.1 材料分层

- `CG-01 [必须]` 与甲方沟通前，必须先产出内部排雷材料，不得直接拿甲方原文或零散口头理解去问甲方。
- `CG-02 [必须]` 对外材料必须至少分为两层：正式参考稿和短消息；禁止把内部排雷稿直接发给甲方。
- `CG-03 [必须]` 甲方回复后，必须单独落到“回复记录与默认前提”材料中，不得只停留在聊天记录里。
- `CG-04 [禁止]` 禁止把内部“防坑”“排雷”“卡人风险”表述直接带入对外材料。
- `CG-05 [允许:前提]` 允许为同一轮沟通准备多份内部材料，但对外出口必须收敛为一份正式参考稿和一份短消息。

### 2.2 问题分类

- `CG-06 [必须]` 所有问题在对外前必须先分成四类：现在必须问、现在不要问但要留痕、明确后置、不进入本轮。
- `CG-07 [必须]` 只有会直接影响主线骨架、对象边界、默认行为或完成判定的问题，才允许进入“现在必须问”。
- `CG-08 [禁止]` 禁止因为甲方文档里写了字段，就自动把该字段提升为“现在必须问”。
- `CG-09 [必须]` 凡是会把范围顺手带大、但当前又不是主线阻塞项的问题，必须归入“现在不要问但要留痕”或“明确后置”。
- `CG-10 [必须]` 对实现成本高、对方未主动要求、且不阻塞当前联调的事项，只能留在内部排雷或后置记录中，不得写进对外正式稿或短消息。
- `CG-10A [禁止]` 禁止在对外材料中主动罗列“不做项”或“暂不处理项”，以免提醒对方追加本来可有可无的范围。
- `CG-10B [允许:前提]` 只有当某个重字段会直接影响当前主线验收，或已经被对方明确点名时，才允许写进正式参考稿；写法必须是当前默认口径和需要纠偏的点，不展开完整体系。

### 2.3 对外提问方式

- `CG-11 [必须]` 对外提问必须优先使用“我们当前默认口径 + 如无异议按此推进 / 请指出不一致部分”的写法，禁止直接使用开放式大问题。
- `CG-12 [必须]` 对外提问必须优先问主线骨架，不得先问完整字段表、完整错误码表或完整展示结构。
- `CG-13 [必须]` 每个对外问题都要说明：当前怎么理解、为什么现在必须确认、如无特殊说明将按什么口径推进。
- `CG-14 [禁止]` 禁止用“是否存在、若存在请给完整规则”这类问题直接邀请甲方展开复杂体系。
- `CG-15 [禁止]` 禁止在甲方未明确前，主动替甲方补完整协议语义、完整状态机或完整验收口径。
- `CG-16 [必须]` 能由我们先定且实现更简单的默认规则，应写成默认口径，不写成向对方请示的问题。
- `CG-17 [禁止]` 不得在对外材料中用“如认为本轮必须纳入，请明确规则”这类句式主动提示扩展范围；除非该事项已经被对方明确要求或阻塞一期验收。
- `CG-18 [必须]` 对外正式参考稿和短消息不得使用明显的内部工作流口吻，例如“聚焦”“收口”“主线闭环”“排雷”“先把最小主线做稳”等；对外只写需要对齐的事项、默认理解和后续推进口径。
- `CG-18A [必须]` 对外材料默认使用“我们”，避免使用过度恭敬或容易拉远责任边界的称呼。

### 2.4 默认前提与回复落档

- `CG-19 [必须]` 每一轮对外沟通前，必须先写清“当前默认前提”；未写默认前提，不得发起正式确认。
- `CG-20 [必须]` 甲方明确回复后，必须把结果转写成稳定结论；不得直接引用聊天原句替代结论。
- `CG-21 [必须]` 甲方回复模糊时，必须明确标记为“未收口”，不得按有利于实现的方向擅自美化成已确认结论。
- `CG-22 [必须]` 甲方未回复的问题，若不构成当前阻塞，必须按已声明的默认前提继续推进，并在回复记录中保留未答状态。
- `CG-23 [禁止]` 禁止因为甲方口头说过一次，就在后续文档中把“倾向”写成“已确认”。

### 2.5 实现前闸门

- `CG-24 [必须]` 下列问题未收口前，不得把相关语义写死到内部主逻辑：
  - 主线对象边界
  - 主线对象命名与关联键口径是否稳定闭合
  - 启动 / 停止控制粒度
  - 最小结果分类
  - 文件完成与业务交付完成的边界
  - 心跳交互模型
- `CG-25 [必须]` 对甲方未收口但又不阻塞主线的问题，内部实现必须采用默认、透传、挂空或后置策略，不得顺手做重。
- `CG-26 [禁止]` 禁止让“留痕字段”提前进入首轮主逻辑，只因为甲方文档里把它写得很重。
- `CG-27 [允许:前提]` 允许在实现前保留少量未确认字段，但前提是其处理方式已经在文档中明确为默认 / 透传 / 挂空 / 后置，且不影响当前主线闭环。

## 3. 正例 / 反例

### 3.1 正例

- 先产出内部“逐字段排雷审查”，再压成对外“接口确认参考稿”和短消息。
- 对外写法使用：
  - “我们先按任务下发后启动当前任务内 case 执行处理；如果启动对象不是 task 下的 case 清单，请指出实际控制对象。”
- 对 `resources`、`checkPoints`、`getSubSysState.data` 这类字段，若没有阻塞当前主线或对方明确点名，只在内部排雷稿中留痕，不主动写入对外材料。
- 对单任务执行、重复任务、停止后报告、时间口径这类我们可以先定的规则，写成“我们默认按此处理，如无异议继续推进”，而不是逐项请示。

### 3.2 反例

- 直接把内部排雷结论发给甲方，让甲方看到“卡人”“防坑”之类内部判断。
- 直接问：
  - “是否存在重试、重复下发、并发任务？若存在请给完整规则。”
- 把 `handleCode`、`checkPoints`、`getSubSysState.data` 这种本来该留痕后置的重字段，又重新抬成当前必问项。
- 在对外稿里列出“本轮不处理完整设备管理、完整告警、升级重启、全过程消息”等清单，提醒对方追加范围。
- 用“如需纳入请明确规则”的方式主动邀请对方展开本来未必会要求的复杂事项。
- 甲方只说“先这样吧”，内部文档就写成“已确认按此实现”。

## 4. 与后续阶段的边界

- 留给 `03-architecture`：
  - 在已收口的默认前提下，内部边界怎么划分。
  - 哪些对象进入主逻辑，哪些只留在边界层。
- 留给 `04-specs`：
  - 已确认主线字段的结构化定义。
  - 具体请求 / 响应对象与最小完成判定。
- 留给 `05-plan`：
  - 开发顺序、验证步骤和联调执行计划。
- 不留给后续阶段重新争论的内容：
  - 已在对外沟通中明确确认的主线骨架问题。
  - 已声明且未被甲方纠偏的默认前提所对应的“当前推进口径”。
  - 未被甲方纠偏的默认前提不得直接升格为后续不可重审的稳定结论。
