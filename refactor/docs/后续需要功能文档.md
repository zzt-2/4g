## 1. 这份文档是干什么的

这份文档不是正式需求说明，也不是最终实施方案。

它的作用只有一个：

**把目前已经收拢下来的项目真实情况，整理成一份能直接带到新对话里继续分析的基础文档。**

后面无论是继续讨论：

- 接口到底怎么对
- 测试怎么组织
- Postman / mock / 集测怎么搞
- 哪些接口先做、哪些后做
- 哪些内容其实不该这期做

都应该以这份文档作为“当前已知前提”。

## 2. 当前已确认的项目背景

### 2.1 项目本质

当前项目是一个**我方与集成控制系统 / 中心控制系统的对接项目**。

我们收到的是对方给的接口文档：《集成测试系统与各二级子系统接口设计》。

当前不应把它理解成“我方要完整实现一个通用二级子系统平台”，而更应理解成：

**我方作为单一二级子系统，需要和中心控制系统把实际要用到的接口链路对接起来。**

### 2.2 当前材料现状

目前对方给到的主要是接口文档。

当前已知问题包括：

- 主要材料还是 Word/Markdown 形式的接口文档
- 还没有完整、稳定的字段字典
- 还没有可直接使用的 OpenAPI / Swagger
- 也没有现成、可靠的 Postman Collection
- 文档里很多地址写的是 `localhost` / `127.0.0.1`

因此，目前的核心问题之一不是“代码怎么写”，而是：

**接口定义还不够可执行，联调前提也还没有完全准备好。**

### 2.3 当前关于网络与联调环境的现实约束

当前理解是：

- 后续正式环境大概率不能直接走公网
- 但开发联调阶段，希望尽量争取一个更容易接入、便于测试的方式

这一点后续需要和对方确认，但在当前阶段应视为一个重要前提：

**正式部署限制与开发联调便利性，可能不是一回事。**

## 3. 当前已经明确纠偏的信息

以下内容是后续分析时必须使用的新前提，不能再沿用之前已经被修正的旧假设。

### 3.1 对方不会单独找我们做中间参数查询和参数配置

目前已明确：

- 对方不会单独找我们要中间参数
- 不需要把“参数查询接口”“参数配置接口”作为当前主线
- 参数不是通过独立参数接口给过来

更准确的理解是：

**参数会跟着测试任务一起过来，并且落在任务中的测试用例里。**

### 3.2 测试任务里会带测试用例，用例里会带参数

当前已经明确：

- 对方会通过“测试任务下发”来驱动我们
- 测试任务中会包含测试用例
- 每个测试用例中会带参数

因此，当前真正该盯住的主线不再是“单独参数接口”，而是：

**测试任务下发 -> 任务中包含用例 -> 用例中包含参数 -> 我方执行。**

### 3.3 对方可能会停止任务，但不需要暂停、继续

当前已明确：

- 对方可能会停止测试任务
- 当前不需要暂停
- 当前不需要继续

因此，执行控制当前应收敛为：

- 启动任务 / 开始执行
- 停止任务

而不是一整套细粒度调度控制。

### 3.4 每个用例完成后需要上报

当前已明确：

- 每个测试用例完成后，需要向对方上报结果

这意味着：

- 后续必须认真看“每个用例完成后上报”到底落在哪个接口上
- 它更像是“用例级结果回传”
- 不应简单混同成“全过程步骤上报”

也就是说，后续分析时要特别区分：

- 用例完成后的结果上报
- 执行过程中的步骤/消息持续上报

这两件事不是一回事。

### 3.5 整个任务完成后需要给对方 JSON 报告

当前已明确：

- 任务完成后，需要给对方一份报告
- 当前理解里，报告形式倾向于 JSON

因此，后续功能设计里不能只考虑“状态和单次响应”，还必须考虑：

**任务完成后，任务级结果 / 报告如何交付。**

### 3.6 “误码率”等关键结果值目前没有确认

这里必须明确纠偏：

- 之前举过“误码率”这个例子
- 但目前它**不是已确认交付项**

当前只能确认：

- 对方需要结果输出
- 但还没有最终确认具体结果字段清单

因此后续不能再把下面这种话当成既定事实：

- “对方明确要误码率”
- “关键结果值已经确认是误码率”

当前正确表述应是：

**对方需要结果，但具体结果字段清单还没有确认。**

## 4. 当前真正应该围绕的主线

结合目前已确认前提，当前最合理的主线应收敛为：

**测试任务下发 -> 任务中携带用例与参数 -> 我方启动执行 -> 对方可能发停止任务 -> 每个用例完成后上报 -> 整个任务完成后输出任务级 JSON 报告**

这条主线比之前更贴近实际。

后续所有功能判断，优先围绕这条主线来做，而不是围绕整本接口文档的所有章节平均发力。

## 6. 基于当前信息，后续真正需要的功能

这一节不是照抄接口标题，而是站在“后续真正要落什么”的角度来整理。

### 6.1 任务接收功能

后续需要有一套能力，能让我们接收来自中心控制系统的测试任务。

这个任务至少要能承载：

- 任务标识
- 与本次执行相关的上下文
- 任务中包含的测试用例
- 每个测试用例对应的参数

当前不应默认把它做成：

- 复杂任务系统
- 多层任务树
- 完整任务生命周期平台

当前更适合把它理解成：

**接收一次可执行的测试任务请求。**

### 6.2 任务启动功能

后续需要具备接到任务后开始执行的能力。

这里的重点不是“中心深入控制我方每一步”，而是：

- 中心下发任务 / 启动信号
- 我方切换状态
- 我方内部按自己的执行方式运行

如果后续一定要保留“测试用例集启动”接口，也应尽量把它收敛成：

**任务开始执行的触发动作**

而不是扩展成：

- 中心对我方完整用例集的编排式控制

### 6.3 任务停止功能

后续需要有“停止任务”的能力。

当前已知不需要：

- 暂停
- 继续

因此控制功能应收敛为：

- 开始
- 停止

不要提前扩成更多动作。

### 6.4 子系统状态反馈功能

后续需要能让中心系统查询我方状态。

当前最需要的状态不是复杂状态树，而是最基本、最能支撑联调的几类状态：

- 是否在线
- 是否空闲
- 是否正在执行任务
- 最近一次执行是否异常
- 当前是否可接收任务

状态接口应尽量做成：

**最小、稳定、可理解**

而不是过度暴露内部处理细节。

### 6.5 心跳保活功能

后续需要有心跳，保证对方知道我方在线、可通信。

心跳应主要解决：

- 在线判断
- 基础可用性判断

不应在心跳里承载过多业务数据。

### 6.6 用例级结果上报功能

这是当前最关键的新结论之一。

既然已经明确“每个用例完成后需要上报”，那后续必须有一套机制，能够在每个用例执行完成后向中心系统回传结果。

这个“用例级结果上报”至少要解决：

- 当前完成的是哪个用例
- 用例执行结果是什么
- 是否成功/失败/异常
- 是否有附加说明
- 是否需要携带结果字段

这里要特别注意：

**它和步骤级过程上报不是一回事。**

后续分析时必须重点比较：

- `测试结果通知`
- `信息上报`

看哪一个更适合作为“用例完成后上报”的承载接口，或者是否需要两者配合。

### 6.7 任务级结果汇总与报告输出功能

除了每个用例完成后上报之外，整个任务完成后还需要形成任务级交付物。

当前已知交付物是：

- JSON 报告

因此后续必须有一套能力，能够在任务结束后形成：

- 整个任务的执行结果
- 用例级结果汇总
- 任务开始/结束信息
- 报告内容
- 报告交付方式

这里要继续确认：

- 报告是通过接口直接推送
- 还是通过文件上传
- 还是先上传文件，再通过完成通知告诉对方

但无论最终走哪条链路，

**任务级 JSON 报告输出，是后续一定要考虑的。**

### 6.8 结果字段确认与结构整理功能

虽然具体结果字段还没定，但后续一定需要一件事：

**把“每个用例完成后要上报什么”“任务完成后报告里要放什么”整理成结构化结果清单。**

这件事当前还没有完成，但它是后续设计和测试的前提。

至少要逐步明确：

- 用例完成后有哪些字段
- 任务完成后有哪些字段
- 哪些字段是必填
- 哪些字段是可选
- 哪些字段是状态
- 哪些字段是真实测试结果

### 6.9 文件 / 报告交付配套功能

既然任务完成后要给 JSON 报告，那么后续至少还需要判断文件/报告交付怎么走。

这可能涉及：

- 报告生成
- 文件命名
- 文件存放
- 文件上传
- 文件完成通知

当前不能直接断言都要做，但必须在后续分析中重点核对：

**JSON 报告的最终交付方式是什么。**

### 6.10 异常反馈功能

后续一定需要有异常反馈，不然联调和运行时都很难闭环。

至少要能表达：

- 任务无法启动
- 执行中止
- 用例失败
- 报告生成失败
- 通知失败

这部分后续可能落在：

- 结果上报里的异常字段
- 状态查询里的异常状态
- 必要时的告警接口

但当前不应直接展开成完整“告警系统”。

## 7. 当前不应再作为主线的内容

结合当前已知情况，以下内容至少在当前阶段不应继续当成主线推进：

### 7.1 单独参数查询 / 参数配置

原因：

- 当前已经明确参数不单独走
- 参数跟着任务里的用例一起下发

因此后续不应再把“参数接口”当成第一优先级主线。

### 7.2 把误码率当成已确认结果

原因：

- 当前并没有确认具体关键结果字段清单
- 误码率只是之前举过的例子，不是已确认项

因此后续不能基于“误码率已确定”往下推实现和测试。

### 7.3 把全过程步骤上报当成已确定需求

原因：

- 当前明确的是“每个用例完成后上报”
- 但这不等于“过程中的所有步骤都要持续上报”

因此后续必须区分：

- 用例完成通知
- 步骤过程上报

不能混为一谈。

## 8. 当前最需要继续确认的问题

这部分是后续新对话特别应该围绕的问题。

### 8.1 联调方式相关

- 文档里的 `localhost / 127.0.0.1` 到底是示例地址，还是真实部署方式？
- 开发联调阶段有没有可访问的真实地址？
- 正式环境不能连公网时，开发阶段能否提供临时联调方式？
- 接口最终到底是谁提供、谁调用谁？

### 8.2 结构化接口定义相关

- 是否能提供 OpenAPI / Swagger？
- 是否能提供 Postman Collection？
- 如果没有，能否补字段字典？
- 每个接口是否能给出完整请求/响应字段定义与样例？

### 8.3 任务主线相关

- 测试任务下发里到底包含哪些字段？
- 任务中的测试用例结构是什么？
- 用例参数结构是什么？
- 任务启动和“测试用例集启动”到底是什么关系？
- 停止任务的语义是什么？是立即停、优雅停，还是当前用例跑完再停？

### 8.4 结果回传相关

- 每个用例完成后，上报的是哪些字段？
- 用例结果上报具体走哪个接口？
- 任务完成后 JSON 报告里需要哪些字段？
- 报告怎么交付？
- 任务级结果和用例级结果如何区分？
- 具体结果字段清单到底是什么？

### 8.5 文件与报告相关

- JSON 报告是接口返回，还是 FTP 上传？
- 如果走 FTP，测试环境怎么给？
- 文件上传完成通知和文件传输完成接口，这期是否真的要用？

### 8.6 辅助接口相关

- 设备列表查询 / 设备信息上报这类接口，这期是真要联调，还是只是文档保留？
- 用例查询 / 用例脚本上传，这期是真要落地，还是不是当前主线？
- 设备告警 / 子系统告警，这期是否真的要接？
- 升级 / 重启，这期是否真属于联调范围？

## 10. 当前版本的一句话总结

截至目前，当前项目后续真正应该围绕的，不再是“简单开始/结束 + 某个已知关键结果值”，而更接近：

**测试任务下发 -> 用例和参数随任务一起过来 -> 启动/停止任务 -> 每个用例完成后上报结果 -> 任务完成后回 JSON 报告**

而当前最大的问题也不是“代码怎么写”，而是：

**接口定义还不够结构化、联调方式还没明确、结果字段还没确认，所以后续必须先把这些前提梳理清楚，项目才能真正进入可测试、可落地状态。**
