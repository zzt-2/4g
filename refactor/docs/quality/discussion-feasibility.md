# 讨论阶段决策质量检查

> 1xx 讨论阶段品控表。先广后深：Phase 0 确保"想全了"，A-D 确保"每个决策可行"。
>
> **硬检查** `[MUST]` — 不通过不能进入 2xx。**软检查** `[SHOULD]` — 可标注为已知风险后继续。
>
> 检查结论附在 session note 的"可行性检查"小节中。豁免条件：纯 UI 文案、纯配置、不涉及代码改动的决策——标注豁免即可。

---

## Phase 0: 覆盖性检查（6 问）

讨论阶段**首个 session** 由子 agent 强制回答。6 问全部回答（含"不适用"标注）才能进入 2xx。

| # | 问题 | 防止什么 |
|---|------|---------|
| Q1 | 用户可见的行为变化是什么? | 脱离用户价值 |
| Q2 | 数据模型（表/字段）怎么变? | 遗漏迁移/兼容性 |
| Q3 | API 契约（接口/DTO）怎么变? | 前后端不匹配 |
| Q4 | 外部依赖（Ragflow/pi-ai/SDK）受影响吗? | 假设外部能力 |
| Q5 | 现有数据怎么处理? | 数据丢失 |
| Q6 | 最小可交付范围是什么? | 范围膨胀 |

执行：Q2-Q5 合并为 2 批子 agent（每批 2 个），Q1/Q6 主对话回答。输出每问一句话结论 + 状态标注（已确认/待定/不适用）。

---

## A-D 逐决策可行性检查

每条涉及代码改动的决策，按以下 10 条检查。不逐条串行，合并为 2-3 轮子 agent 并发（每轮 2-3 个，不超过 3 并发）。

### A. 代码接触面 `[SHOULD]`

| # | 检查内容 | 判断标准 | 输出 |
|---|---------|---------|------|
| A1 | 触及文件与改动性质 | 每文件标注 `新增/改逻辑/改类型/改引用/删代码`，>15 文件标注复杂度 | 分层表格（不展开具体改法） |
| A2 | 依赖链与改动顺序 | 循环依赖或 >3 层串行依赖 → 风险项 | "A 先于 B" / "C 和 D 可并行" |
| A3 | 实施成本量级 | S(<5 文件) / M(5-15) / L(>15 或 3+ 重度) / XL(跨模块大重构) | 总量级 + 各层分量级 |

### B. 技术前提

| # | 检查内容 | 级别 | 判断标准 | 阻塞条件 |
|---|---------|------|---------|---------|
| B1 | 外部系统 API 能力确认 | `[MUST]` | `已确认支持/已确认不支持/未验证/有可靠性风险` | "未验证"项阻塞 2xx |
| B2 | 前序决策与文档一致性 | `[MUST]` | 标注 `继承/补充/修正/撤销` | 未解决的矛盾阻塞 2xx |
| B3 | 数据安全与可逆性 | `[MUST]` | `可逆/不可逆但有备份/不可逆且无备份` | 第三级需用户确认 |

### C. 规范兼容性 `[SHOULD]`

| # | 检查内容 | 判断标准 |
|---|---------|---------|
| C1 | 代码规范合规预检 | `完全兼容/有冲突（列规则编号）/需新增规则` |
| C2 | 运行时机制兼容性 | `现有机制直接支持/需扩展/需新建` |

### D. 替代方案与简化 `[SHOULD]`

| # | 检查内容 | 判断标准 |
|---|---------|---------|
| D1 | 替代方案与简化路径 | 三个子问题：不改代码行不行? 有可复用机制吗? 能拆 MVP 吗? |
| D2 | 遗漏风险扫描 | 新发现的遗漏按影响分级 `P0(运行时报错)/P1(编译报错)/P2(应同步更新)` |

### 执行编排

**第 1 轮**（代码扫描类）：A1 + A2 + A3 + D2 并行
**第 2 轮**（前提验证类）：B1 + B2 + B3 并行
**第 3 轮**（规范+方案类）：C1 + C2 + D1 并行

## 参考实例（按需阅读）

以下实例来自 ragflow-kb S107 收口，仅供首次使用本检查表时参考模式，日常执行不需要每次阅读。

**A1 实例**：S107 按 4 层列出缺口（Ragflow 基础设施 3 文件 / 统一检索 tool 2 文件 / 模板与 workflow 2 文件 / Spec 与文档），但 P1 遗漏了 `document.service.ts:getPackAggregation()` 和 `packs/index.vue` 分组算法。

**B1 实例**：S107 验证 P1（chunk 写 tag_kwd → 支持）、P2（retrieval 按 tag_kwd 过滤 → 不支持）、P3（metadata_condition → 支持但有可靠性风险）。P2 导致整个检索流程设计调整。

**B2 实例**：S107 发现 Spec 13 "不做 metadata/tag" 与 1xx 决策直接矛盾、DOCUMENT_TAG 误删、metadata_condition 定位漂移。

**B3 实例**：删 categories 字段（6349 条数据中 47 条需迁移）、删 4 个 DictKey、重命名多字段（Prisma @map 零迁移）。

**D1 实例**：WorkType 单选 vs 多选——保持 `String[]` 平时只填一个值就达到同样效果，改动量从 M 降为 S。

**D2 实例**：最后一轮子 agent 发现 S106 未列出的 8 个遗漏文件，含 `knowledge-query.port.ts`（改 port 不改 adapter 会类型报错）。
