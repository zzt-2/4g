# 甲方接口文档与我方当前方案契合度审查

> 文档定位：甲方原始接口文档理解阶段 / 我方当前方案契合度审查阶段 / 集测前外部接口风险识别阶段 / 甲方沟通准备阶段
>
> 审查基线：以我方当前已经收口的内部边界和“任务下发 -> 启动 / 停止 -> 状态 / 心跳 -> 用例结果上报 -> 任务报告交付 -> 集测验收闭环”为唯一主线，不把长期平台能力、资产维护能力和运维能力混入本轮集测范围。

## 第一部分：本轮契合度审查的判断基线

### 1. 本轮审查基线是什么

这轮审查不是复述甲方文档，也不是替甲方拍板协议细节，而是用我方当前已经明确的主线和边界去判断：

- 甲方接口设计是否足以支撑本轮集测主线闭环
- 哪些接口可以直接纳入本轮主线
- 哪些接口虽然存在，但当前语义不稳、资产不全或范围过大
- 哪些问题属于“实现前先确认事实”
- 哪些问题已经不是实现细节，而是建议甲方收敛或调整接口设计

本轮统一以以下事实作为审查基线：

- 外部协同对象不等于内部骨架对象
- 任务系统是独立能力域
- 北向协同边界需要单独处理
- 当前主线只看任务下发、启动 / 停止、状态 / 心跳、用例结果上报、报告交付、联调与验收条件
- 当前不把独立参数接口、完整用例资产维护、完整告警体系、升级 / 重启、全过程过程消息当作本轮集测前置项

### 2. 四类判断口径

#### 2.1 `基本契合，可沿用`

满足以下条件：

- 与我方当前主线直接相关
- 接口职责边界基本清楚
- 即使字段后续仍需细化，也不会直接把 spec 写偏

#### 2.2 `存在歧义，需要甲方确认`

满足以下条件：

- 方向基本对，但关键交互事实未闭合
- 文档存在多义、缺关键定义或前后不一致
- 如果不先确认，后续 spec、联调流程或验收口径会失真

#### 2.3 `与我方当前计划不契合，建议甲方调整`

满足以下条件：

- 接口语义与我方当前主线边界明显冲突
- 接口把外部事务、内部运行语义和交付副作用混在一起
- 如果不调整，会明显拖累本轮 spec 切分、联调闭环或验收口径

#### 2.4 `不建议纳入本轮集测范围`

满足以下条件：

- 不是本轮主线闭环前置条件
- 当前纳入会明显放大范围
- 即使甲方文档里有，也不应默认本轮必须打通

### 3. 什么属于事实确认，什么属于建议甲方调整

属于事实确认的问题，主要包括：

- 任务接收与启动是否为两个独立外部事务
- 停止、终止、拒绝、不可执行、超时分别怎么定义
- 心跳到底是单向上报、双向保活还是“无响应但又有响应示例”
- 报告最终通过接口回传、文件上传还是上传后再通知
- 本轮集测是否真要使用 FTP、文件拉取、过程消息、参数直连接口
- 是否有可直连测试环境、鉴权方式、账号、网络开通要求和联调入口

属于建议甲方调整的问题，主要包括：

- 本轮集测接口范围过宽，建议按主线收敛
- 结果通知、文件上传完成、报告交付三个层次混杂，建议拆清
- 心跳章节前后自相矛盾，建议统一交互口径
- 当前仅有文字文档和本地部署 URL，建议补 Swagger / OpenAPI 与在线环境
- 对外结果语义过粗或词义不稳，建议统一定义成功 / 失败 / 拒绝 / 不可执行 / 超时

## 第二部分：甲方接口文档与我方当前方案契合度审查

### 1. 协议与公共规则

#### 1.1 通用消息外壳与公共字段

- 分类判断：`基本契合，可沿用`
- 判断原因：`method`、`requestId`、`subSysType`、`subSysId`、`sessionId` 这一套公共外壳，和我方“北向协同边界统一承接、内部再做翻译”的思路并不冲突，也适合作为后续所有北向动作的统一事务外壳。
- 对我方当前计划的影响：这部分可以直接作为北向协同对象组的公共约束，不会卡住内部骨架和任务系统的对象切分。
- 是否需要和甲方沟通调整：不需要先建议改写，但需要确认 `requestId / sessionId` 的生成方、透传规则和事务边界。

#### 1.2 HTTP / HTTPS 写法与调用入口

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：公共规则写 HTTP，具体接口示例大量写 `https://127.0.0.1:5001/...`，说明协议、部署方式和实际联调入口并未真正闭合。
- 对我方当前计划的影响：如果这一点不确认，后续 spec 很容易把“文档示例地址”误当成真实联调入口，直接影响环境准备和联调排期。
- 是否需要和甲方沟通调整：需要先确认事实；如果甲方当前只有本地部署地址，建议补充可直连测试环境入口和正式访问方式。

#### 1.3 “消息接口 + FTP 文件传输”双轨机制

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：甲方把 FTP 作为公共规则主干之一，但我方当前真正需要的是“报告怎么交付、文件链路本轮是否可用、成功怎么判”，而不是先默认所有文件类动作都必须依赖 FTP 落地。
- 对我方当前计划的影响：如果不先确认 FTP 是否为本轮必选链路，后续 spec 会把报告交付、结果附件交付、资产文件传输全都提前写死到 FTP 上，增加联调不确定性。
- 是否需要和甲方沟通调整：需要确认本轮集测哪些动作真的依赖 FTP；如果甲方暂无稳定 FTP 联调条件，建议本轮优先收敛为主线结果与报告交付闭环。

#### 1.4 文件目录、命名和脚本编码规则

- 分类判断：`不建议纳入本轮集测范围`
- 判断原因：这些规则更偏长期资产协同与文件治理，当前对本轮主线真正有影响的，只是“报告交付和必要附件是否能用、怎么判送达”，不是先把完整命名体系和脚本资产规则做满。
- 对我方当前计划的影响：如果现在把这一套作为本轮前置，会把讨论从“主线闭环”拉回“平台资产治理”，拖慢 spec 和联调准备。
- 是否需要和甲方沟通调整：可以保留为后续范围，不建议作为本轮集测阻塞项。

### 2. 任务主链

#### 2.1 `setTestTask` 测试任务下发

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：任务下发本身与我方主线高度契合，但 `immediate`、`isEnd`、`resources`、`ftpInfo`、`testCaseInfo.inputPars` 这些字段说明甲方把“任务接入、任务分批、资源描述、文件链路”揉在一个入口里，语义较重。
- 对我方当前计划的影响：如果不先确认“任务接收”和“启动执行”是否分离，后续任务接入对象、控制对象和任务上下文对象会被写偏。
- 是否需要和甲方沟通调整：需要先确认事实；如果甲方本轮集测只验证主线闭环，建议把必填范围收敛到任务标识、用例列表、参数和最小必要上下文，避免把资源编排和 FTP 一并压进首轮闭环。

#### 2.2 `startTestCaseList` 测试用例集启动

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：从我方当前计划看，“接收任务”和“启动任务”允许拆开，但甲方这里把启动粒度写成“测试集或测试 id”，而不是稳定的任务级控制语义，边界仍不清。
- 对我方当前计划的影响：如果启动动作到底作用于任务、测试集还是单用例不清，生命周期控制状态和任务系统控制输入都无法稳定落 spec。
- 是否需要和甲方沟通调整：需要确认事实；若甲方本轮只验主线，建议把启动语义明确收敛到任务级，避免本轮就引入复杂的部分启动或局部选择执行。

#### 2.3 `controlTestTask` 测试任务执行控制中的 `stop / abort`

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：我方当前明确只关心停止，而甲方同时给出 `stop`、`abort` 和 `handleCode`，说明停止、终止、拒绝、环境不具备之间的边界尚未写稳。
- 对我方当前计划的影响：如果不确认，后续“停止后的任务状态”“停止是否产出报告”“停止与失败如何区分”都会直接失真。
- 是否需要和甲方沟通调整：需要确认事实；建议甲方明确 `stop`、`abort`、`拒绝`、`不可执行` 的口径关系。

#### 2.4 `controlTestTask` 中的 `pause / continue`

- 分类判断：`不建议纳入本轮集测范围`
- 判断原因：我方当前已明确本轮不需要暂停、继续，这部分不是当前主线前置条件。
- 对我方当前计划的影响：如果把暂停 / 继续纳入，本轮生命周期控制会被不必要地扩张，拖慢任务主链闭环。
- 是否需要和甲方沟通调整：建议和甲方明确本轮先不验这两个动作，只保留启动 / 停止主线。

#### 2.5 `ccSysGetFileRequest` 中心控制系统文件获取请求

- 分类判断：`不建议纳入本轮集测范围`
- 判断原因：文档自己已经标注“是否有需求场景待定，暂保留”，而且它是中心侧反向取文件，不属于当前主线的必需动作。
- 对我方当前计划的影响：如果现在把它并入，会让“报告交付”和“中心反向取文件”两条不同方向的链路混在一起。
- 是否需要和甲方沟通调整：建议明确这条接口暂不纳入本轮集测范围。

### 3. 结果、文件与报告回传

#### 3.1 `testCaseResultReport` 测试结果通知

- 分类判断：`与我方当前计划不契合，建议甲方调整`
- 判断原因：我方当前主线明确需要“每个用例完成后上报结果”，这一点是契合的；但甲方当前给出的 `result=success/fail/tbd` 过于粗糙，`tbd` 无法承载我方后续必须区分的拒绝、不可执行、超时等语义。
- 对我方当前计划的影响：如果不调整，后续 spec 会被迫把多种不同结果压缩进 `tbd`，验收口径和异常归因都会失真。
- 是否需要和甲方沟通调整：需要，建议甲方至少把“成功、失败、拒绝、不可执行、超时”这几类外部结果语义拆开定义，避免继续使用含义过宽的 `tbd`；具体字段名和枚举值后续再确认。

#### 3.2 `testDataFileTranslationComplete` 测试数据文件上传完成

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：这条接口表达的是“文件传输完成并通知”，不是“业务结果已受理”或“报告交付已完成”，但文档又把测试报告文件也纳入这条文件回传语境，容易和真正的结果交付、报告交付混淆。
- 对我方当前计划的影响：如果不先拆清，后续 spec 会把“用例结果上报”“报告对象形成”“报告交付成功”误写成同一个动作，直接影响交付层设计和验收标准。
- 是否需要和甲方沟通调整：需要先确认事实；建议甲方明确“文件上传完成通知”和“报告交付成功判定”是否为两个不同层次。

#### 3.3 `fileTranslationComplete` 通用文件传输完成

- 分类判断：`不建议纳入本轮集测范围`
- 判断原因：这条接口主要面向与任务无关的通用文件传输，例如脚本文件，和本轮“任务执行闭环”主线并不直接相关。
- 对我方当前计划的影响：如果本轮纳入，会把结果交付链路和资产文件维护链路混杂在一起。
- 是否需要和甲方沟通调整：建议明确这条接口不作为本轮集测必验项。

#### 3.4 测试任务报告文件（`TestReport` JSON）

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：任务完成后形成 JSON 报告，与我方当前计划基本契合；但报告内容层次很重，包含 `judgmentMsg`、`checkPoints`、`processAndDatas`、`attachItems` 等大量扩展项，远超当前已确认的主线结果事实。
- 对我方当前计划的影响：如果不先确认最小必交内容，后续 spec 会被迫把报告对象一次性写成大而全结构，既影响首轮范围，也会把未确认的过程明细提前冻结。
- 是否需要和甲方沟通调整：需要先确认事实；建议甲方明确本轮报告最小必交内容，过程明细、分析定位和附件先作为可选增强项处理。

#### 3.5 报告交付成功判定

- 分类判断：`与我方当前计划不契合，建议甲方调整`
- 判断原因：甲方文档当前有报告文件结构，也有文件上传完成通知，但没有把“报告对象形成”“报告送达”“甲方业务受理成功”三层语义正式拆开。
- 对我方当前计划的影响：如果这一点不调整，后续 spec 和联调都无法稳定判断“报告到底什么时候算交付完成”，验收标准也会直接悬空。
- 是否需要和甲方沟通调整：需要，建议甲方单独定义报告交付动作及其成功 / 失败 / 暂不可交付回执口径，不要仅依赖 FTP 文件到位或通用回执替代。

### 4. 运维控制与链路保活

#### 4.1 `heartbeat` 心跳接口

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：我方当前主线确实需要心跳，但甲方文档在同一章节里同时写了“此消息无响应消息”和 `heartbeatResponse` 示例，且又把周期配置挂到 `setPars` 上，说明心跳交互事实尚未闭合。
- 对我方当前计划的影响：如果不先确认，后续无法判断心跳属于单向上报、双向保活还是参数驱动的可配置链路，也会影响联调时谁先发起、成功怎么判。
- 是否需要和甲方沟通调整：需要，建议甲方统一心跳的发起方向、是否有响应、成功判定和周期配置方式，不要保留相互冲突的写法。

#### 4.2 `softwareUpgrade` 软件版本升级

- 分类判断：`不建议纳入本轮集测范围`
- 判断原因：这是运维升级能力，不是本轮任务执行闭环前置条件。
- 对我方当前计划的影响：如果纳入，会把联调范围从测试执行闭环扩展到包管理和升级流程，明显偏离当前阶段。
- 是否需要和甲方沟通调整：建议明确本轮不验。

#### 4.3 `neControl` 重启 / 开关机控制

- 分类判断：`不建议纳入本轮集测范围`
- 判断原因：这是子系统 / 设备运维控制，不是当前“任务下发 -> 执行 -> 结果 -> 报告”主线的一部分。
- 对我方当前计划的影响：纳入后会把任务控制和设备控制混成一层，增加状态边界复杂度。
- 是否需要和甲方沟通调整：建议明确本轮不验。

### 5. 用例资产与脚本维护

#### 5.1 用例查询、菜单维护、新增 / 删除用例

- 分类判断：`不建议纳入本轮集测范围`
- 判断原因：这整组接口处理的是用例资产维护，不是本轮任务执行主链。当前已知事实是“用例和设备信息由我方报给甲方使用”，并不意味着本轮必须先把完整用例资产平台打通。
- 对我方当前计划的影响：如果把这组接口带入本轮，会直接扩大到菜单树、用例属性库、生命周期维护，偏离当前目标。
- 是否需要和甲方沟通调整：建议明确本轮只确认用例随任务下发和最小资产事实，不验完整资产维护链路。

#### 5.2 脚本上传、修改、下载

- 分类判断：`不建议纳入本轮集测范围`
- 判断原因：脚本资产链路属于长期协同能力，不是当前主线闭环前置。尤其是 FTP 脚本下载 / 上传和属性维护耦合在一起，会明显放大联调范围。
- 对我方当前计划的影响：如果提前纳入，会把当前“任务执行”和“脚本资产维护”混为一体，导致本轮 spec 无法收敛。
- 是否需要和甲方沟通调整：建议明确本轮暂不验完整脚本维护能力，如需引用脚本资产，只保留最小事实确认。

### 6. 设备、状态与告警

#### 6.1 `getSubSysState` 二级子系统状态查询

- 分类判断：`存在歧义，需要甲方确认`
- 判断原因：我方当前确实需要“状态查询响应”，但甲方这里把“子系统整体状态”和“环境自检详细过程信息”放在一个接口里，且 `status`、`data`、`keyPars` 的最小必需范围没有收紧。
- 对我方当前计划的影响：如果不先确认，后续状态投影对象会被写成“大状态 + 大自检明细”组合体，既影响本轮状态查询最小闭环，也影响验收口径。
- 是否需要和甲方沟通调整：需要，建议甲方明确本轮状态查询最小必需状态面，把环境自检过程信息作为增强项，而不是默认主线必填。

#### 6.2 设备列表 / 设备信息查询 / 设备信息上报

- 分类判断：`不建议纳入本轮集测范围`

## 第五部分：当前结论 / 待确认事项 / 下一步建议

### 1. 当前结论

1. 甲方当前接口设计与我方当前主线计划是“部分契合，但远未到可直接按文档实现”的程度。

2. 真正和我方当前主线直接契合的，主要只有以下几类能力：
- 任务下发
- 启动 / 停止
- 状态查询
- 心跳
- 每个用例完成后结果上报
- 任务结束后形成并交付报告

6. 当前最容易拖偏 spec 的，不是字段细节，而是这几类边界没有闭合：
- 任务接收与启动的边界
- 停止 / 终止 / 拒绝 / 不可执行的边界
- 结果通知、文件通知、报告交付的边界
- 心跳的交互方式
