# 甲方回复记录与默认前提

> 用途：集中记录每一轮向甲方发出确认后的回复结果、当前默认前提和未收口事项，避免结论散落在聊天记录中。
>
> 使用约束：
> 1. 甲方回复后，先转写为稳定结论，再更新本文件；不要直接把聊天原句当结论引用。
> 2. 甲方回复模糊时，明确标记为“未收口”，不要擅自按有利于实现的方向补足。
> 3. 甲方未回复但当前不阻塞推进的问题，保留“未回复”状态，并写明当前默认推进口径。
> 4. 对外未主动提及、且不阻塞一期联调的重字段或扩展能力，只在内部排雷材料中留痕，不在本文件中扩展成待问清单。

## 1. 当前轮次概览

- 当前轮次：`一期对接口径确认`
- 对外发送材料：
  - `一期对接口径确认参考稿.md`
  - `一期对接口径确认短消息.md`
- 发送时间：`待填写`
- 当前状态：`待发送`
- 当前默认推进口径：
  - 我们按完整二级子系统接入，不按单个设备或单个执行节点处理。
  - `deviceId` 按业务设备身份理解；串口、网口、SCOE 通道属于我们内部连接对象。
  - `setTestTask` 是任务下发，`startTestCaseList` 是任务启动。
  - 一期按单任务串行执行；已有任务运行时，新任务或再次启动先按不可执行处理。
  - 同一个 `taskId` 重复下发时，按重复请求处理。
  - 一期控制动作按 `stop` 处理，stop 后任务状态按 `stopped` 处理。
  - stop 后仍生成 JSON 报告，记录 stopped、已完成 case 和未完成 case。
  - case result 和 task result 分开处理。
  - JSON 报告生成、文件传输完成、业务接收完成分开判断。
  - 结果和报告时间先按我们系统时间记录；如需统一时区或固定格式，等甲方提供规则。
  - 一期先按基础状态和心跳对接。

## 4. 内部留痕事项

以下事项不主动放入对外确认稿；如果甲方后续明确提出，再按当时要求单独确认最小规则：

- `resources`、`isEnd`、`ftpInfo` 等任务附带重字段。
- `handleCode` 等控制回执细节。
- `TestReport` 中的重展示字段或过程明细字段。
- `getSubSysState.data`、`keyPars` 等状态扩展明细。

当前处理原则：

- 不因为字段在甲方文档里出现，就自动进入一期主逻辑。
- 不把没有稳定来源的字段写成已经承诺的一期能力。
- 不主动把扩展字段列入对外材料，避免扩大一期范围。

## 5. 维护说明

1. 每次甲方新增回复时，先更新“问题跟踪表”，再同步刷新“结论沉淀区”。
2. “当前结论”只使用这几类状态：`已确认`、`未收口`、`未回复`、`后置`。
3. “是否阻塞”只回答当前是否阻塞主线推进，不表达长期优先级。
4. 若后续进入新一轮沟通，复制当前模板并重置“问题跟踪表”中的轮次状态，不要覆盖已形成的历史结论。
