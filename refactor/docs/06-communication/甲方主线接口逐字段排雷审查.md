# 甲方主线接口逐字段排雷审查

## 第一部分：整体判断

当前甲方主线接口文档属于：`主线可看，但埋雷较多`。

原因不是甲方完全没给主线，而是主线动作虽然基本齐了，但关键语义没有闭合，且多处把“主线最小闭环”和“扩展能力 / 展示字段 / 过程明细 / 运维能力”混写在一起。现在如果直接按文档字段重实现，后面最容易返工的不是某个单字段，而是任务对象、控制对象、结果对象、文件对象、报告对象和状态对象的边界一起被拖散。

本轮逐字段排雷的总体结论是：

1. 任务下发、启动、完全停止、用例结果回传、任务报告交付、心跳、状态查询这条主线可以继续往前推，但不能按“文档标必填就先全接”的思路推进。
2. 这轮真正该先想清楚的，是哪些字段会进入我方内部主逻辑，哪些字段只适合默认、透传、挂空或后置，而不是先替甲方补完整协议。
3. 当前最危险的卡点集中在五类字段上：
   - 启动粒度字段：`startTestCaseList.testCaseId`
   - 停止语义字段：`controlTestTask.action`、`handleCode`
   - 结果归类字段：`testCaseResultReport.result`
   - 文件与报告边界字段：`testDataFileTranslationComplete.result`、`fileType`、`filePath`，以及 `TestReport` 的重字段
   - 状态与保活字段：`heartbeat` 是否有响应、`getSubSysState.data` 与 `keyPars`
4. 当前最稳的推进方式不是“补齐所有字段”，而是：
   - 先收住任务级主线
   - 先把每用例结果回传和任务报告交付的最小闭环守住
   - 先把明显会卡人的字段留痕
   - 先把没有稳定数据来源的字段从内部主逻辑里隔离出去

## 第二部分：当前边界与排雷原则

### 1. 这轮已经成立的边界

1. 文档中的“测试集”和“任务”当前按同一个对象理解，先作为本轮内部分析基线使用。
2. 本轮主线只围绕：
   - 任务下发
   - 启动
   - 完全停止
   - 状态查询
   - 心跳
   - 每个用例完成后的结果回传
   - 任务报告交付
3. `stop` 和 `abort` 最终保留哪一个仍需确认，但本轮只接受一个“完全停下”的动作，不主动把两套近义控制一起做进主逻辑。
4. 每个用例完成后单独回结果，不做全过程步骤级持续上报。
5. 用例和设备信息当前按“我方提供给甲方使用”理解，不把完整资产管理和完整设备管理一起带入主线。
6. 当前不默认信任甲方文档，需要始终带着以下风险看字段：
   - 甲方会把没用字段说成必须
   - 甲方会把示例字段说成正式要求
   - 甲方会前文一套、后文一套
   - 甲方会在联调时临时拿边角字段卡人
7. 当前不希望把完整资产管理、完整设备管理、升级重启、参数协同、完整告警、全过程消息一起拉进来。
8. 本轮输出目标不是把 schema 定稿，而是提前识别：
   - 哪些字段真会进入我方主逻辑
   - 哪些字段可以先不接
   - 哪些字段可以默认、透传、挂空或后置
   - 哪些字段必须提前留证据防扯皮

### 2. 本轮排雷原则

1. 只认当前文档和既有讨论材料里可回溯的事实，不把“正常合作方大概会怎样”当事实。
2. 不因为字段在文档里标成必填，就默认它本轮必须实现。
3. 不因为字段暂时用不上，就完全跳过；还要判断它会不会在后续被甲方翻出来卡人。
4. 只要字段没有稳定数据来源，就不能默默拉进我方内部主逻辑。
5. 对同一字段，同时看四件事：
   - 它是否真参与本轮主线
   - 它是否有稳定来源
   - 它是否能降级处理
   - 它是否会成为联调和扯皮抓手
6. 对字段的建议处理方式只使用以下几类，不额外发明复杂方案：
   - `本轮真接`
   - `先默认`
   - `先透传`
   - `先挂空`
   - `后置`
   - `必须和甲方确认`
7. 对甲方当前文档里前后不一致、同层混写、语义冲突的字段，优先写“怎么留证据”，而不是先替甲方补设计。
8. 第三批里的 `getSubSysState` 不在 `07-运维控制与链路保活.md`，而是在甲方原文和拆分文档 `02-设备、状态与告警.md` 中出现；本稿按用户给定审查范围补齐，不把这一处文档分布差异当成可忽略细节。

## 第三部分：分批逐字段排雷审查

### 第一批：任务进入与控制

这一批最容易被坑的点：

1. 甲方把“任务接收”“任务启动”“任务控制”写成三段，但对象边界并没有稳定落在同一层。
2. `setTestTask` 被写得过重，容易把资源、参数、分批、文件链路一起当成本轮前置。
3. `controlTestTask` 同时承载动作语义和处理结果语义，最容易把停止、拒绝、不可执行、异常混成一团。

这一批最可能被甲方写重的字段：

1. `resources`
2. `isEnd`
3. `orbitProtectTime`
4. `ftpInfo`
5. `handleCode`

这一批最可能在联调时被拿来卡人的字段：

1. `startTestCaseList.testCaseId`
2. `controlTestTask.action`
3. `controlTestTask.handleCode`
4. `setTestTask.testCaseInfo.inputPars`
5. `setTestTask.resources`

#### 第一批共用外壳字段

当前事实：任务入口相关接口分别固定为 `setTestTask`、`startTestCaseList`、`controlTestTask`。  
本轮可能怎么用：只作为外部动作标识，不进入内部任务主逻辑。  
我方是否有稳定数据来源：有，外部请求直接提供。

## 第五部分：高风险卡人字段清单

### 1. `startTestCaseList.testCaseId`

为什么它危险：它直接决定启动动作到底命中任务、测试集还是用例。  
甲方最可能怎么卡：前期默认按任务启动，联调时突然要求部分启动或测试集级启动。  
我方最省事的处理策略：先按“任务内用例集合启动清单”理解推进，并要求甲方明确控制粒度。

### 2. `controlTestTask.action`

为什么它危险：`stop` 和 `abort` 同时存在，且语义明显重叠。  
甲方最可能怎么卡：一会儿拿 `stop` 说完全停下，一会儿又拿 `abort` 说必须立即打断当前用例。  
我方最省事的处理策略：本轮只接受一个“完全停下”的动作，请甲方明确唯一口径。

### 3. `controlTestTask.handleCode`

为什么它危险：它把同步处理原因、错误码和后续动作建议揉在一起，但没有正式码表。  
甲方最可能怎么卡：要求按未知完整错误码驱动重试、等待、拒绝等分支。  
我方最省事的处理策略：先把它限定为同步回执原因码，不承诺首轮按错误码驱动复杂控制。

### 4. `testCaseResultReport.result`

为什么它危险：只有 `success / fail / tbd`，无法稳定承载非正常完成语义。  
甲方最可能怎么卡：要求区分拒绝、不可执行、超时、停止未完成，但又不给更细分类。  
我方最省事的处理策略：直接指出当前枚举不足，要求甲方给最小结果分类口径。

### 5. `testDataFileTranslationComplete.result`

为什么它危险：文档备注已经说明它偏文件传输结果，但甲方很容易把它拿来顶替业务交付结果。  
甲方最可能怎么卡：文件传完就认定报告交付完成。  
我方最省事的处理策略：坚持把它限定为文件链路结果，不等同于报告交付成功。

### 6. `testDataFileTranslationComplete.testCaseId`

为什么它危险：它支持多个，说明对象更像文件归属清单，不像单用例结果对象。  
甲方最可能怎么卡：倒逼我方实现复杂文件到多用例映射。  
我方最省事的处理策略：先允许按任务级文件处理，只在明确需要时再细化到用例级。

### 7. `TestReport.checkPoints`

为什么它危险：原文直接把它写成“用例通过与否的判断依据”。  
甲方最可能怎么卡：把它当首轮验收核心字段，要求同时给期望值、测试值、通过结论和说明。  
我方最省事的处理策略：要求甲方先给最小检查点集合和生成规则，否则不承诺首轮必交。

### 8. `TestReport.processAndDatas`

为什么它危险：它已经把全过程步骤、参数配置和步骤结果数据拉进报告。  
甲方最可能怎么卡：把“报告示例”当成首轮过程数据规范。  
我方最省事的处理策略：明确它属于扩展项，本轮不进入主线最小交付面。

### 9. `heartbeatResponse`

为什么它危险：原文自己前后矛盾。  
甲方最可能怎么卡：双方各按单向 / 双向心跳开发，联调时互判失败。  
我方最省事的处理策略：先要求甲方统一唯一交互模型，再实现。

### 10. `getSubSysState.data`

为什么它危险：它会把状态查询变成自检过程详细回传。  
甲方最可能怎么卡：把所有自检步骤、设备状态、关键参数一起压成本轮必回内容。  
我方最省事的处理策略：坚持先交最小整体状态面，若甲方要完整自检明细，请明确最小必回结构。

## 第六部分：现在值得问甲方的字段 / 现在不要问但要留痕的字段
