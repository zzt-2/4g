# 一期对接口径确认参考稿

为减少后续开发和联调中的理解偏差，我们先给出一期对接的默认口径。下面内容不展开完整字段表，也不讨论内部实现方案，只确认任务、设备、结果、报告、状态和文件交付这些会直接影响对接判断的事项。

如无异议，我们后续会按以下口径推进；如某项与接口预期不一致，请直接指出需要调整的部分。

## 一期先处理的范围

一期先围绕任务执行和结果交付处理：

1. 接收任务。
2. 启动任务内 case 执行。
3. 停止当前任务。
4. 查询基础运行状态。
5. 处理心跳。
6. 回传 case 执行结果。
7. 生成任务 JSON 报告。
8. 按约定目录或文件服务交付报告。
9. 如接口需要，发送文件生成或上传完成通知。

## 我们默认的一期流程

我们默认按下面的顺序处理一期任务：

1. 收到 `setTestTask` 后，记录一条外部任务。
2. 任务未启动前处于已接收或待启动状态。
3. 收到 `startTestCaseList` 后，启动当前任务内 case 执行。
4. 执行过程中按 case 记录结果。
5. 任务正常结束时，汇总 task result，并生成 JSON 报告。
6. 收到 `stop` 时，停止当前任务，状态记为 `stopped`。
7. stop 后仍生成 JSON 报告，报告中记录 stopped、已完成 case 的真实结果，以及未执行或中断 case 的未完成状态。
8. 报告生成后，按约定目录保存；如果一期要求上传，再按提供的地址和账号上传。
9. 如果一期要求 HTTP 完成通知，则在文件生成或上传完成后发送通知。

这套流程里，任务执行、结果汇总、报告生成、文件交付是分开的。文件生成完成不等于业务接收完成，文件上传完成也不自动等同于对方业务系统已经处理完成。

## 子系统和设备身份

我们按完整二级子系统接入，不按单个设备或单个执行节点处理。

对外 `deviceId` 按业务设备身份理解。串口、网口、SCOE 通道属于我们内部连接对象，不直接等同于 `deviceId`。

需要确认或提供：

- `subSysId`、`subSysType`、`deviceId`、设备类型、设备地址或编码由哪一方分配。
- 如果我们内部有多个实际连接对象，接口中应体现为几个 `deviceId`。

如果没有额外说明，我们会先按“外部设备身份”和“内部连接通道”分开处理。

## 任务、case 和启动

`setTestTask` 按外部任务下发理解，不等同于我们软件内部已有的本地发送任务。

task 和 case 分两层：task 是任务容器，case 是任务内执行单元；一个 task 可以只包含一个 case。

`startTestCaseList` 一期按启动当前任务内 case 执行处理。

需要确认：

- `startTestCaseList` 一期是否可以按“启动当前任务内 case 执行”处理。
- `startTestCaseList` 中的用例信息是否表示当前 task 内要执行的 case 清单。
- 如果启动对象不是 task 下的 case 清单，请指出实际控制对象。

如果没有额外说明，我们会先按“任务下发后，再启动该任务内 case”处理。

## 任务执行限制

一期我们按单任务串行执行处理。

默认处理方式如下：

- 已有任务运行时，新任务下发先按不可执行处理。
- 已有任务运行时，再次启动先按不可执行处理。
- 同一个 `taskId` 重复下发时，按重复请求处理。
- 未收到或未接受任务时调用启动接口，按不可执行处理。
- 已结束任务再次收到启动或停止请求时，按当前状态返回，不重新执行。

需要确认：

- 同一时间只执行一个任务是否可以满足一期对接。
- 已有任务运行时，新任务下发、重复下发或再次启动是否可以返回不可执行。
- 同一个 `taskId` 重复下发时，是否可以按重复请求处理。

这些限制是为了先保证一期任务链路稳定。

## 停止控制

一期控制动作按 `stop` 处理。

stop 后任务状态按 `stopped` 处理，不按 completed 或 failed 处理。

stop 只表示当前任务被停止，不表示已经正常完成，也不直接表示执行失败。任务中已经完成的 case 保留真实结果；未执行或中断的 case 不记为成功。

需要确认：

- `stop` 是否就是一期唯一需要支持的停止动作。
- stop 后任务状态是否按 `stopped` 返回。
- stop 后是否仍按阶段性报告处理，记录 stopped、已完成 case 和未完成 case。

如果没有额外说明，我们会按上述 stop 口径处理。

## 结果分类

case result 和 task result 分开处理：

- case result 记录单个 case 的执行结果。
- task result 汇总整个任务的执行结果和结束原因。
- stop 不抹掉已完成 case 的真实结果。
- 未执行或中断的 case 不记为成功。

需要确认：

- case result 至少需要哪些分类，例如成功、失败、停止导致未完成、异常。
- task result 至少需要哪些分类和原因说明。
- 是否已有必须使用的结果码、错误码或原因枚举；如果有，请提供一期最小集合。
- 结果和报告中的时间是否可以按我们系统时间记录。
- 如果时间需要统一时区或固定格式，请提供规则。

如果暂时没有固定码值，我们建议先用最小结果分类和简短原因说明。

## JSON 报告和文件交付

我们一期按“任务结束或停止后生成 JSON 报告，再处理文件交付和完成通知”的顺序处理。

一期先把报告是什么、放哪里、是否通知这几个点确认清楚。

需要确认：

- TestReport 是否就是一个 JSON 报告文件。
- JSON 报告一期必须包含哪些最小内容，例如任务层、case 层、必要明细或统计项。
- 是否可以提供一份期望的 TestReport 示例。
- 报告文件名、目录和交付方式由哪一方确定。
- 如果需要上传，请提供测试地址、账号和目录权限。
- 文件生成或上传完成后，是否还需要通过 HTTP 接口通知。

如果没有额外说明，我们会先按“生成 JSON 报告文件，按约定位置保存或上传，必要时发送完成通知”的方式处理。

## 状态查询和心跳

一期我们先按基础状态和心跳对接。

需要确认：

- 状态查询一期是否只需要整体运行状态。
- 任务未运行、运行中、停止、完成、失败等状态名称是否有固定要求。
- 心跳是无响应消息，还是需要返回 `heartbeatResponse`。
- 心跳周期、超时时间和异常判定规则。

如果没有额外说明，我们会先按基础状态处理。

## 联调前资料

进入联调前，还需要同步以下资料：

1. Swagger / OpenAPI，或等价接口描述。
2. 任务下发示例。
3. 任务启动示例。
4. 停止控制示例。
5. 用例结果示例。
6. TestReport 示例。
7. 可直连的测试环境地址。
8. 如接口涉及鉴权，需要鉴权方式、测试账号和网络访问要求。
9. 如涉及 FTP 或其他文件服务器，需要测试环境、账号、目录权限和完成通知方式。
10. 如果有联调验收用例，请提供一期需要通过的最小用例清单。

如果部分资料暂时不能提供，也请说明当前可用的替代方式。

## 后续处理

如无异议，我们将按“一期默认口径”继续推进开发和联调准备。需要调整的地方请直接指出，我们按确认后的口径更新后续实现和测试安排。

如果后续接口文档或口头说明发生变化，也请尽量以更新文档或书面确认同步，避免开发和联调阶段按不同版本理解。
