# 甲方接口返工风险专项审查

## 第一部分：整体判断

甲方当前主线接口属于：`可对接，但返工风险较高`。

判断依据：

1. 主线接口表面上已经覆盖了任务下发、启动、停止、状态查询、心跳、用例结果上报和报告相关回传，说明不是“完全无法落地”的状态。
2. 但任务对象边界、启动粒度、停止语义、结果语义、报告交付语义和心跳交互方式都还没有真正收稳，当前如果直接按文档实现，后续很容易出现对象写偏、状态口径不一致、联调完成判定不稳定的问题。
3. 当前真正的风险不在字段多少，而在接口骨架没有完全闭合：同一对象命名并存，多个接口对同一链路分层不清，且主线回执无法稳定支撑“这一步到底算不算完成”的判断。

## 第二部分：当前事实

以下内容只记录当前文档中已经成立、且可回溯的事实：

1. 本轮主线已经收敛到：任务下发、启动、完全停止、状态查询、心跳、每个用例完成后的结果回传、任务报告交付。
2. 文档中的“测试集”和“任务”当前按同一个对象理解，但甲方接口原文里仍同时出现“测试任务”“测试用例集”“测试集或测试 id”等写法。
3. `setTestTask` 和 `startTestCaseList` 是两个独立接口；`setTestTask.immediate=false` 明确表达了“任务先下发、后启动”的可能。
4. `controlTestTask` 当前同时给出 `pause`、`continue`、`stop`、`abort` 四种动作；本轮已明确只希望保留一个“完全停下”的动作，但最终保留 `stop` 还是 `abort` 仍需确认。
5. `testCaseResultReport` 是单个用例结束后的结果通知接口，当前结果字段枚举为 `success / fail / tbd`。
6. `testDataFileTranslationComplete` 表达的是测试相关文件传输完成后的通知，覆盖测试数据文件和测试报告等文件类型；它本身是文件传输完成通知，不等于报告业务已受理。
7. 测试任务报告文件当前倾向于 JSON，且文档中给出的 `TestReport` 结构包含任务级、用例级和过程明细级三层内容。
8. `heartbeat` 章节同时出现“此消息无响应消息”和 `heartbeatResponse` 响应参数表、响应示例，原文内部存在冲突。
9. `getSubSysState` 属于主线范围内保留的状态查询能力，但当前相关材料已经指出，该接口把整体状态和较重的自检 / 明细信息放在了一起。
10. 当前已明确：用例和设备信息由我方提供给甲方使用；本轮目标不是展开完整资产管理、完整设备管理、升级重启、参数协同、完整告警或全过程消息。
11. 当前沟通原则是：能按最简单方式推进的先按最简单方式推进；只有明显冲突、或会直接影响能否对接和后续返工的问题，才在本轮抬出来。

## 第三部分：返工风险审查

### 1. 对象与命名

#### 1.1 “任务 / 测试集 / 测试用例集”并存

当前事实：`setTestTask` 明确使用“测试任务下发”，`startTestCaseList` 又写“启动之前下发但未执行的测试用例集”，其请求参数 `testCaseId` 还被说明为“测试集或测试 id”。

风险判断：这是当前最直接的一物多名问题，会让任务对象、启动对象和用例对象的边界一起发散。

为什么会导致返工：如果实现阶段先按“任务级启动”建模，后续甲方又把 `startTestCaseList` 解释成“测试集级”甚至“用例级”控制，我方任务上下文、启动命令入参、状态投影和结果关联都会被迫重改；如果一开始按更细粒度实现，又会把本轮本来可以收窄的主线做重。

问题分类：`必须本轮确认`

#### 1.2 同一对象的结果层和文件层命名混杂

当前事实：`testCaseResultReport` 用“测试结果通知”表达单用例结果；`testDataFileTranslationComplete` 用“测试数据文件上传完成”覆盖测试数据文件和测试报告；`TestReport` 又单独定义了任务报告文件结构。

风险判断：当前文档把“结果”“文件”“报告”三个层次都放在任务回传面里，但没有稳定命名来区分“业务结果对象”和“文件交付对象”。

为什么会导致返工：如果当前不拆清，后续我方很容易把用例结果对象、报告对象和文件通知对象写成彼此替代关系，最后在联调时才发现甲方需要的是三种不同语义，再回头拆接口映射、补状态机和补完成判定。

问题分类：`必须本轮确认`

#### 1.3 `stop / abort` 与 `handleCode` 共同表达“停不下来/不能执行”

当前事实：`controlTestTask.action` 同时给出 `stop`、`abort`，响应里又给出 `handleCode` 表达“用例忙”“环境不具备”等处理结果。

风险判断：动作语义和处理结果语义被放在同一控制面里，但边界没有写稳。

为什么会导致返工：如果现在不收口，后续停止、拒绝、不可执行、超时这些外部语义会同时散落在动作枚举、同步回执和异步结果里，代码中会出现多处重复判断，联调时也难以统一“到底是哪一种失败”。

问题分类：`必须本轮确认`

### 2. 任务主链接口

#### 2.1 `setTestTask` 过重，接收动作和执行上下文混在一起

当前事实：`setTestTask` 同时携带 `immediate`、`isEnd`、`resources`、`ftpInfo`、`testCaseInfo.inputPars` 等信息，既像任务入口，也像资源编排和文件链路入口。

风险判断：任务下发接口当前过重，但文档没有明确哪些字段是本轮主线必需、哪些只是增强项。

为什么会导致返工：如果我方按全量结构接入，后续甲方实际只要求最小任务信息时，任务接收对象会被做重；如果先按最小结构接入，又可能在联调时发现 `isEnd`、`resources` 或 `ftpInfo` 被甲方当成主链事实，导致任务组装和校验逻辑重做。

问题分类：`可顺带确认`

#### 2.2 启动动作的控制粒度不稳

当前事实：`startTestCaseList` 使用 `taskId + testCaseId[]` 作为入参，但 `testCaseId` 的说明不是稳定的“用例 id”，而是“测试集或测试 id”。

风险判断：接口名是“启动测试用例集”，参数又可能指向测试集或测试项，意味着启动控制面没有稳定落到单一对象。

为什么会导致返工：启动粒度不稳会直接影响我方如何维护“任务已接收但未启动”“部分启动”“全部启动”这些状态，也会影响停止动作是针对整任务还是某个子集合。如果不先收口，任务系统和外部适配层都会反复改。

问题分类：`必须本轮确认`

#### 2.3 停止后的后续行为没有闭合

当前事实：当前材料已经指出本轮只关心“完全停下”，但原文没有把“停止后是否继续产生结果”“停止后是否仍需交报告”“停止和失败如何区分”写清。

风险判断：停止动作存在，但停止后的主线出口没有稳定口径。

为什么会导致返工：一旦停止后的默认行为不清，我方就无法稳定定义任务终态、用例未完成时的结果表达、报告是否生成、以及甲方看到什么才算“任务结束”。这会同时拖累实现、联调脚本和验收说法。

问题分类：`必须本轮确认`

### 3. 结果、文件与报告链路

#### 3.1 用例结果语义过粗

当前事实：`testCaseResultReport.result` 当前只有 `success / fail / tbd` 三类。

风险判断：单用例结果通知虽然存在，但结果枚举不足以承载本轮已经显性的外部状态差异。

为什么会导致返工：如果继续把拒绝、不可执行、超时、停止未完成都压进 `tbd`，我方后续只能在 `msg` 或报告里补语义，结果通知就失去稳定的验收价值。等甲方要求区分这些情况时，既要改接口映射，也要补历史状态解释。

问题分类：`必须本轮确认`

#### 3.2 文件上传完成通知和报告交付成功判定没有拆开

当前事实：`testDataFileTranslationComplete` 表达文件传输完成；`TestReport` 表达报告内容结构；当前文档没有单独定义“报告交付成功”的稳定回执。

风险判断：文件到位、报告形成、甲方业务已受理三层语义当前没有被正式拆开。

为什么会导致返工：如果我方先把“通知成功”当作“报告交付完成”，后续甲方再补“业务已受理”或“报告校验通过”口径，就必须重做报告交付状态机；反过来如果我方预留过重的交付确认链路，本轮又会平白增加实现和沟通成本。

问题分类：`必须本轮确认`

#### 3.3 报告结构明显重于本轮主线

当前事实：`TestReport` 除任务级和用例级结果外，还包含 `judgmentMsg`、`checkPoints`、`statisticsItems`、`attachItems`、`processAndDatas`、`testParsInfo` 等较重内容。

风险判断：报告对象已经被写成“大而全”结构，但本轮还没有足够依据说明这些过程明细都是首轮必交。

为什么会导致返工：如果现在按完整结构实现，后续很可能发现大量字段在首轮联调中既无数据来源也不参与验收；如果一开始只交最小内容，又可能因为文档没有明确“哪些是可选增强项”而在联调时被追问为什么没交全。

问题分类：`可顺带确认`

### 4. 状态与心跳

#### 4.1 `heartbeat` 章节内部自相矛盾

当前事实：原文同一章节同时写“此消息无响应消息”，又给出 `heartbeatResponse` 参数表和响应示例。

风险判断：这是文档内部已经成立的冲突，不是实现理解差异。

为什么会导致返工：如果不先统一，联调双方会直接按不同模型开发，一边按单向心跳实现，一边按请求响应链路验收，最终连超时、重试、成功判定都无法对齐。

问题分类：`必须本轮确认`

#### 4.2 心跳与参数配置关系会把范围带散

当前事实：当前材料已经指出心跳周期可通过 `setPars` 的 `heartbeatTimer` 配置；但本轮边界又明确不希望把独立参数接口带入主线。

风险判断：心跳本身属于主线保活能力，但它又被文档连接到了本轮不想展开的参数协同体系。

为什么会导致返工：如果本轮把“心跳周期可配”一并追问，很容易顺着带出 `setPars`、参数回执、参数生效时机等更多话题；但如果完全忽略，又会留下“联调到底按什么周期发”的执行空洞。

问题分类：`可顺带确认`

#### 4.3 `getSubSysState` 的最小状态面没有收紧

当前事实：现有材料明确指出，`getSubSysState` 把整体状态和较重的自检 / 过程信息放在了一起。

风险判断：状态查询接口存在，但没有明确本轮最小必需状态面。

为什么会导致返工：如果我方现在按大状态面去建模，后续很容易被过程细节拖进实现；如果只按最小状态面实现，又可能在联调中被要求补更多明细，导致状态对象和回包结构二次返工。

问题分类：`可顺带确认`

### 5. 关联键与完成判定

#### 5.1 主线关联键前后可用，但没有真正闭合

当前事实：主线接口普遍携带 `taskId`、`testCaseId`、`sessionId`、`requestId`，但各接口对这些键的职责没有统一说明；例如启动接口的 `testCaseId` 含义不稳，文件通知里 `testCaseId` 还是数组。

风险判断：关联键看起来很多，但还没有形成一套稳定的“谁标识事务、谁标识任务、谁标识用例、谁标识文件归属”的闭环。

为什么会导致返工：如果关联键职责不先收口，后续实现中就会出现同一对象在不同接口靠不同字段对齐的情况，联调时极易出现“消息收到了，但落不到同一任务上下文”。

问题分类：`必须本轮确认`

#### 5.2 多个主线动作都有接口，但完成判定不稳定

当前事实：任务下发、启动、停止、结果通知、文件通知、报告文件、心跳都已经有接口或文档结构，但当前材料多次指出缺少稳定的“这一步算完成”的统一口径。

风险判断：甲方接口是“动作齐全”，不是“闭环已稳”。

为什么会导致返工：如果完成判定不稳，我方实现阶段无法决定哪些动作只看同步回执、哪些还需要后续异步确认、哪些需要报告层收口。后续一旦甲方补口径，就会牵连状态机、重试逻辑和联调验收标准整体重改。

问题分类：`必须本轮确认`

#### 5.3 重试、重复下发、并发任务没有形成稳定约束

当前事实：已有沟通材料已把“重复下发、重试和并发”列为高优先级确认项，但当前主线原文没有给出稳定规则。

风险判断：这不是字段细节，而是主线事务边界缺失。

为什么会导致返工：如果不先约束，我方无法明确是否需要去重、幂等、并发隔离和控制命令精确命中机制。等联调时再补这层约束，任务上下文管理和控制逻辑会被整体返工。

问题分类：`现在不要问`

## 第四部分：值得本轮抬给甲方的最小问题集

1. `startTestCaseList.testCaseId` 本轮是否仅作为任务内引用，不单独作为独立控制对象；若无特殊要求，我方先按任务级启动理解推进。
为什么现在必须提：这个字段直接决定启动动作到底打在任务、测试集还是用例上。
如果不提，后面会怎么返工：我方很可能先按任务级实现，联调时再被要求拆成更细粒度控制。
建议往哪个更简单的方向收口：本轮先按任务级主线推进，`testCaseId` 只保留为任务内引用。

2. `controlTestTask.action` 本轮是否只保留一个“完全停下”的动作；若无特殊要求，我方先按单一停机动作推进，不同时实现 `stop` 和 `abort` 两套语义。
为什么现在必须提：这个字段直接决定停止控制和任务终态怎么建模。
如果不提，后面会怎么返工：我方会在动作枚举、结果状态和回执原因里重复表达“停下”，后续很难统一。
建议往哪个更简单的方向收口：本轮先收敛成一个“完全停下”的动作，其余差异放到回执说明或后续扩展。

3. `testCaseResultReport.result` 本轮是否至少要区分“成功 / 失败 / 非正常完成”，而不是继续把多种情况都压进 `tbd`。
为什么现在必须提：这个字段决定用例完成后的最小结果口径。
如果不提，后面会怎么返工：我方会先用 `tbd` 混装拒绝、不可执行、超时、停止未完成等状态，后面再拆时要重改结果对象。
建议往哪个更简单的方向收口：本轮先确认最小分类层级，不展开完整枚举和字段 schema。

4. `testDataFileTranslationComplete.result` 是否只表示文件传输完成，不等于报告业务已受理；若无额外业务口径，我方先按“文件通知成功即该步完成”理解。
为什么现在必须提：这个字段决定“文件到位”和“报告交付完成”是不是同一件事。
如果不提，后面会怎么返工：我方可能把文件通知成功直接当成报告交付完成，联调时再发现甲方还要额外确认。
建议往哪个更简单的方向收口：本轮先确认最小完成口径，只补必要的层次区分。

5. `heartbeat` 本轮最终按无响应消息实现，还是按带 `heartbeatResponse` 的请求响应链路实现。
为什么现在必须提：这是原文内部直接冲突，且决定心跳成功判定。
如果不提，后面会怎么返工：双方会按不同交互模型实现，联调时需要重做超时、重试和成功判断。
建议往哪个更简单的方向收口：本轮先固定一种最简单可联调的模式，周期配置后置。

6. `taskId`、`testCaseId`、`sessionId`、`requestId` 本轮各自是否可按以下最小职责理解：`taskId` 标识任务、`testCaseId` 标识任务内用例、`sessionId` 贯穿本次外部事务、`requestId` 标识单次消息请求。
为什么现在必须提：这些字段决定主线对象能不能在不同接口间稳定关联起来。
如果不提，后面会怎么返工：任务、用例、文件、报告会在不同接口中靠不同字段拼接，最后出现“消息到了但归不到同一上下文”。
建议往哪个更简单的方向收口：本轮先确认最小职责，不扩展更多关联规则。

## 第五部分：现在不值得抬出去的问题

1. 不建议现在追问 `setTestTask.resources` 的完整资源结构、`testCaseInfo` 的所有细分字段和 `ftpInfo` 的完整使用场景。
为什么现在问不划算：这些内容很容易把讨论带进资源编排、资产组织和文件治理细节，超出本轮主线。
更适合放到哪个后续阶段：主线对象和最小任务结构确认之后，再进入字段级 schema 或联调细化阶段。

2. 不建议现在展开 `TestReport` 中的 `checkPoints`、`statisticsItems`、`attachItems`、`processAndDatas`、`testParsInfo` 的完整定义。
为什么现在问不划算：这会把当前“先把主线做成”拉回“大而全报告设计”，明显放大工作量。
更适合放到哪个后续阶段：等报告最小必交内容和交付方式确认后，再做报告 schema 细化。

3. 不建议现在把 `pause / continue`、升级、重启、完整告警、完整设备管理、独立参数接口、过程消息、中心反向取文件一起抬出去。
为什么现在问不划算：这些能力都不属于当前主线闭环前置，只会把范围问散。
更适合放到哪个后续阶段：主线联调跑通后，再按能力域分批进入扩展能力确认。

4. 不建议现在追问 FTP 目录命名规范、图片音频展示字段、脚本文件传输细节。
为什么现在问不划算：这些问题不是当前返工风险主因，提前问只会引出更多长期治理话题。
更适合放到哪个后续阶段：当甲方明确本轮确实依赖 FTP 且报告交付口径已稳定后，再进入文件链路细化。

5. 不建议现在把 `getSubSysState` 的详细状态面、自检项和过程明细全部问细。
为什么现在问不划算：状态查询当前只需要支撑最小可用状态面，过早细化会把实现和沟通都做重。
更适合放到哪个后续阶段：主线闭环确认后，结合我方真实状态模型再单独收口。

## 第六部分：本轮已确认结论 / 仍未确认问题 / 需要用户决策的问题

### 1. 本轮已确认结论

1. 本轮主线只围绕任务下发、启动、完全停止、状态查询、心跳、每个用例完成后的结果回传和任务报告交付展开。
2. 文档中的“测试集”和“任务”当前按同一个对象理解，可以作为本轮内部分析基线继续使用。
3. 是每个用例完成后单独回结果，不是全过程步骤级实时上报。
4. 用例和设备信息由我方提供给甲方使用，本轮不应把完整资产管理和完整设备管理带入主线。
5. 本轮应优先识别会导致返工的接口骨架问题，而不是把接口字段一次性问细。
6. 当前已经可以确定的高风险区包括：任务 / 测试集命名并存、`stop / abort` 并存、心跳章节自相矛盾、结果通知 / 文件通知 / 报告交付三层语义混杂。

### 2. 仍未确认问题

1. `startTestCaseList` 的真实控制对象到底是任务、测试集还是用例。
2. 本轮最终保留 `stop` 还是 `abort`，以及停止后是否仍需生成结果和报告。
3. `testCaseResultReport.result` 是否需要在本轮就拆开非正常完成状态。
4. `testDataFileTranslationComplete` 与报告交付成功判定之间到底是什么关系。
5. `heartbeat` 最终是无响应消息还是有 `heartbeatResponse` 的双向交互。
6. `getSubSysState` 本轮最小必需状态面是什么。
7. `taskId`、`testCaseId`、`sessionId`、`requestId` 的职责边界和前后链路归属关系。
8. 当前没有足够依据证明本轮必须支持重试、重复下发和并发任务；若甲方无明确例外，应先按单任务主线、无并发、无额外重试处理理解推进。

### 3. 需要用户决策的问题

1. 对外沟通时，是否继续坚持“本轮只保留一个完全停下动作”的收口原则；如果要先给出偏向，需决定优先推 `stop` 还是 `abort`。
2. 对外沟通时，是否要把“结果最小分类需要进一步拆开”作为本轮明确问题抬出去；这会增加一点确认成本，但能显著降低后续结果返工。
3. 对外沟通时，是否把“主线关联键闭环”列入最小问题集；这类问题抽象度稍高，但确实直接关系后续实现是否写偏。
4. 若甲方当前暂时给不出完整报告口径，是否接受本轮先按“最小报告交付”推进，并把报告重字段后置处理。
