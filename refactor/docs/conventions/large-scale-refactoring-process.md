# 大规模重构流程规范

> 适用场景：涉及数据模型、后端链路、前端交互多层改动的结构性重构
> 来源：R079 元数据重构实践中的教训总结

## 流程总览

```
第一阶段：摸清全量影响面（不动代码）
  ↓
第二阶段：逐个专题讨论收结论（不动代码）
  ↓
第三阶段：汇编 spec（不动代码）
  ↓
第四阶段：实施
```

## 第一阶段：全量影响分析

目标：搞清楚这次重构到底改了哪些维度，每个维度影响哪些链路。

### 必须做的事

1. **从实际代码出发，不从设计文档出发**
   - 先读 Prisma schema / 数据库定义，确认字段真实存储形态
   - 再读运行时代码，确认哪些字段真正被消费
   - spec / 设计文档只作为交叉验证，不作为事实基线
   - "字段存在"不等于"链路已消费"

2. **列出全部变更维度，不只看最显眼的那个**
   - 把每个维度做成一行：数据层改了什么 / 后端影响 / 前端影响 / 是否已对齐
   - 如果只围绕一个维度展开讨论，其余维度一定会被遗漏

3. **产出必须是独立 md 文件**
   - 不能只留在对话里
   - 后续专题以这份文件为输入

### 不能做的事

- 不讨论方案
- 不动代码
- 不预设后续专题拆分（拆分由分析结果决定）

## 第二阶段：专题讨论

目标：把全量分析中识别的每个未对齐点逐个讨论清楚，形成结论。

### 拆分原则

- 拆分计划由第一阶段产出决定，不提前预设
- 按依赖关系排序：底层（持久化）-> 中层（语义边界）-> 上层（消费机制）
- 每个专题只聚焦一个子域，明确写"不讨论 XXX"
- 拆分后画依赖图：标注哪些必须串行（有输入依赖）、哪些可以并行（无依赖）

### 执行方式

- 同一个对话能接着就接着，上下文不够再开新的
- 每个专题产出结论，不只是观察
- 产出后回头修订前序专题中留下的"待比较方向"
- 不需要每个专题都交叉审查，只在方向性分歧时才请第二方审

### 必读文件分级

每个专题的必读文件必须分三档：

- **事实来源**：schema、运行时代码（必须读）
- **按需补充**：相关 DTO / service / 前端代码（需要时读）
- **参考**：前序专题产出、旧讨论产物（交叉验证用）

阅读顺序：先看事实来源（建立正确基线），再看约束文档（知道边界），最后看提示词。

### 执行提示词标准格式

每个专题开新对话时，执行提示词应包含以下 5 段：

```
1. 入口指令
   - 指向提示词文档路径
   - 本轮范围（一句话）
   - 不做什么（不改代码、不改 schema、不进入 XXX）

2. 阅读顺序与文件清单
   - 事实来源文件（附路径，必读）
   - 按需补充文件（附路径，需要时读；未读要说明原因）
   - 最后读提示词文档本身

3. 执行要求
   - 必须优先回答的问题
   - 必须判断的结论
   - 必须区分的层次

4. 输出要求
   - 按提示词结构写
   - 保存到指定路径

5. 特别提醒
   - 本专题容易踩的具体坑
```

## 第三阶段：汇编 spec

目标：把各专题结论合并为正式 spec 文档。

- 不是重新讨论，是汇编
- 如果专题之间有矛盾，开一轮专门解决矛盾
- spec 定稿后才允许进入实施

## 第四阶段：实施

进入实施的前置条件：

- 全部专题结论已确认
- spec 已定稿
- 没有关键疑点
- 链路清晰、用法简单、预期代码质量可接受

## 踩过的坑和对应规则

| 坑 | 规则 |
|:--|:--|
| 围绕最显眼的维度展开四轮讨论，忽略其余五个维度 | 第一阶段必须列出全部变更维度，不能只看一个 |
| 在全量分析前就预设了 5 个子对话 | 拆分计划由第一阶段产出决定 |
| 一个提示词试图覆盖前端+后端+规范 | 大任务必须拆分，每个只聚焦一个子域 |
| 拿 spec 设计文档当事实，结果和实际 schema 不一致 | 事实基线是代码和 schema，不是设计文档 |
| 把"字段存在于 front matter"等同于"运行时已消费" | 必须区分"字段存在"和"链路已消费" |
| 四轮讨论完了才发现方向偏了，只能从头重建 | 第一阶段先确认讨论范围覆盖全部变更 |
| 中间工件只留在对话里，下一轮对话读不到 | 每轮产出必须保存为独立 md 文件 |
| 第一阶段范围不断膨胀，什么都想塞进去 | 明确写死阶段上限和不包含项 |
| tags 看起来什么都承载，实际什么都不稳定 | "可查询"不等于"已接入运行时" |
