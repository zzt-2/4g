# 协作约定分层说明

## 定位

本文档用于说明 Thesis Platform 当前多阶段协作中，各类 conventions 与路由 skill 分别负责什么。

它不是新的总规范，不重复展开各层正文，只回答四个问题：

1. 当前正式采用哪套阶段名。
2. 常见工件与运行层对象各自负责什么。
3. 每份 convention 或 skill 分别承接哪一层。
4. `thesis-workflow-router` 后续压缩时，应保留什么，不应继续承接什么。

本文档的目标是减少规则归属歧义，避免同一类规则在多份文档和 skill 中重复扩写。

## 正式阶段名

当前正式采用以下四个阶段名：

- `分析`
- `收口`
- `实施`
- `回流`

其他叫法只作为映射说明，不作为正式枚举。

常见映射关系如下：

| 正式阶段名 | 常见映射叫法 | 说明 |
|---|---|---|
| `分析` | `analysis`、设计探索、框架澄清 | 重点是摸清事实、锁边界、识别风险 |
| `收口` | `convergence`、方案收口 | 重点是统一口径、排除错误方向、形成稳定结论 |
| `实施` | `implementation`、实施移交 | 重点是按已锁边界落地与验证 |
| `回流` | `backflow`、结论回收 | 重点是把结果同步回主线与长期材料 |

`review` 不单列为一级阶段名，当前视为可嵌入 `实施` 与 `回流` 之间的独立审查环节。

## 对象边界

### 流程对象

#### `多对话协作`

回答“协作怎么走”：

- 什么时候拆线程
- 什么时候交叉审查
- 什么时候回主线
- 什么时候禁止混做实施

#### `交接工件`

回答“下一轮怎么承接”：

- `next-session prompt`
- `状态快照`
- `discussion log`
- `总入口 prompt`

### 运行层对象

#### `workstream`

回答“这个持续专题的运行层如何落地”。

#### `entry.md`

回答“新对话或新一轮接手时，从哪里进入、接下来该做什么”。

#### `state.md`

回答“当前专题状态是什么”。

#### `artifacts-index.md`

回答“当前有效工件有哪些、该看哪份”。

#### `workstream` 下的专题讨论日志

回答“这个专题是如何从讨论走到当前收口结果的”。

它适合放在 `design/workstreams/<workstream-id>/analysis/` 下，承担专题内部的过程基准。

### 长期沉淀对象

#### `discussion log`

回答“哪些稳定边界、正式结论或淘汰路径需要长期保留”。

#### `conventions`

回答“长期复用的规则、流程和写法约定是什么”。

#### `OpenSpec`

回答“需要独立 change 管理的正式变更是什么”。

项目默认口径：

- 当任务决定新增 OpenSpec change 时，默认优先使用 `ff` 路径快速补齐所需 artifacts。
- 不默认采用 proposal → design → specs → tasks 的逐 artifact 交互式推进。
- 若用户明确要求逐步创建 artifact，或当前轮确实需要逐 artifact 审查，才改走逐步路径。
- `explore` 仍保留用于前置探索、范围澄清和进入 change 前的讨论，不因默认 `ff` 而取消。
- 当 `ff` 已补齐到可实施状态后，不在同一对话里直接 `apply`。
- `ff` 所在对话只负责把 change 推到 apply-ready；实施应切到正式新对话再开始。

## 文档分层

### `multi-conversation-workflow.md`

职责固定为：`流程层`。

它应主要回答：

- 什么时候需要分析、实施、审查、回流
- 什么时候该拆对话
- 什么时候该回主线
- 多对话协作的主要模式如何流转

它不应详细承接：

- `entry.md / state.md / artifacts-index.md` 的职责
- `next-session prompt` 的具体写法
- 路由 skill 的判型实现细节

### `next-session-prompt-writing.md`

职责固定为：`工件写法层`。

它应主要回答：

- `next-session prompt` 何时适用
- 交接提示词如何写
- 必读集、输出要求、边界、状态快照如何组织
- `总入口 prompt` 在交接工件体系中的位置

它不应详细承接：

- 多对话整体流转模式
- `workstream` 创建条件
- `entry.md / state.md / artifacts-index.md` 的运行层职责

### `工作流专题入口与运行层约定.md`

职责固定为：`运行层`。

它应主要回答：

- 什么时候创建 `workstream`
- 运行层最小结构是什么
- `entry.md / state.md / artifacts-index.md` 各自负责什么
- `analysis/` 下的专题讨论日志何时需要补
- 新对话默认如何从入口层接手
- 什么时候必须更新入口层与状态层

它不应详细承接：

- 多对话流程模式
- `next-session prompt` 的写法细节
- OpenSpec 升格规则

### `design/notes/*`

职责固定为：`上游草案层`。

它们适合承接：

- 尚未完全稳定的机制原理
- 仍在试跑中的字段设计
- 目录、命名和对象细节
- 后续可能继续调整的中间层协议

它们不应直接替代现行 conventions。

## `workstream` 讨论日志与其他对象的边界

- `workstream` 下的专题讨论日志服务当前专题，重点回答“这次为什么这样收口”。
- OpenSpec 服务正式变更基准，重点回答“最终决定改什么、要求是什么、如何实施”。
- 长期 `discussion log` 服务跨任务复用的稳定结论，重点回答“哪些边界应长期保留”。

当前默认顺序是：

1. 先在 `workstream` 下补专题讨论日志，承接当前专题过程。
2. 若后续发现该结论开始跨任务复用，再升格为长期 `discussion log`。

## `router` 的职责边界

### `thesis-workflow-router` 应保留的内容

只保留运行时必须立即判断的内容：

- 当前轮一级判型
- 是否进入交接分支
- 是否形成持续专题
- 是否应创建或进入 `workstream`
- 是否优先读取 `entry.md`
- 当前轮能否直接实施的最小阻断条件
- 结束前是否必须同步入口层、状态层、交接工件或主线

### `thesis-workflow-router` 不应继续承接的内容

不应继续承接大段正文型规范，例如：

- `entry.md / state.md / artifacts-index.md` 的详细职责解释
- `workstream` 目录结构与文件说明
- `next-session prompt` 的详细写法
- 多对话协作模式的正文描述
- 外部 skill 与子 agent 的长篇附录说明
- 详细 OpenSpec 计数与扩展规则

这些内容应由 conventions 或上游草案层承接。

## 当前使用顺序

当前建议的使用顺序如下：

1. 新任务先由 `thesis-workflow-router` 判断当前轮路径。
2. 若已形成持续专题，则创建或进入 `workstream`。
3. 新对话默认先读 `entry.md`。
4. 再按 `entry.md` 的阅读顺序进入 `state.md`、`artifacts-index.md`、`handoff` 或主真相源。
5. 需要长期沉淀时，再回到 `discussion log`、`conventions` 或 `OpenSpec`。

## 后续修改顺序

后续若要继续收口这套体系，建议按以下顺序推进：

1. 先依据本文调整三份 conventions 的边界。
2. 再压缩 `thesis-workflow-router`。
3. 最后再决定是否需要把更稳定的入口规则抽进其他 skill 或模板。
