# 外部反馈系统化分析规范

> 用途: 收到大量外部反馈（导师意见、审稿意见、内测报告等）时，如何系统化地分类、映射、推导和规划改动。
> 适用: 任何需要将「一批具体意见」转化为「文档/代码改动方案」的场景。

---

## 一、原则

1. **先分类完，再讨论方案**。逐条修补会导致前后矛盾。所有意见录入分类后，才开始按层级讨论。
2. **区分样本和系统**。每条意见是一个样本。要从样本推导系统性问题，而不是只修单点。
3. **控制上下文**。使用紧凑参考文档（工作流地图、文档职责地图）代替原文全量阅读。只在需要验证时读原文。
4. **分层处理**。架构级影响最先确认（可能影响所有后续改动），局部改动最后执行。

---

## 二、工作流程

### 阶段 1: 录入与分类

对每条意见执行：

```
意见原文 → 问题类型 → 影响层级 → 映射文档 → 是否已有规则覆盖
```

#### 问题类型

分两大类：**产出质量问题**（W-前缀）和**系统缺陷**（T/K/A-前缀）。

| 前缀 | 类型 | 含义 |
|:---|:---|:---|
| W-结构 | 产出 | 全文结构（章节缺失/多余/顺序不当） |
| W-论证 | 产出 | 论证逻辑（Gap不清/闭环不完整/过度泛化） |
| W-内容 | 产出 | 具体内容（数据/图表/公式/示例） |
| W-表达 | 产出 | 措辞/用词/语体 |
| W-格式 | 产出 | 排版/引用/标注格式 |
| T-模板 | 系统 | 产出格式骨架缺陷 |
| T-指令 | 系统 | instruction 约束不足/错误/遗漏 |
| T-路由 | 系统 | 数据路由（信息不足/冗余/优先级不当） |
| K-缺失 | 系统 | 文档库缺少对应规则/示例 |
| K-不足 | 系统 | 文档库有规则但不够充分/具体 |
| K-错误 | 系统 | 文档库规则本身有误 |
| A-架构 | 系统 | 工作流架构（步骤设计/分工/参数传递） |

#### 影响层级

| 层级 | 范围 | 示例 |
|:---|:---|:---|
| L0-局部 | 改一个文档 | 增加一条规则 |
| L1-跨文档 | 改多个文档保持一致 | 增加新写作模式 |
| L2-跨步骤 | 改 instruction + knowledge | 改变某章节的写作策略 |
| L3-架构级 | 改工作流定义/步骤设计 | 新增步骤或改变分工 |

### 阶段 2: 映射到文档体系

对每条意见，定位责任文档。

**映射逻辑**:
- 问题发生在哪个步骤 → 查该步骤的 instruction 和注入的文档库文档
- 问题属于哪个知识领域 → 查文档职责地图中的「关键词」和「典型触发」
- 判断「规则缺失」vs「规则未执行」：

| 情况 | 对策 |
|:---|:---|
| 有规则但 AI 没遵守 | 检查 instruction 引导 + 注入优先级 |
| 有规则但规则有问题 | 修改规则文档 |
| 没有对应规则 | 新增规则/示例 |
| 规则正确但模板不支持 | 修改模板 |
| 模板正确但分工不当 | 修改 instruction 或步骤设计 |

### 阶段 3: 衍生问题推导

从具体意见推导系统性问题。四条推导路径：

| 方向 | 问题 |
|:---|:---|
| 向上追溯 | 根源是更上游的什么决策缺失？ |
| 横向扩展 | 同类问题在其他步骤/章节是否也存在？ |
| 向前预见 | 不修的话，加新领域/新类型时会怎样？ |
| 反事实测试 | 如果文档库有完美规则，AI 能避免吗？还是 instruction/路由也有责任？ |

### 阶段 4: 方案设计

改动方案按影响层级分组。每项标注：
- 改什么文件 + 改什么内容
- 连锁影响（改 A 是否导致 B 也要改）
- 优先级（P0 必须 / P1 建议 / P2 可选）
- 是否需要框架级变更（需独立确认）

如果改动超过单次对话可管理的量（通常 >5 文件或 >3 层级），拆分子对话并标注依赖关系。

## 三、上下文管理

### 紧凑参考优先

大量反馈讨论容易上下文过载。应优先使用紧凑参考文档：

| 文档 | 用途 |
|:---|:---|
| 工作流参考地图（如 `journal-workflow-map.md`） | 步骤链/数据流/instruction 摘要/文档库索引 |
| 文档职责地图（`knowledge-base-document-map.md`） | 每个文档的职责/关键词/边界 |
| 文档库目录（`knowledge-base-catalog.md`） | 文件列表和大小 |

**只在以下情况读取原文**：
- 需要确认某个规则的具体措辞
- 需要检查模板骨架是否覆盖了某种情况
- 需要验证某条 instruction 的具体约束

### 提示词文档编写

为外部反馈分析编写提示词文档时，应包含：

1. **背景段**: 工作流概述 + 文档体系概述（或指向参考地图）
2. **分类体系**: 上述问题类型和影响层级（可根据领域裁剪）
3. **紧凑上下文**: 列出参考文档路径，明确说明不需要全量读取
4. **分层讨论约束**: 强调先分类后讨论、架构级先确认
5. **产出格式**: 意见分类表 + 衍生推导 + 改动方案表 + 子对话划分建议
6. **约束**: 本对话不执行改动 / 不遗漏意见 / 考虑扩展性

## 四、产出格式模板

### 意见分类表

```markdown
| # | 意见摘要 | 类型 | 层级 | 映射文档 | 已有规则 |
|---|---------|------|------|---------|---------|
| 1 | ... | W-论证 | L1 | 全文论证流.md | 有但不够具体 |
```

### 衍生问题

```markdown
### D-01: [问题标题]

来源意见: #1, #5, #12
直接问题: ...
系统根因: ...
影响范围: L2
涉及文档: ...
```

### 改动方案表

```markdown
| # | 改什么 | 文件 | 连锁影响 | 优先级 | 框架变更 |
|---|-------|------|---------|-------|---------|
| 1 | ... | ... | ... | P0 | 否 |
```
