# 与甲方沟通确认项（短版）

您好，我们这边想先和贵方把后续对接的几个关键点对齐一下。

当前我们的想法是，先尽量把主线闭环和接口事实说清楚；如果条件合适，后续最好可以进入联调；如果当前阶段暂时不方便直接联调，也可以先把接口理解、所需资产和后续推进条件统一起来。

我们建议本轮先聚焦以下几项：

1. 测试任务下发
2. 任务启动
3. 任务停止
4. 状态查询
5. 心跳
6. 每个用例完成后的结果上报
7. 任务完成后的报告交付
8. 用例和设备的最小引用关系

我们这边希望先确认的关键点主要是：

1. 任务下发和任务启动是否按两个独立动作处理
2. 启动和停止是否都按任务级处理
3. 任务、用例、设备分别通过什么标识关联，后续结果和报告里最终按哪套标识识别
4. 是否存在重复下发、重试下发或并发任务；如果存在，按什么规则处理
5. 心跳到底是单向上报还是双向交互，是否有响应体
6. 每个用例完成后是否单独回结果；如果接口本身有正常响应，是否可以先按“调用成功即完成本步”理解
7. 报告最终通过什么方式交付；如果按约定方式传上去且接口或通知链路有正常响应，是否可以先按“报告交付完成”理解

如果后续希望进入联调，我们建议优先补齐这些条件：

1. 可直连的测试环境地址
2. 鉴权方式、账号和网络访问要求
3. Swagger / OpenAPI，或等价的接口说明与示例材料
4. 如果报告或文件链路仍依赖 FTP，需提供可用地址、账号、目录权限和基本使用方式

另外，按我们当前理解，用例和设备信息是由我方提供给贵方使用。本轮我们更希望先确认主数据归属和最小引用方式，不先展开完整的用例和设备管理能力。

如果贵方当前阶段暂时不方便直接联调，我们建议也可以先按最简单方式推进：先把主线范围、关键接口交互事实和所需接口材料对齐，等环境和条件具备后，再进入联调。
