# 对接准备沟通确认项

为便于后续对接顺利推进，我们希望先和贵方把本轮范围、关键接口交互方式、联调前提和后续可协同方式对齐。我们的总体原则是：能按最简单方式先跑通的，就先按最简单方式理解和推进；只有文档里存在明显冲突、或者会直接影响能不能对接时，再把问题单独提出来。

我们的期望是，如果条件允许，后续最好可以进入联调；如果当前阶段暂不方便直接联调，也希望先把可确认的接口事实、所需资产和后续推进条件统一清楚。

## 第一部分：本轮沟通重点

### 1. 本次沟通目的

本次沟通主要希望先对齐四类事项：

1. 本轮建议优先对齐的主线范围
2. 任务、状态、心跳、结果、报告几个关键链路的交互方式
3. 用例、设备等主数据的归属和最小引用方式
4. 后续如果进入联调，需要具备哪些前提条件

我们当前更关注先把主线闭环和外部事实对齐，再逐步展开扩展能力，不希望一开始同时把资产维护、运维控制和长期平台能力全部拉进来。

### 2. 建议本轮先聚焦的范围

结合当前文档和我方准备情况，建议本轮先聚焦以下主线能力：

1. 测试任务下发
2. 任务启动
3. 任务停止
4. 状态查询
5. 心跳
6. 每个用例完成后的结果上报
7. 任务完成后的报告交付
8. 用例和设备的最小引用关系

如果双方先把这条闭环跑通，后续再扩展参数协同、完整资产维护、升级重启、完整告警等能力，会更容易收敛。

### 3. 建议优先补齐的联调环境与接口资产

如果后续希望进入联调，建议优先补齐以下前提：

1. 可直连的测试环境地址，而不是仅有本地部署示例地址
2. 鉴权方式、联调账号和网络开通要求
3. 联调入口、对接联系人和问题反馈路径
4. Swagger / OpenAPI，或等价的在线接口描述
5. 如本轮仍依赖 FTP / 文件传输链路，需同步提供可用的地址、账号、目录权限和成功判定方式

### 4. 建议优先对齐的关键问题

建议本轮先优先对齐以下问题：

1. 任务下发和任务启动是否按两个独立动作处理
2. 启动和停止是否都按任务级处理
3. 任务、用例、设备分别通过什么标识关联，甲方最终消费哪套标识
4. 是否存在重复下发、重试下发、并发任务等情况；若存在按什么规则处理
5. 心跳到底是单向上报还是双向交互，是否有响应体
6. 每个用例完成后上报的结果，是否需要把成功、失败、拒绝、不可执行、超时等语义区分开
7. 任务报告最终通过什么方式交付，是否调用成功即可视为完成
8. 如果后续进入联调，最低完成标准和判断依据是什么

### 5. 本轮建议暂不展开的内容

为避免本轮范围过大，建议以下内容暂不作为本轮前置项：

1. 完整用例资产维护
2. 脚本上传 / 下载 / 修改全链路
3. 独立参数查询 / 配置和参数转发
4. 过程消息实时上报
5. 升级 / 重启
6. 完整设备管理
7. 完整告警体系
8. 中心反向取文件

如果贵方认为其中某项确实需要提前展开，也建议优先说明必要性和预期目标，便于双方及时收敛范围。

## 第二部分：具体接口讨论

### 1. 任务下发与控制

#### 1.1 测试任务下发

优先级：高

从当前文档看，测试任务下发应该是整条链路的起点，这一点我们理解是一致的。

这里我们最想先确认三件事：

1. 任务下发后，是不是先进入“已接收”状态，后面再单独启动
2. 当前任务下发里带的资源信息、分批信息、FTP 信息，本轮是不是都必须要用；如果不是，能不能先按最小必要信息推进
3. 本轮是否可以先按“任务标识 + 用例列表 + 参数 + 最小必要上下文”这套最小信息来对接

我们的建议是：本轮先按最小必要信息推进。只要能把任务接进来、识别清楚、后续能启动执行，就先不要把额外信息一起做重。

#### 1.2 任务启动

优先级：高

当前文档里，任务启动看起来和任务下发是分开的。

目前按我们的理解，文档里的“测试集”和“任务”说的是同一个对象。因此这里我们建议本轮直接按“任务级启动”理解，不再把“测试集”和“任务”拆成两套概念。

如果贵方这边也是这个意思，那这部分我们建议就不再继续展开，直接按“任务级启动”推进。

#### 1.3 任务停止与终止

优先级：高

当前文档中已经有停止、终止这些控制动作，但从本轮目标看，我们更希望只保留一个“完全停下”的动作，不希望同时保留两套近似语义。

我们这里最希望先对齐的是：

1. 本轮最终保留 `stop` 还是 `abort`
2. “拒绝执行”“当前不可执行”“超时”这些情况分别算哪一类结果
3. 任务被完全停下以后，是不是还要形成结果，是否还要交报告

我们的建议是：本轮优先先把“完全停下”这个动作统一下来，其他更细的控制语义如果当前不是必须项，可以先不展开。

#### 1.4 暂停 / 继续

优先级：低

从本轮目标看，我们建议暂停 / 继续先不纳入第一轮。

原因很简单：如果本轮连启动、停止、结果、报告这条主线都还没收稳，就先把暂停 / 继续加进来，范围会一下子变大很多，测试场景也会复杂很多。

#### 1.5 重复下发、重试与并发

优先级：高

这部分当前文档里如果没有写清，后面很容易在联调时反复扯皮。

我们建议至少先确认：

1. 同一个任务是否可能重复下发
2. 如果请求超时或甲方重试，是否可能再次发同一个任务
3. 本轮是否只按“同一时刻一个活动任务”理解，还是可能并发多个任务

我们的建议是：如果贵方当前没有明确并发要求，本轮先按最简单的单任务主线理解；如果存在重试或重复下发，也请提前说明去重口径。

### 2. 状态查询与心跳

#### 2.1 状态查询

优先级：中

我们理解当前文档里已经有状态查询能力，但现在状态查询里既有整体状态，也带了不少自检过程和明细信息。

这部分我们这轮先不细展开。我们的想法是，后面可以再结合我方实际状态面和贵方接口要求，单独对齐一个更具体的状态口径。也就是说，本轮先不在状态细节上花太多精力。

#### 2.2 心跳

优先级：高

心跳这一块是我们当前最想先说清楚的地方之一。

原因是当前文档里，一处写“心跳没有响应”，后面又给了响应示例。这样看下来，我们没法判断到底应该按哪种方式理解。

这里建议直接确认四件事：

1. 心跳是我们主动发，还是双方一来一回
2. 心跳到底有没有响应体
3. 什么情况下算心跳成功
4. 心跳周期是不是本轮就必须一起确认

我们的建议是：如果贵方当前没有特殊要求，心跳先按最简单的可调用方式理解；只有“无响应”和“有响应”这件事需要先统一，不然双方没法按同一种方式做。

### 3. 用例、设备与最小引用关系

#### 3.1 用例和设备由谁提供、谁维护

优先级：高

按我方当前理解，用例和设备信息由我方提供给贵方使用。

这里建议再明确两件事：

1. 本轮是否按这个归属关系推进，不再要求先打通完整的用例和设备管理接口
2. 如果后续用例或设备信息有变化，双方按什么方式同步

我们的建议是：本轮先只确认主数据归属和最小同步方式，不把完整资产管理能力带进来。

#### 3.2 任务、用例、设备如何在接口里对应起来

优先级：高

这部分不需要现在细到字段级，但建议先把最基本的关联关系说清楚。

我们建议至少确认：

1. 任务里引用用例时，以什么标识为准
2. 用例里引用设备时，以什么标识为准
3. 甲方后续在结果、报告、状态里最终识别的是任务 ID、用例 ID、设备 ID，还是名称加 ID 的组合

如果这部分不先说清楚，后面就算接口打通，也容易出现“能调通但对不上对象”的问题。

#### 3.3 是否需要最小设备信息

优先级：中

我们当前不建议把完整设备管理接口纳入本轮，但如果后续状态查询、结果上报或报告里需要带设备信息，建议先确认最小必要范围。

比如可以先明确：

1. 本轮是否只需要最小设备标识
2. 是否需要附带设备名称、类型或少量状态信息

我们的建议是：本轮如无必要，先只保留最小设备标识，不把完整设备对象一起带重。

### 4. 用例结果上报与报告交付

#### 4.1 用例结果上报

优先级：高

从当前文档理解，用例跑完以后要把结果回给贵方，这一点我们是认可的。

这里最需要先讲清楚的，不是字段细节，而是结果到底怎么分。

我们建议先确认：

1. 是不是每个用例完成后就单独回结果
2. 结果是否至少需要区分成功、失败、拒绝、不可执行、超时这几类
3. 这个结果上报接口本身是否有正常响应；如果有，是否以接口调用成功作为本轮默认完成口径

我们的建议是：这个接口如果本身有正常响应，本轮先按“调用成功即完成上报”理解；如果贵方还有额外口径，再单独补充说明。

#### 4.2 文件上传完成通知

优先级：低

当前文档里，文件上传完成通知覆盖了测试数据文件、报告文件等多类场景。

这部分我们想先按最简单的方式理解：文件传完以后，调用对应通知接口，如果接口有正常响应，本轮是否就可以按“通知完成”处理。

如果贵方这边对“文件传完”和“业务正式接收”还有进一步区分，也请一并说明。若没有额外区分，我们希望先按最简单的调用成功口径推进。

#### 4.3 任务报告

优先级：低

当前文档中已经给出了报告文件结构，这对我们理解整体方向是有帮助的。

不过现在报告内容比较重，除了任务和用例结果，还有过程明细、分析定位、附件信息等很多内容。

我们的建议是：本轮先确认报告最少必须包含什么，不先把所有扩展内容都带上。

比如可以先对齐：

1. 任务级基本信息
2. 用例级结果汇总
3. 必要的结果说明

至于过程明细、分析定位、附件这些内容，如果贵方觉得不是本轮前置项，建议后面再逐步补齐。

#### 4.4 报告交付成功判定

优先级：高

这部分是我们最希望优先对齐的点之一。

这里建议直接确认：

1. 报告最终是通过接口回传、文件上传，还是上传后再通知
2. 如果我们按约定方式把报告传上去，并且接口或通知链路有正常响应，本轮是否就可以按“报告交付完成”处理
3. 如果贵方还有额外的业务完成口径，也请直接说明

这样做的目的是先避免把问题问得过重，优先把本轮能落地的最小交付行为先对齐。

### 5. 建议的沟通顺序

为提高沟通效率，建议本轮按以下顺序推进：

1. 先确认本轮范围，以及用例、设备等主数据归属
2. 再确认任务下发、启动、停止、重复下发和并发约束
3. 接着确认状态查询和心跳交互方式
4. 然后确认用例结果上报、报告交付方式和默认完成口径
5. 最后确认如需联调，还缺哪些环境、账号、网络要求、Swagger / OpenAPI 和 FTP 条件
