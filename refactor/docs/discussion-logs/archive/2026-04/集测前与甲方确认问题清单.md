# 集测前与甲方确认问题清单

> 文档定位：甲方沟通 / 集测前确认清单
>
> 适用阶段：内部架构梳理已基本收口，准备进入北向协同事实确认、集测组织确认和验收边界确认阶段。
>
> 使用原则：本清单只用于确认外部事实、协同方式和验收口径，不替甲方拍板协议细节，不提前冻结接口 schema，不把内部对象直接等同为对外对象。

## 第一部分：为什么现在要和甲方确认这些问题

这份清单的用途，是在内部架构边界已经基本稳定的前提下，把仍然依赖甲方确认的外部事实一次性问清，避免后续 `04-specs`、联调安排和集测验收建立在猜测上。

现在要问，原因主要有三点：

1. 内部架构已经基本收口，但北向协同事实还没有闭合。
如果这一步不补齐，后面很容易把“中心侧怎么协同”误写成“我方内部准备怎么实现”。

2. 当前真正悬空的不是内部骨架，而是外部交互方式、结果交付方式和验收口径。
这些问题不问清，后面写 spec 时最容易失真的，就是任务接入对象、状态与心跳投影对象、结果上报对象、报告交付对象以及对外回执语义。

3. 集测不是单纯接口联通，还包含环境、入口、配合方式和成功标准。
如果只问技术接口、不问集测组织和验收边界，后面即使接口能打通，也可能因为回执口径、报告交付方式或验收标准不一致而反复返工。

其中，如果现在不问，会直接影响 spec 或集测的关键问题主要包括：

1. 甲方如何下发任务，任务接收与启动是否是同一个外部事务。
这会直接影响任务接入与控制语义的 spec 边界。

2. 启动、停止、查询、心跳分别怎么交互，分别是同步请求还是异步协同。
这会直接影响北向接入对象组和北向投影对象组的拆分方式。

3. 用例结果上报和任务报告交付分别怎么做，回执怎么判成功。
这会直接影响结果上报对象、报告交付对象和对外交付失败语义。

4. 集测时甲方提供什么环境、接口、账号、联调入口和样例数据。
这会直接影响联调准备、集测排期和问题归因方式。

5. 集测的完成标准是什么，验收看哪些结果。
这会直接影响我们后续 spec 的验收视角，也决定哪些能力必须优先做成闭环。

## 第二部分：甲方沟通问题清单

以下问题按主题分组。每个问题都标注优先级，并说明为什么要问、如果不确认会卡住什么。

### 1. 北向协同方式

#### 1.1 本轮集测阶段，甲方要求打通的北向协同范围具体包括哪些动作？

- 优先级：必须现在确认
- 为什么要问：要先确认本轮范围到底包含任务下发、启动、停止、状态查询、心跳、用例结果上报、报告交付中的哪些，不然很容易把后续阶段内容提前混入本轮。
- 不确认会卡住什么：会直接卡住北向协同 spec 的范围边界，也会影响集测排期和验收项定义。

#### 1.2 甲方北向协同是直接对接甲方中心系统，还是先经过网关、适配层或第三方平台？

- 优先级：必须现在确认
- 为什么要问：这决定我方实际面对的是单一协同对象还是多层中转对象，影响接入方式、环境准备和问题归因。
- 不确认会卡住什么：会卡住联调入口确认和环境准备，也会让后续 spec 错把中转约束当成业务协同事实。

#### 1.3 任务下发是甲方主动推送给我方，还是我方按约定方式主动拉取？

- 优先级：必须现在确认
- 为什么要问：任务进入方式不同，北向接入边界和集测验证路径完全不同。
- 不确认会卡住什么：会直接卡住任务接入对象组的写法，也会影响集测时谁先发起动作。

#### 1.4 甲方是否要求所有北向动作都走同一条协同通道，还是不同能力走不同入口？

- 优先级：最好尽快确认
- 为什么要问：如果任务接入、状态查询、结果上报、报告交付分别走不同入口，后续沟通和集测安排要按能力拆开准备。
- 不确认会卡住什么：会影响联调清单拆分和接口分组方式，也容易导致后续 spec 把不同性质的协同动作混成一类。

### 2. 任务接入与控制语义

#### 2.1 甲方下发任务后，是“接收任务”和“启动任务”两个独立动作，还是任务一接收就默认开始执行？

- 优先级：必须现在确认
- 为什么要问：这是任务主链最核心的外部事实，决定任务接入和控制语义是否拆分。
- 不确认会卡住什么：会直接卡住北向任务接入对象与控制对象的边界，也会让后续启动流程 spec 失真。

#### 2.2 停止任务是按任务粒度控制，还是允许精确到某个用例或某个执行阶段？

- 优先级：必须现在确认
- 为什么要问：停止语义决定我方只需要处理任务级停止，还是要支持更细粒度控制。
- 不确认会卡住什么：会影响生命周期控制 spec、联调场景设计和验收用例设计。

#### 2.3 同一时刻甲方是否只会有一个活动任务，还是可能并发下发多个任务？

- 优先级：必须现在确认
- 为什么要问：这会直接影响任务上下文与集测组织假设，尤其影响集测是否只验证单任务主线。
- 不确认会卡住什么：会卡住任务接入范围判断，也会让后续 spec 对任务上下文容量作出错误假设。

#### 2.4 甲方是否可能重复下发同一个任务，或在任务执行过程中补发控制请求？

- 优先级：最好尽快确认
- 为什么要问：重复任务和补发控制会引出幂等、去重和状态允许性判断。
- 不确认会卡住什么：会影响接入响应语义、错误语义和联调异常场景设计。

#### 2.5 对于“当前状态不允许启动 / 停止”的情况，甲方期望我方如何反馈？

- 优先级：最好尽快确认
- 为什么要问：这是控制语义和外部响应语义的交叉点，必须先确认甲方接受什么口径。
- 不确认会卡住什么：会影响控制请求响应口径，也会让联调时的异常处理没有统一判定标准。

### 3. 状态查询与心跳

#### 3.1 状态查询是甲方主动查询我方，还是我方向甲方主动推送状态？

- 优先级：必须现在确认
- 为什么要问：状态查询的发起方不同，决定它属于接入层读取动作还是交付层主动动作。
- 不确认会卡住什么：会直接卡住北向查询对象的边界定义，也会影响集测流程编排。

#### 3.2 心跳是我方向甲方主动发送，还是甲方发起保活请求、我方回响应？

- 优先级：必须现在确认
- 为什么要问：心跳和保活在边界上差别很大，必须先确认真实协同方式。
- 不确认会卡住什么：会直接卡住心跳对象和保活语义的 spec，也会影响联调责任分工。

#### 3.3 甲方希望查询结果和心跳中最少看到哪些状态类别？

- 优先级：必须现在确认
- 为什么要问：虽然现在不写字段 schema，但必须先确认对外最小状态面到底要表达哪些事实。
- 不确认会卡住什么：会卡住状态投影对象的范围，也会影响验收时“状态正常”如何判定。

#### 3.4 心跳是否只需要带在线和基础运行状态，还是还要带当前任务摘要？

- 优先级：最好尽快确认
- 为什么要问：这决定心跳是纯保活视图，还是带最小任务视图。
- 不确认会卡住什么：会影响心跳投影对象的粒度，也会影响后续联调验证点。

#### 3.5 状态查询和心跳的频率、超时和重试口径是否已有甲方要求？

- 优先级：可以联调时确认
- 为什么要问：这属于交互参数层面的事实，最好尽早拿到，但不一定阻塞本轮清单和边界判断。
- 不确认会卡住什么：短期主要影响联调细节和集测压测安排，不必阻塞第一版边界 spec。

### 4. 用例结果上报

#### 4.1 甲方要求用例结果按“每个用例完成即上报”，还是允许任务内批量上报或延迟汇总后上报？

- 优先级：必须现在确认
- 为什么要问：这决定结果上报是实时单条交付，还是支持批量或补报模式。
- 不确认会卡住什么：会直接卡住用例结果上报对象和交付语义，也会影响集测用例编排。

#### 4.2 用例结果在甲方口径里，至少需要哪些关键字段类别？

- 优先级：必须现在确认
- 为什么要问：现在不需要写 schema，但必须先确认字段类别范围，例如任务标识、用例标识、执行状态、时间信息、结果说明、结果值或附件引用是否必需。
- 不确认会卡住什么：会直接卡住结果对象组的字段边界，也会让后续 spec 只能凭猜测写字段。

#### 4.3 用例结果上报成功的判断口径是什么？

- 优先级：必须现在确认
- 为什么要问：必须区分“接口收到了”就算成功，还是“甲方业务系统确认受理”才算成功。
- 不确认会卡住什么：会卡住交付回执语义、重试策略假设和集测成功判定。

#### 4.4 如果某个用例结果上报失败，甲方是否接受补报、重报或任务结束后统一补交？

- 优先级：最好尽快确认
- 为什么要问：这会影响上报失败后的联调策略和验收时如何处理局部失败。
- 不确认会卡住什么：会影响交付失败语义和联调补救流程，但不一定阻塞第一版架构边界。

### 5. 报告生成与报告交付

#### 5.1 任务报告最终由我方通过什么方式交付给甲方？

- 优先级：必须现在确认
- 为什么要问：这是当前最关键的悬空事实之一，必须明确是接口回传、文件上传、生成后通知下载，还是其他方式。
- 不确认会卡住什么：会直接卡住报告交付对象组和集测准备，也会让后续 spec 在交付边界上失真。

#### 5.2 甲方要求哪些任务结束场景必须生成报告？

- 优先级：必须现在确认
- 为什么要问：要确认只有自然完成才生成报告，还是停止、异常中止、部分完成也需要报告。
- 不确认会卡住什么：会影响报告生成边界、结果闭环判断和验收场景覆盖。

#### 5.3 甲方对任务报告最少要求哪些内容类别？

- 优先级：最好尽快确认
- 为什么要问：即使不先落字段 schema，也要先确认报告是否至少包含任务基本信息、整体结果、用例结果汇总、异常说明等内容类别。
- 不确认会卡住什么：会影响报告对象边界和验收视角，也会让后续 spec 难以确定报告对象最小范围。

#### 5.4 报告交付成功、交付失败、暂不可交付，甲方分别按什么口径判定？

- 优先级：必须现在确认
- 为什么要问：报告对象已经形成，不等于报告已经交付成功，必须先确认交付层回执语义。
- 不确认会卡住什么：会直接卡住报告交付回执对象和集测结果判定。

### 6. 错误 / 回执 / 拒绝语义

#### 6.1 在甲方口径里，“成功 / 失败 / 拒绝 / 不可执行 / 超时 / 异常”分别怎么定义？

- 优先级：必须现在确认
- 为什么要问：这几个词看起来接近，但集测和验收时一旦口径不一致，最容易反复争议。
- 不确认会卡住什么：会直接卡住北向响应语义和验收判定，也会导致 spec 提前把内部错误映射到外部语义。

#### 6.2 哪些结果应该作为同步响应返回，哪些应该作为异步回执或后续交付结果返回？

- 优先级：必须现在确认
- 为什么要问：同一个动作里，接入成功与业务完成成功往往不是一回事，必须区分同步响应和异步结果。
- 不确认会卡住什么：会直接卡住接入层与交付层对象拆分，也会影响联调时对“成功”的理解。

#### 6.3 对于参数不完整、状态不允许、任务不存在、重复任务、环境不可用这些情况，甲方分别希望归到哪类对外语义？

- 优先级：最好尽快确认
- 为什么要问：这些是最常见的联调异常分支，最好提前统一归类。
- 不确认会卡住什么：会影响异常场景联调和测试脚本口径，但可以晚于主线事实确认。

#### 6.4 超时是只指接口交互超时，还是也包括任务执行超时和报告交付超时？

- 优先级：可以联调时确认
- 为什么要问：这属于异常细分问题，建议提前留意，但可以在主线跑通后结合真实联调再收紧。
- 不确认会卡住什么：短期主要影响异常细分和超时测试脚本，不阻塞主线 spec。

### 7. 集测环境与联调方式

#### 7.1 甲方本轮集测会提供哪些环境条件？

- 优先级：必须现在确认
- 为什么要问：至少要确认联调环境地址、认证方式、测试账号、网络开通要求、白名单或安全策略是否齐备。
- 不确认会卡住什么：会直接卡住联调准备和集测排期。

#### 7.2 甲方是否会提供可用于联调和集测的样例任务、样例结果、报告示例或模拟入口？

- 优先级：必须现在确认
- 为什么要问：没有样例任务和示例数据，很多事实会停留在抽象层，联调效率会非常低。
- 不确认会卡住什么：会卡住联调验证路径和异常场景准备，也会降低 spec 对外事实的准确性。

#### 7.3 集测期间双方的联调入口和问题闭环方式是什么？

- 优先级：最好尽快确认
- 为什么要问：需要确认谁负责答疑、问题通过什么渠道反馈、出现阻塞时找谁、是否有固定联调窗口。
- 不确认会卡住什么：会影响问题定位效率和集测节奏，但不阻塞第一版边界文档。

#### 7.4 甲方是否提供日志、抓包、回放数据或接口调用记录，用于定位对接问题？

- 优先级：最好尽快确认
- 为什么要问：集测阶段最怕双方各自看到一半事实，最好提前确认可共享的诊断材料。
- 不确认会卡住什么：会影响联调问题定位和责任归因效率。

### 8. 验收口径与成功标准

#### 8.1 甲方认定“本轮集测通过”的最低标准是什么？

- 优先级：必须现在确认
- 为什么要问：必须先确认是“接口可通”就算通过，还是必须完成“任务下发 -> 执行 -> 用例结果回传 -> 报告交付”的整条闭环。
- 不确认会卡住什么：会直接卡住验收目标定义，也会影响我方 spec 的优先级排序。

#### 8.2 本轮验收必查的能力项有哪些？

- 优先级：必须现在确认
- 为什么要问：要确认甲方是重点看任务主链、状态查询、心跳、结果上报、报告交付，还是只验其中一部分。
- 不确认会卡住什么：会影响集测准备重点，也会让我们对“哪些能力必须现在做成闭环”判断失真。

#### 8.3 验收时，甲方会以哪些结果作为成功证据？

- 优先级：必须现在确认
- 为什么要问：需要确认是以接口回执、甲方系统页面展示、任务报告入库、日志记录还是现场截图为准。
- 不确认会卡住什么：会影响验收材料准备、问题归因和最终交付口径。

#### 8.4 如果联调通过但个别异常语义或字段细节仍待细化，甲方是否接受分阶段验收？

- 优先级：可以联调时确认
- 为什么要问：这关系到主线闭环先通过、细节后补的空间大小。
- 不确认会卡住什么：短期主要影响收口策略，不阻塞本轮事实确认。

## 第三部分：建议沟通顺序

建议和甲方沟通时，按下面顺序推进：

1. 先确认“本轮到底要打通哪些能力”和“集测完成标准是什么”。
原因：先定范围和成功标准，后面的问题才不会越问越散。

2. 再确认“任务如何进来、是否接收即启动、停止怎么控制、是否允许并发任务”。
原因：这是整条主线的入口事实，直接决定北向接入对象和控制语义。

3. 接着确认“状态查询和心跳分别怎么交互”。
原因：这组问题最好跟任务接入一起问，可以一次性把接入层和投影层的边界问清。

4. 然后集中确认“用例结果怎么上报、任务报告怎么交付、成功怎么判、失败怎么回”。
原因：结果上报、报告交付和回执语义强相关，拆开问容易来回反复。

5. 最后再收口“集测环境、联调入口、问题闭环方式、验收证据”。
原因：这组问题更偏组织和执行安排，适合在事实主线达成一致后一起定。

不适合一上来就问太细的问题主要有两类：

1. 具体字段枚举、字段命名、字段是否可选。
这些问题需要建立在任务主线和交付方式已经明确的前提下，否则容易把沟通拉进 schema 细节。

2. 心跳频率、重试次数、超时秒数、日志格式这类参数细节。
这些问题很重要，但更适合在主协同方式明确后进入联调细化阶段再收紧。

建议最好放在同一次沟通里一起问的问题有三组：

1. “任务下发 + 启动 + 停止 + 并发约束”
这是同一条主线的入口和控制事实，分开问容易互相打架。

2. “状态查询 + 心跳”
这两者都属于对外状态面，分开问容易导致重复定义状态口径。

3. “用例结果上报 + 报告交付 + 回执语义 + 验收证据”
这四项共同决定对外交付闭环，最好一次问清楚。

## 第四部分：哪些问题可以先不问太细

以下问题建议先确认方向和口径，不必在第一轮就追到字段级或参数级：

1. 心跳报文的完整字段结构。
第一轮先确认是主动心跳还是保活响应、最少要带哪些状态类别即可。

2. 状态查询和心跳中的精确字段命名。
第一轮先确认需要表达哪些事实，不必先追字段名。

3. 用例结果和任务报告的完整字段全集。
第一轮先确认关键字段类别、必填范围和验收关注项即可。

4. 回执对象和错误对象的具体字段。
第一轮先确认语义分类和同步 / 异步边界，不必先追 schema 形式。

5. 超时、重试、补报的精确参数。
第一轮先确认是否存在这类机制以及甲方是否接受，不必先追数值。

6. 用例结果上报与报告交付是否复用同一套底层交付机制。
这是实现组织问题，不是第一轮必须由甲方拍板的事实问题。

## 第五部分：当前结论 / 待确认事项 / 下一步建议

### 5.1 当前结论

1. 内部架构第一轮已经基本收口，当前不需要再回头争论内部骨架、任务系统归属和 SCOE 的架构位置。

2. 当前真正需要甲方确认的，不是内部怎么实现，而是北向协同事实、集测组织方式和验收口径。

3. 当前最优先要问清的，是会直接影响后续 spec 是否失真的那批问题：
- 任务如何下发
- 接收与启动是否分离
- 停止怎么控制
- 状态查询与心跳怎么交互
- 用例结果最少要回什么
- 报告怎么交付
- 成功 / 失败 / 拒绝 / 不可执行怎么定义
- 集测提供什么环境和入口
- 本轮集测的完成标准是什么

### 5.2 待确认事项

1. 北向协同方式是否已经固定为单一中心入口，还是存在中转层。
2. 任务接入、启动、停止是否为独立外部事务。
3. 状态查询与心跳分别由谁发起、最少表达哪些状态事实。
4. 用例结果上报的关键字段类别和成功判定口径。
5. 报告交付方式、交付回执语义和必须生成报告的场景。
6. 集测环境、样例数据、联调入口和问题闭环方式。
7. 本轮验收的最低标准、必查项和成功证据。

### 5.3 下一步建议

1. 先把本清单作为对甲方的一轮集中确认提纲，优先把“必须现在确认”的问题一次性问清。

2. 沟通完成后，把确认结论沉淀成一份“北向协同事实确认记录”，只记录外部事实，不掺入实现方案。

3. 在事实确认完成前，后续 spec 只宜继续推进内部骨架相关内容；凡是直接依赖外部事实的问题，尤其是状态查询、心跳、结果上报、报告交付、回执语义，不要凭内部假设提前冻结。
