其中，“任务下发”和“任务启动”当前我方仍按两个独立动作理解推进；如贵方本轮口径已有变化，也请直接指出。

## 当前本轮聚焦范围

为避免双方在首轮对接阶段同时展开过多内容，我方当前建议本轮仍先聚焦以下主线能力：

1. 任务进入
2. 任务启动

4. 最小状态查询
5. 心跳保活
6. 每个用例完成后的结果回传

相应地，以下内容当前不建议作为本轮同步展开前提：

2. 完整设备管理与设备详情协同
3. 独立参数查询 / 配置体系

6. 完整报告展示增强项

我方这样收口的目的，是先把最小主线做稳，再逐步展开扩展能力，避免后续实现与联调范围过早放大。

## 当前默认理解

在贵方未明确给出不同口径前，我方当前默认按以下理解推进：

1. 任务下发与任务启动分离。
2. `startTestCaseList` 当前先按任务级主线推进，不单独引入测试集级或局部启动级控制模型。
3. 本轮只保留一个“完全停下”的控制动作，不同时承接两套近义停止语义。
4. 每个用例完成后单独回传结果，且结果需要有最小稳定分类。
5. 文件传输完成和业务交付完成先分开理解，不先混为同一层完成口径。
6. 心跳最终只按一种交互模型实现，不接受同一轮同时按“有响应”和“无响应”两套方式推进。
7. 状态查询首轮先按最小整体状态面理解，不默认扩展到完整自检明细。

基于当前文档，我方这轮希望优先确认以下几个会直接影响主线实现和联调收口的问题。

## 1. `startTestCaseList.testCaseId`

我方当前先按“`taskId` 下的用例清单引用”理解推进，不单独引入测试集级或局部启动级控制模型。

这里希望确认的是：当前 `startTestCaseList` 中的 `testCaseId`，是否仅作为任务内待启动用例的引用清单使用；如果贵方这里实际表示“测试集”或其他更细粒度的控制对象，也请明确当前控制粒度。

如果这一点不先统一，后续启动、停止、状态和结果关联都会受到影响。

## 2. `controlTestTask.action`

我方当前本轮只准备承接一个“完全停下”的控制动作，不同时把 `stop` 和 `abort` 两套近义控制一起做进主逻辑。

这里希望确认的是：本轮最终希望保留哪一个动作作为“完全停止”的唯一控制口径；如果两者在贵方当前确有明确区别，也请直接说明最小差异。

我方这样处理的目的，是避免在首轮就把停止、终止、拒绝、异常等多类语义一起混入控制逻辑。

## 3. `testCaseResultReport.result`

我方当前理解是：每个用例完成后单独回传结果，这一主线没有问题。

这里希望进一步确认的是：当前 `result` 是否至少需要稳定区分“成功 / 失败 / 非正常完成”这一级最小结果分类，而不是继续把多种不同情况都压进 `tbd`。如果贵方已有更明确的最小结果口径，也请直接说明。

如果这一点不先收口，后续结果回传、报告汇总和验收口径都容易失真。

## 4. `testDataFileTranslationComplete.result`

我方当前先按“文件传输完成结果”理解这一字段，不直接把它等同于“报告业务已受理成功”。

这里希望确认的是：`testDataFileTranslationComplete.result` 当前是否只表达文件链路传输完成；如果贵方还希望它同时承担业务交付完成语义，也请明确当前最小判定口径。

我方这样区分的目的，是避免把“文件传完”和“业务已完成接收”混为一层，影响后续联调判断。

## 5. `heartbeatResponse`

当前文档中，心跳章节一处写“此消息无响应消息”，后面又给出了 `heartbeatResponse` 参数表和响应示例。

这里希望确认的是：本轮心跳最终按“无响应”实现，还是按“有响应”的请求响应方式实现。只要这一点统一，我方即可按唯一模型推进。

如果这一点不先统一，双方后续很容易按两套不同链路方式开发，联调时也难以按同一口径判定成功。
