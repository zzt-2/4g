# 第一部分：文档中已经明确的外部事实

## 1. 任务下发方式

文档明确写明，中心控制子系统与二级子系统之间的消息接口采用 HTTP/Restful API，文件传输采用 FTP。

文档明确存在“测试任务下发”接口，中心控制子系统通过 `setTestTask` 向二级子系统下发测试任务；一次 `setTestTask` 可包含一个或多个用例，文档称其为一个“子任务”。

文档明确任务下发和任务启动可以拆开处理。`setTestTask` 中有 `immediate` 语义：可立即执行，也可先下发、后等待启动命令。

文档明确每个测试用例完成后必须上报测试结果通知；每个子任务对应一个测试报告文件；如果有测试数据文件则需要上报，没有则无需上报。

## 2. 启动 / 停止 / 查询 / 心跳

文档明确存在“测试用例集启动”接口，作用是启动“之前下发但未执行”的测试用例集。

文档明确存在“测试任务执行控制”接口，控制动作至少包括：暂停、继续、停止、终止。

文档明确存在查询类接口，至少包括：设备列表查询、设备信息查询、二级子系统状态查询、用例查询。

文档明确“二级子系统状态查询”用于环境自检时获取系统运行状态，响应内容应包含自检详细过程信息。

文档明确存在心跳接口，描述为中心控制子系统和二级子系统之间“互发链路监视信息”；默认每 15 秒一次，周期可通过参数配置接口调整。

## 3. 用例结果上报

文档明确存在“测试结果通知”接口，二级子系统在单个测试用例结束后主动上报。

文档明确该通知是按“单个用例完成”粒度上报，而不是只在任务结束时统一上报。

文档明确用例结果状态至少包括：`success`、`fail`、`tbd`。

## 4. 报告生成与交付

文档明确测试过程中产生的测试数据文件传输完成后，二级子系统需要主动调用“测试数据文件上传完成”接口告知中心控制系统。

文档明确与测试任务无关的文件传输完成后，使用单独的“文件传输完成”接口上报。

文档明确测试任务报告文件后缀为 `.json`，并给出了任务级结果、用例级结果、检查点、统计项、附加项、步骤过程信息等内容类别示例。

文档明确文件类型中至少包含：测试报告文件 `TestReport`（json）、测试截图文件 `TestWordDat`（docx）、设备信息文件 `DeviceInfo`（json）、测试简报 `TestBrief`（后缀由二级决定）。

## 5. 错误 / 回执 / 拒绝语义

文档明确通用约定是“请求和响应一一对应”，并要求响应方法名为“请求方法名 + Response”。

文档明确 `requestId` 用于关联请求和响应，`sessionId` 用于标识同一事务。

文档明确大多数同步响应只定义了两类响应状态：`1: 正常`、`2: 异常`，并通过 `msg` 携带说明信息。

文档明确“测试任务执行控制”响应里额外存在 `handleCode`，用于表示更细粒度原因；文中至少举了“用例忙”“环境不具备”两个例子。

## 6. 集测环境 / 联调入口

文档明确所有接口示例均按 HTTPS URL 方式展示，请求方式均为 POST。

文档明确多处文件相关接口支持携带 `ftpInfo`，包括 FTP 的地址、端口、用户名、密码、目录。

文档明确文档面向“开发、联调、测试人员”，但联调环境本身只给出了示例地址形式。

## 7. 验收口径

文档明确测试报告中可承载“检查点、期望值、测试值、二级判决结果、说明信息”等内容，这些内容可作为结果说明材料。

文档未明确甲方最终验收以什么为准，也未明确“接口联通成功”“结果被甲方受理”“报告被甲方认可”三者之间的关系。

## 8. 文档交付形式

文档当前交付形式是文字版接口说明，包含按接口展开的参数表、请求示例、响应示例，以及部分文件内容格式示例。

文档未提到 Swagger、OpenAPI、在线接口文档、Postman 集合或其他可直接导入调试的接口资产。

文档有请求示例和响应示例，但系统化的错误示例、拒绝示例、超时示例未见明确提供。

# 第五部分：这份文档对后续 spec / 集测最关键的影响

1. 后续 spec 不能直接把文档里的外部接口对象映射成我方内部对象，尤其是任务、启动控制、状态、结果、报告这几组边界需要继续按“外部语义”单独确认。

2. 在甲方确认“任务接收/启动/停止/结果/报告”的真实事务边界之前，不适合冻结对外 schema；否则很容易把当前文档中的模糊写法提前固化。

3. 集测前最先要补齐的不是实现细节，而是联调资产和入口材料，尤其是 Swagger/OpenAPI 或等价在线接口描述、正式环境地址、认证方式、测试账号、FTP 交付链路。

4. 如果不先统一“成功、失败、拒绝、不可执行、超时”的外部语义，后续即使接口打通，也很难形成稳定的集测判定和验收口径。

5. 报告交付是当前最容易被低估的环节。文档说明了报告内容大致长什么样，但没有说清甲方如何接收、如何确认交付成功、如何把它纳入验收。

6. 文档本身已经暴露出若干未收口点，像心跳是否有响应、状态查询结构待协商、部分方法名不一致、部分接口暂保留，这意味着后续对外 spec 必须先做一轮“事实澄清”，不能直接照文档落实现或落正式协议。
