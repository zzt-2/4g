## 这一块在整份文档里的作用

这一组内容构成了“基础对象面”和“运行状态面”。

如果说任务主链是在回答“这次测试怎么跑”，那这一块回答的是：
- 系统里有哪些设备
- 设备当前是什么状态
- 子系统当前整体状态如何
- 当前有没有发生设备级或子系统级异常事件

## 本主题包含的具体接口

### 这一组接口共用的请求体外壳

除了个别特殊接口外，这一组接口的请求体基本都带这些公共字段：
- `method`：接口方法名
- `requestId`：请求标识
- `subSysType`：二级子系统类型
- `subSysId`：二级子系统 id
- `sessionId`：事务上下文 id

### 这一组接口共用的响应体外壳

这一组接口的响应体大多也共享同一层公共结构：
- `method`：通常为请求方法名加 `Response`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`
- `statusCode`
- `msg`

### 1. 设备列表查询

- 接口标题：`设备列表查询`
- method：`getDeviceList`
- URL：`https://127.0.0.1:5001/api/deviceInfo/getDeviceList`
- 请求方式：`POST`
- 原文用途：中心控制子系统向各二级子系统获取设备列表及设备信息时使用。
- 请求体业务字段：无额外业务字段，主要依赖公共请求外壳。
- 响应体业务字段：
  - `fileReport`：是否改为文件方式上报设备信息
  - `datas`：设备列表数组
  - 每个设备对象中重点包括：`name`、`deviceId`、`type`、`status`、`swVer`、`ip`、`pars`
- 最值得和甲方确认的字段：
  `fileReport`：设备列表什么时候走消息体、什么时候必须改走文件。
  `datas`：如果走消息体，甲方真正依赖哪些设备字段。
  `swVer`：甲方是否要求每次都返回实时真实版本，而不是缓存值。
  `type`：设备类型枚举是否已有统一表，不然后续设备分类会对不上。

### 2. 设备信息上报

- 接口标题：`设备信息上报`
- method：`deviceInfoReport`
- URL：`https://127.0.0.1:5001/api/deviceInfo/deviceInfoReport`
- 请求方式：`POST`
- 原文用途：二级子系统主动向中心控制系统上报设备信息时使用。
- 请求体业务字段：
  - `datas`：设备列表数组
  - 每个设备对象中重点包括：`name`、`deviceId`、`type`、`status`、`swVer`、`ip`、`pars`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `datas`：主动上报是全量覆盖还是只上报变化项。
  `deviceId`：设备唯一标识是否由甲方直接消费，格式是否必须严格统一。
  `status`：状态值是否会被甲方直接映射到页面或流程判断。

### 3. 设备信息查询

- 接口标题：`设备信息查询`
- method：`getDeviceInfo`
- URL：`https://127.0.0.1:5001/api/deviceInfo/getDeviceInfo`
- 请求方式：`POST`
- 原文用途：中心控制子系统查询二级子系统某一设备时使用。
- 请求体业务字段：
  - `deviceId`：目标设备 id
- 响应体业务字段：
  - `data`：单个设备对象
  - 设备对象重点包括：`name`、`deviceId`、`type`、`status`、`swVer`、`ip`、`pars`
- 最值得和甲方确认的字段：
  `deviceId`：是否只会按单设备精确查询，不会出现模糊或批量查询需求。
  `data.status`：设备状态枚举甲方是否已经固定。
  `data.pars`：参数列表是否要求全量返回，还是只返回甲方关心项。

### 4. 设备信息配置

- 接口标题：`设备信息配置`
- method：`setDeviceInfo`
- URL：`https://127.0.0.1:5001/api/deviceInfo/setDeviceInfo`
- 请求方式：`POST`
- 原文用途：中心控制子系统修改二级子系统设备信息时使用。
- 请求体业务字段：
  - `deviceId`：设备 id
  - `deviceData`：设备要修改的数据
  - `deviceData` 中原文重点包括：`name`、`type`、`status`、`swVer`、`ip`、`pars`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `deviceData`：哪些字段允许被甲方改，哪些字段只能查询不能配置。
  `pars`：参数配置是全量覆盖还是局部修改。
  `status`：状态字段是否允许被配置，还是仅由设备实时状态决定。

### 5. 二级子系统状态查询

- 接口标题：`二级子系统状态查询`
- method：`getSubSysState`
- URL：`https://127.0.0.1:5001/api/subSystem/getSubSysState`
- 请求方式：`POST`
- 原文用途：中心控制子系统查询二级子系统状态；环境自检时，二级子系统需返回系统运行状态和自检详细过程信息。
- 请求体业务字段：无额外业务字段，主要依赖公共请求外壳。
- 响应体业务字段：
  - `swVer`：软件版本
  - `status`：子系统整体状态
  - `data`：自检详细信息数组
  - 自检项重点包括：`stepName`、`deviceId`、`status`、`keyPars`、`msg`
- 最值得和甲方确认的字段：
  `status`：甲方到底按哪些状态值判断“具备联调/集测条件”。
  `data`：自检明细是否已有固定结构；原文这里实际上留有待协商空间。
  `keyPars`：哪些关键参数必须带，哪些只是可选展示项。
  `msg`：甲方是否会把这段说明直接作为环境不具备原因展示。

### 6. 设备告警上报

- 接口标题：`设备告警上报`
- method：`deviceAlarmReport`
- URL：`https://127.0.0.1:5001/api/deviceInfo/deviceAlarmReport`
- 请求方式：`POST`
- 原文用途：二级子系统上报设备级告警时使用。
- 请求体业务字段：
  - `datas`：告警数组
  - 每条告警重点包括：`alarmId`、`deviceId`、`severity`、`msg`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `alarmId`：告警编号是甲方统一编码还是我方自定义即可。
  `severity`：严重级别枚举是否和甲方现有系统完全一致。
  `msg`：告警说明是否有长度、语言或模板要求。

### 7. 二级子系统告警上报

- 接口标题：`二级子系统告警上报`
- method：`subSysAlarmReport`
- URL：`https://127.0.0.1:5001/subSystem/subSysAlarmReport`
- 请求方式：`POST`
- 原文用途：二级子系统上报子系统级告警时使用。
- 请求体业务字段：
  - `datas`：子系统告警数组
  - 每条告警重点包括：`alarmId`、`severity`、`msg`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `alarmId`：是否需要和设备告警共用同一套编码规则。
  `severity`：子系统级告警和设备级告警是否共用同一等级体系。
  `msg`：甲方是否要求能区分“恢复告警”和“新发告警”。

## 重要内容

### 1. 设备类接口是一组完整的增量认知链

原文不是只给了一个设备查询接口，而是把设备相关能力写成了一组：
- 设备列表查询：看全量设备及设备信息
- 设备信息上报：由二级主动上报设备信息
- 设备信息查询：看单个设备
- 设备信息配置：改设备信息

阅读时建议把这四节放在一起，不要拆开看。它们共同定义了设备对象在北向接口里的存在方式。

### 2. 设备列表支持“直接返回”和“文件方式上报”两种交付形态

在设备列表查询里，原文专门写了 `fileReport` 语义：
- 如果设备信息不大，可以直接放在响应 `datas` 中
- 如果设备信息超出消息限制，可以改为文件方式上报

这说明文档并没有把“查设备列表”理解成单纯接口返回，而是允许根据数据体量切换为文件交付。

### 3. 设备软件版本号必须是实时真实值

原文对设备列表查询有一个非常明确的约束：
设备的软件版本号 `swVer` 必须是二级子系统实时查询设备的真实版本号，不可固定写死。

这属于基础对象字段里的强约束，阅读时很值得单独记下来。

### 4. 二级子系统状态查询不只是“在线/离线”，还带环境自检语义

原文在“二级子系统状态查询”里强调：
- 环境自检时，二级子系统根据中心控制子系统的查询，响应自身系统运行状态
- 上报内容应包含自检详细过程信息

所以这节不能只按普通状态查询来读。它更像“查询 + 自检明细回传”的组合接口。

### 5. 状态查询结果里有两层状态

这节原文实际上有两层状态：
- 子系统整体状态，例如 `online / alarm / error`
- 自检步骤或设备级状态，例如 `online / offline / alarm / error / busying / available`

阅读时要把“整体状态”和“步骤状态”分开理解，不然容易把子系统状态和设备状态混掉。

### 6. 告警上报是事件型接口，不是状态查询的替代品

原文后半部分单独写了两类告警上报：
- 设备告警上报
- 二级子系统告警上报

它们和状态查询不是同一类动作：
- 状态查询更像“被查询时返回当前面貌”
- 告警上报更像“有事件时主动推送”

### 7. 设备级告警和子系统级告警的对象粒度不同

设备告警围绕 `deviceId` 展开，强调具体设备、告警编号、严重级别和告警说明。

子系统告警则是围绕子系统整体运行问题展开，粒度更高，不应和设备告警混在一起看。

## 这一块建议重点精读的内容

### 1. 设备信息结构

设备列表、设备信息上报、设备信息查询里其实复用了高度相似的设备信息结构。

第一次精读时，建议把这些字段当成“设备对象的北向投影”集中看一遍，包括：
- 设备 id、名称、类型
- 软件版本
- 状态
- IP 及设备参数列表

### 2. 二级子系统状态查询中的 `data`

这一节对后续联调和集测通常很关键，因为它携带了自检步骤、设备、自检关键参数和说明信息。

### 3. 告警级别

设备告警里给了明确的严重级别枚举，例如 `critical / major / warn / clear`。这类枚举通常会成为对外展示或处理逻辑的公共依据。

## 读这一块时最容易混淆的地方

### 1. “设备信息上报” 和 “设备列表查询”

两者都能让中心侧看到设备信息，但一个偏主动上报，一个偏被动查询。

### 2. “子系统状态” 和 “子系统告警”

状态是一个相对稳定的当前面，告警是事件型变化，不要等同。

### 3. “设备状态” 和 “自检步骤状态”

原文在状态查询里把这些内容写在同一节里，第一次阅读时很容易混成一个层级。
