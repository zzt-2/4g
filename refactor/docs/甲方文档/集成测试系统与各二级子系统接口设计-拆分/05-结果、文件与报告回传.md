# 05-结果、文件与报告回传

## 覆盖范围

对应原文：4368-5096

- `### 测试数据文件上传完成`
- `### 文件传输完成`
- `### 测试结果通知`
- 以及“测试数据文件上传完成”章节内部给出的测试任务报告文件内容格式

## 这一块在整份文档里的作用

这一块处理的是任务跑起来之后，二级子系统怎样把结果、文件和报告回传给中心控制系统。

它不是任务入口，而是任务交付出口。

## 本主题包含的具体接口

### 这一组接口共用的请求体外壳

这一组接口通常都带：
- `method`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`

任务相关回传中还会经常带：
- `taskId`
- `testCaseId`

### 这一组接口共用的响应体外壳

响应体通常包含：
- `method`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`
- `statusCode`
- `msg`

### 1. 测试数据文件上传完成

- 接口标题：`测试数据文件上传完成`
- method：`testDataFileTranslationComplete`
- URL：`https://127.0.0.1:5001/api/report/testDataFileTranslationComplete`
- 请求方式：`POST`
- 原文用途：二级子系统测试过程中产生的测试数据文件传输完成后，主动向中心控制系统上报。
- 请求体业务字段：
  - `taskId`
  - `result`
  - `msg`
  - `testCaseId`：数组
  - `imsi`
  - `ueFlag`
  - `fileType`
  - `filePath`
  - `fileFormat`
  - `fileTitle`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `result`：这里表示“文件传输成功”，还是“业务文件已被甲方接受”。
  `testCaseId`：一个文件是否允许关联多个用例。
  `fileType`：哪些文件类型是本轮集测必须上报的。
  `filePath`：FTP 路径规则是否已有正式约定。
  `fileFormat` / `fileTitle`：图片、音频是否必须支持页面报告展示。

### 2. 文件传输完成

- 接口标题：`文件传输完成`
- method：`fileTranslationComplete`
- URL：`https://127.0.0.1:5001/api/report/fileTranslationComplete`
- 请求方式：`POST`
- 原文用途：与测试任务无关的文件上传完成后上报，例如测试脚本文件传输。
- 请求体业务字段：
  - `tranType`
  - `result`
  - `fileType`
  - `fileIndex`
  - `filePath`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `tranType`：是否真的既支持上传也支持下载回执。
  `fileIndex`：这个索引值在本轮有没有对应上游命令来源。
  `fileType`：哪些非任务类文件需要走这条回执链路。

### 3. 测试结果通知

- 接口标题：`测试结果通知`
- method：`testCaseResultReport`
- URL：`https://127.0.0.1:5001/api/report/testCaseResultReport`
- 请求方式：`POST`
- 原文用途：二级子系统在单个测试用例结束后上报测试结果通知。
- 请求体业务字段：
  - `taskId`
  - `testCaseId`
  - `loopIndex`
  - `result`
  - `msg`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `result`：`success/fail/tbd` 三个状态甲方分别怎么解释。
  `loopIndex`：重复执行场景下甲方是否按循环次数区分结果。
  `msg`：这段结果说明是展示用，还是会参与验收判定。

### 4. 测试任务报告文件

- 这不是单独消息接口，但原文在本主题中给出了完整文件结构示例。
- 文件类型：`TestReport`
- 文件后缀：`.json`
- 文件体重点分三层：
  - 任务级：`subSysType`、`subSysId`、`sessionId`、`taskId`、`startTime`、`endTime`、`result`、`msg`
  - 用例级：`testCaseList` 下的 `testCaseId`、`resources`、`deviceIds`、`startTime`、`endTime`、`result`、`msg`、`judgmentMsg`
  - 明细级：`checkPoints`、`statisticsItems`、`attachItems`、`processAndDatas`、`testParsInfo`
- 最值得和甲方确认的字段：
  `result`：任务级和用例级结果是否都按同一口径解释。
  `judgmentMsg`：甲方是否期待我方给出初步定位分析。
  `checkPoints`：检查点是否就是甲方验收看的核心字段。
  `processAndDatas`：过程明细是必需交付，还是可选增强材料。

## 重要内容

### 1. 原文把“结果通知”和“文件传输完成”分成了不同层次

这里至少有三类不同的回传动作：
- 单个用例结束后的结果通知
- 测试数据文件上传完成通知
- 与任务无关的通用文件传输完成通知

这三者虽然都像“回传”，但关注的对象不同：
- 一个关注执行结果
- 一个关注测试过程中产生的文件
- 一个关注更通用的文件传输动作

### 2. 测试数据文件上传完成是任务相关文件回传的关键接口

原文在这里写明：
二级子系统测试过程中产生的测试数据文件，如信令、电文、性能、测试报告等，传输完成后都要主动向中心控制系统上报。

也就是说，文件真正到位，不是只靠 FTP 路径存在，还要通过这条消息明确告知中心侧。

### 3. 单个用例结果通知是独立接口，不等于报告文件

“测试结果通知”接口 `testCaseResultReport` 用于单个测试用例结束后主动上报结果。

原文给出的结果字段重点包括：
- `taskId`
- `testCaseId`
- `loopIndex`
- `result`
- `msg`

而且 `result` 的枚举里已经出现：
- `success`
- `fail`
- `tbd`

这说明用例结果通知本身就是一个独立对象，不需要等任务报告生成后才有结果表达。

### 4. 报告文件结构写得很重，是这一块最值得精读的内容

原文在“测试数据文件上传完成”之后，给了测试任务报告文件的 JSON 内容格式示例。

这一段信息量很大，可以分成三层：

第一层，任务级信息：
- 子系统类型、子系统 id
- `sessionId`
- `taskId`
- 任务开始时间、结束时间
- 任务结果、结果信息

第二层，用例级结果信息：
- `testCaseId`
- 资源信息
- 设备 id
- 用例开始结束时间
- 用例结果、说明信息
- 初步分析定位信息 `judgmentMsg`

第三层，用例内部明细：
- 检查点 `checkPoints`
- 统计项 `statisticsItems`
- 附加项 `attachItems`
- 测试流程及配置 `processAndDatas`
- 测试配置信息 `testParsInfo`

### 5. 图片、音频等文件在回传时有专门说明字段

原文在“测试数据文件上传完成”里为文件补充了：
- `fileFormat`
- `fileTitle`

并说明只有需要在页面版测试报告中显示下载链接的文件才需填这些值，目前主要是图片和音频文件。

这说明文件回传不只是“传上去就完了”，还考虑了后续页面报告如何展示。

### 6. 通用文件传输完成接口不只用于任务数据

`fileTranslationComplete` 在原文中被定义为与测试任务无关的文件上传完成上报，例如测试脚本文件传输。

阅读时必须和“测试数据文件上传完成”区分开，不然很容易把所有文件回传都混到一个接口里理解。

## 这一块建议重点精读的内容

### 1. 测试任务报告文件结构示例

如果要理解原文对“测试报告”的想象，这一段是全篇最关键的材料之一。

### 2. `testCaseResultReport`

这是看“单个用例完成时怎么表达结果”的最直接入口。

### 3. `testDataFileTranslationComplete`

这是理解“结果文件”和“结果消息”如何配合的关键接口。

## 读这一块时最容易混淆的地方

### 1. 用例结果通知 和 任务报告文件

前者是消息通知，后者是文件化汇总结果。原文两者都写得很重，容易误以为只保留一个就够了。

### 2. 测试数据文件上传完成 和 通用文件传输完成

二者名字很像，但适用场景不同。一个偏任务相关数据文件，一个偏通用文件传输。

### 3. 文件上传完成 和 文件已被中心侧业务受理

原文这里主要表达“文件传输完成并已通知”，阅读时不要自动把它理解成“中心侧业务已经完成处理”。
