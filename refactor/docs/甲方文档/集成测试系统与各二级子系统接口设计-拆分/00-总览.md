# 集成测试系统与各二级子系统接口设计-拆分总览

## 这套拆分文件是做什么的

这套文件的目标，不是替原文下结论，也不是做甲方沟通稿。

它做的是两件事：
- 把原始大文档重新按主题归类，避免阅读时在前后几十页之间来回跳
- 在每个主题里把重要内容先拎出来，帮助后续继续精读原文

使用原则：
- 仍以原文为准
- 这里只做主题拆分和重点梳理，不补 schema，不替原文做决定
- 每个主题文件都保留原文章节和行号范围，方便回查

## 建议阅读顺序

如果是第一次读这份接口文档，建议按下面顺序看：

1. 先看“协议与公共规则”
先把协议、命名、消息共性、文件规则看清，否则后面每个接口都会反复卡在公共字段上。

2. 再看“任务主链”
先建立“任务怎么下发、怎么启动、怎么控制”的主线认识。

3. 接着看“结果、文件与报告回传”
把任务跑完之后怎么回传结果和文件接上。

4. 再看“设备、状态与告警”
这是基础对象面和运行状态面，适合在主线之后补全。

5. 再看“用例资产与脚本维护”
这是测试资产管理，不属于任务主链，但和用例定义强相关。

6. 最后看“参数协同与过程消息”“运维控制与链路保活”
这两块更偏扩展协同和系统运行能力。

## 原文中最容易混在一起的几组内容

### 1. 用例资产维护 和 任务执行

原文里既有“用例查询 / 新增 / 删除 / 脚本上传下载”，也有“任务下发 / 启动 / 控制”。

前者是在维护测试资产；后者是在执行一次测试任务。读的时候如果不先拆开，很容易把“定义一个用例”和“下发一个任务”当成一回事。

### 2. 测试结果通知 和 文件上传完成 和 测试报告

这三类内容都在“回传”语义下，但不是同一层东西：
- 测试结果通知：偏单个用例结果
- 文件上传完成：偏文件交付完成
- 测试报告：偏任务级或子任务级汇总结果

### 3. 参数查询配置 和 参数转发

原文里有直接对二级子系统的参数查询/配置，也有通过中心控制子系统转发到其他系统的查询/配置。这两类接口名字很像，但语义并不一样。

### 4. 设备状态 / 子系统状态 / 告警 / 心跳

它们都在表达“系统当前怎么样”，但粒度不同：
- 状态查询：查询式
- 告警上报：事件式
- 心跳：链路监视式

## 当前拆分方式的取舍

这次没有按“原文每个三级标题一个文件”去切，因为那样会得到很多很碎的小文件，阅读体验反而更差。

现在的拆法是“少量主题文件”，一个主题文件里保留一组强相关接口和说明，既能读顺，又不会把原文切得太碎。
