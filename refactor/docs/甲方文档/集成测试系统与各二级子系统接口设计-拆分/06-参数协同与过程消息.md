# 06-参数协同与过程消息

## 覆盖范围

对应原文：5398-6171，6592-7082

- `### 测试参数查询转发`
- `### 测试参数配置转发`
- `### 参数查询接口`
- `### 参数配置接口`
- `### 测试参数配置反馈接口`
- `### 数据转发接口`
- `### 信息上报`

## 这一块在整份文档里的作用

这一块比较容易被忽略，因为它不像任务下发或结果回传那样直观。

但从文档结构看，它其实在描述三条很重要的协同支线：
- 参数的直接查询 / 配置
- 参数在其他系统之间的转发协同
- 测试过程中的实时过程消息和异步结果消息

## 本主题包含的具体接口

### 这一组接口共用的请求体外壳

这组接口的请求体常见公共字段包括：
- `method`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`

与测试过程强相关的接口还可能带：
- `taskId`
- `testCaseId`
- `relationId`

### 这一组接口共用的响应体外壳

响应体通常包含：
- `method`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`
- `statusCode`
- `msg`

部分接口还会返回参数值列表、状态码或结果数据。

### 1. 测试参数查询转发

- 接口标题：`测试参数查询转发`
- method：`testCaseGetParsForward`
- URL：`https://127.0.0.1:5001/testCase/testCaseGetParsForward`
- 请求方式：`POST`
- 原文用途：测试用例执行过程中，经中心控制子系统向其他子系统发送参数查询信息。
- 请求体业务字段：
  - `relationId`
  - `taskId`
  - `subSysType`：目标子系统类型
  - `pars`：查询参数列表
- 响应体业务字段：
  - `relationId`
  - `taskId`
  - `pars`：查询返回值列表
- 最值得和甲方确认的字段：
  `subSysType`：目标系统类型枚举是否已有正式清单。
  `pars`：参数列表结构是否已固定，还是按目标系统协商。
  `relationId`：转发链路上如何做请求和返回的关联。

### 2. 测试参数配置转发

- 接口标题：`测试参数配置转发`
- method：`testCaseSetParsForward`
- URL：`https://127.0.0.1:5001/testCase/testCaseSetParsForward`
- 请求方式：`POST`
- 原文用途：测试用例执行过程中，经中心控制子系统向其他子系统发送参数配置信息。
- 请求体业务字段：
  - `relationId`
  - `taskId`
  - `subSysType`：目标子系统类型
  - `pars`：配置参数列表
- 响应体业务字段：
  - `relationId`
  - 其余主要为通用回执字段
- 最值得和甲方确认的字段：
  `pars`：是否默认即时生效，还是会有异步生效场景。
  `relationId`：后续配置反馈是否用它关联。
  `subSysType`：转发目标是否只限文档中列出的几个系统。

### 3. 参数查询接口

- 接口标题：`参数查询接口`
- method：`getPars`
- URL：`https://127.0.0.1:5001/testCase/getPars`
- 请求方式：`POST`
- 原文用途：中心控制子系统查询各二级子系统的参数时使用。
- 请求体业务字段：
  - `taskId`：可选
  - `imsi`：可选
  - `parIds`：查询参数 id 数组
- 响应体业务字段：
  - `taskId`：可选
  - `pars`：参数返回列表
- 最值得和甲方确认的字段：
  `parIds`：参数 id 字典是否已有正式资产。
  `imsi`：在什么场景下必须带终端 IMSI。
  `pars`：返回值是否要求带单位、名称、时间戳等补充信息。

### 4. 参数配置接口

- 接口标题：`参数配置接口`
- method：`setPars`
- URL：`https://127.0.0.1:5001/testCase/setPars`
- 请求方式：`POST`
- 原文用途：中心控制子系统配置各二级子系统参数时使用。
- 请求体业务字段：
  - `taskId`：可选
  - `pars`：参数配置列表
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `pars`：配置是批量原子生效还是逐项生效。
  `taskId`：参数配置是否一定属于某个任务上下文。

### 5. 测试参数配置反馈接口

- 接口标题：`测试参数配置反馈接口`
- method：`parsSetFeedback`
- URL：`https://127.0.0.1:5001/subSystem/parsSetFeedback`
- 请求方式：`POST`
- 原文用途：参数配置为非即时生效时，参数生效后由一级子系统通知二级子系统。
- 请求体业务字段：
  - `relationId`
  - `status`
  - `msg`
  - `parList`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `relationId`：是否明确对应前面的哪一次配置请求。
  `status`：失败时是整体失败，还是允许部分成功。
  `parList`：反馈中要不要回完整参数值，还是只回受影响项。

### 6. 数据转发接口

- 接口标题：`数据转发接口`
- method：`dataTransmit`
- URL：`https://127.0.0.1:5001/subSystem/dataTransmit`
- 请求方式：`POST`
- 原文用途：中心控制子系统给二级子系统转发异步命令结果或数据文件。
- 请求体业务字段：
  - `relationId`
  - `taskId`
  - `data`
  - `data` 中重点包括：`statusCode`、`statusMsg`、`dataType`、`data`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `dataType`：是否已有固定类型枚举。
  `data.statusCode`：这个状态码体系是否和外层 `statusCode` 同义。
  `data.data`：承载结果时是否有统一结构，还是完全按场景变化。

### 7. 信息上报

- 接口标题：`信息上报`
- method：`msgReport`
- URL：`https://127.0.0.1:5001/api/report/msgReport`
- 请求方式：`POST`
- 原文用途：二级子系统向中心控制子系统实时上报用例流程步骤信息，例如入网、呼叫、去注册等。
- 请求体业务字段：
  - `taskId`
  - `testCaseId`
  - `title`
  - `msg`
  - `result`：可选
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `title`：过程步骤名称是否有推荐枚举或展示规范。
  `msg`：是面向人读的描述，还是面向系统处理的结构化信息。
  `result`：哪些步骤需要带结果，哪些步骤只做过程提示。

## 重要内容

### 1. 参数类接口分“直接操作”和“转发操作”两类

原文里至少有两组参数能力：
- `getPars / setPars`：中心控制子系统直接和二级子系统做参数查询配置
- `testCaseGetParsForward / testCaseSetParsForward`：测试用例执行过程中，经由中心控制子系统转发到其他二级子系统或其他系统

阅读时一定要先区分这两组，不要只看名字相似就混为一类。

### 2. 转发类接口强调“目标格式由协商确定”

原文在测试参数查询转发和配置转发里都专门写到：
不同目标子系统的参数格式定义可能不同，由中心控制子系统和各子系统及运控系统协商后确定。

这说明这两节更像协同框架，不是完全封死的统一参数模型。

### 3. 参数配置转发目前默认是即时生效场景

原文在“测试参数配置转发”里特别说明：
当前这套方法是针对参数配置即时生效的场景。如果后续有非即时生效的参数配置，需要再定义新接口。

这是一条很重要的边界说明，读这一块时不应略过。

### 4. 参数配置反馈接口是参数类动作的后续闭环

“测试参数配置反馈接口”不是新的配置动作，而是配置完成后的结果反馈。

原文给出的核心语义是：
- `status` 为 1 成功、2 失败
- 有一项失败即置失败
- `msg` 可携带失败原因

所以这节更适合和 `setPars`、参数转发放在一条线上理解。

### 5. 数据转发接口用于承载异步结果或数据文件

原文说明 `dataTransmit` 一般用于中心控制子系统给二级子系统转发异步命令结果或数据文件，例如核心网业务服务器 ping 命令统计结果。

它更像一条通用异步回传通道，而不是固定某一个业务对象的专用接口。

### 6. 信息上报接口承载测试过程中的实时步骤信息

原文在“信息上报”中给出的例子包括：
- 入网
- 呼叫
- 去注册

这说明这节不是最终结果，也不是配置反馈，而是测试过程中的实时过程消息。

并且其中的 `result` 是可选的，表明这类消息可以只表达过程节点，不必每条都形成完整判定结果。

## 这一块建议重点精读的内容

### 1. 四类参数动作的边界

建议把下面四节放在一起看：
- 参数查询接口
- 参数配置接口
- 测试参数查询转发
- 测试参数配置转发

这能帮助快速看清“直接查配”和“跨系统转发”的区别。

### 2. 参数配置反馈接口

这是理解参数协同闭环的关键章节。

### 3. 信息上报接口

如果后续要梳理测试执行过程中有哪些实时消息，这节是最直接的入口。

## 读这一块时最容易混淆的地方

### 1. 参数查询配置 和 测试过程消息

两者都发生在执行过程中，但一个偏配置协同，一个偏过程记录。

### 2. 数据转发接口 和 文件传输完成接口

两者都可能涉及数据，但一个偏异步数据/结果转发，一个偏文件传输完成通知。

### 3. 参数配置反馈 和 一般响应消息

一般响应消息是请求当下的同步回执；参数配置反馈更像配置动作落地后的业务结果反馈。
