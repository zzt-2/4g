## 这一块在整份文档里的作用

这是全篇的公共前提。

后面所有接口虽然分散在不同章节，但协议、消息外壳、命名、文件交付方式，基本都受这一块约束。第一次阅读时，这部分值得先单独吃透。

## 重要内容

### 1. 通信方式是“消息接口 + 文件传输”两套并行机制

原文总述里明确写了两件事：
- 消息传输采用 HTTP 协议通信，接口采用 Restful API 格式
- 文件传输采用 FTP 协议方式

阅读时要先建立这个基本框架：
- 消息接口负责通知、查询、控制、结果回执
- 文件传输负责脚本、报告、信令、日志、测试数据等文件类交付

### 2. 文件规则在这份文档里不是附属内容，而是主干规则

原文用了较大篇幅定义文件类型标识、目录、文件名和内容格式，说明文件交付不是边缘能力，而是北向协同的一部分。

这里要重点看四个层次：
- 文件类型标识：例如 `CfgParam`、`ScriptDat`、`TestReport`、`DeviceInfo`
- 文件目录：信源、信宿、目录层级的约定
- 文件命名：由设备标识、文件类型标识、时间要素、后缀构成
- 文件内容格式：对信令、日志、性能、测试数据、测试报告给出格式说明

### 3. 测试报告文件在公共规则里已经有一层抽象定义

原文在前面就先给了测试报告文件的内容抽象：
- 子系统类型、子系统 id
- 任务 id、开始时间、结束时间
- 任务执行结果状态、任务结果信息
- 每个用例的测试结果信息

这说明“报告文件”并不是到任务章节才突然出现，而是在公共文件规则层面就已经被定义成一个重要交付物。

### 4. 消息通用约定是后续所有接口的共同外壳

原文对消息共性要求写得很集中，几个最重要的点是：
- 消息分请求和响应两类，并要求一一对应
- 响应方法名为“请求方法名 + Response”
- `subSysId` 是二级子系统唯一标识，且必须在所有接口消息中携带
- `deviceId` 在子系统内唯一
- `method` 在全子系统范围内必须唯一
- `sessionId` 标识同一事务，由发起方产生

这一段决定了后面大多数接口看起来为什么都有类似的固定字段。

### 5. 文件上报范围由二级子系统根据用例决定

原文在消息约定里明确提到：
在一个测试任务中，具体向中心控制子系统上报哪些文件，由二级子系统根据用例决定，不是由中心控制子系统决定。

这一句虽然放在公共规则里，但意义很强。它会直接影响后面怎么理解“结果回传”和“文件上传完成”相关接口。

### 6. 用例脚本不是任意格式，文档给了可编辑性约束

原文要求各二级上传给中心控制子系统的用例脚本文件必须可以用 VSCode 编辑修改，并建议使用 Python 或 XML。

这说明脚本文件在文档里的角色，不只是“二进制附件”，而是预期可编辑、可维护的测试资产。

## 这一块最值得做的回查点

1. 文件类型标识表
这是后面很多 `fileType` 字段的来源。

2. 文件目录与命名规则
这是后面涉及 FTP 路径、上报文件、脚本下载时的背景规则。

3. 消息约定中的 `method / requestId / subSysId / sessionId`
这是看任何接口时都要不断回用的公共字段。

## 阅读时容易忽略的细节

### 1. 原文总述写的是 HTTP，但接口示例很多地方又写 HTTPS

这一点在公共规则里就埋下了阅读注意点。即使此处不做判断，也建议在精读接口时始终留意协议写法是否一致。

### 2. 文件类规则和消息类规则是并列主线

很多人第一次读时会把文件规则当成附录，但这份文档里文件规则实际上和消息接口同等重要。

### 3. 公共规则里已经出现了一些后文才会展开的概念

例如：
- 测试报告
- 设备信息文件
- 用例脚本文件
- 子系统唯一标识

如果先不读这块，后面看到这些对象时会缺公共背景。
