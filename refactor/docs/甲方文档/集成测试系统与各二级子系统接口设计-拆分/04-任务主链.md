# 04-任务主链

## 覆盖范围

对应原文：3597-4367

- `### 测试任务下发`
- `### 测试用例集启动`
- `### 中心控制系统文件获取请求`
- `### 测试任务执行控制`

## 这一块在整份文档里的作用

这是整份文档里最接近“测试任务生命周期主线”的部分。

它回答的是：
- 任务怎么进入二级子系统
- 任务是立刻执行还是等待启动
- 已下发任务怎么被启动
- 执行中怎么被暂停、继续、停止、终止
- 某些文件是否会被中心侧反向请求获取

## 本主题包含的具体接口

### 这一组接口共用的请求体外壳

这一组接口通常都带：
- `method`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`
- 任务相关接口还会高频使用 `taskId`

### 这一组接口共用的响应体外壳

响应体通常包含：
- `method`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`
- `statusCode`
- `msg`

任务控制类接口还会增加 `handleCode` 等业务反馈字段。

### 1. 测试任务下发

- 接口标题：`测试任务下发`
- method：`setTestTask`
- URL：`https://127.0.0.1:5001/api/task/setTestTask`
- 请求方式：`POST`
- 原文用途：中心控制子系统向二级子系统下发测试任务。
- 请求体业务字段：
  - `resources`：依赖资源
  - `taskId`、`taskName`
  - `immediate`
  - `repeatCount`
  - `isEnd`
  - `orbitProtectTime`
  - `testCaseInfo`
  - `ftpInfo`
  - 其中 `testCaseInfo` 里重点包括：`testCaseId`、`deviceIds`、`masterTest`、`testMode`、`ephMode`、`orbitInfo`、`inputPars`
- 响应体业务字段：
  - `taskId`
  - `data.imsi`：如果测试设备有 IMSI，可返回给中心控制子系统用于抓信令
- 最值得和甲方确认的字段：
  `immediate`：任务接收和任务启动是否真的拆成两个外部事务。
  `isEnd`：是否表示同一任务允许分批下发，最后一批显式收口。
  `resources`：资源信息哪些是甲方一定会给、哪些只是可选增强信息。
  `testCaseInfo.inputPars`：参数值是甲方页面填写后下发，还是由我方补齐。
  `ftpInfo`：任务下发时给 FTP 信息到底是备用还是正式链路。

### 2. 测试用例集启动

- 接口标题：`测试用例集启动`
- method：`startTestCaseList`
- URL：`https://127.0.0.1:5001/api/task/startTestCaseList`
- 请求方式：`POST`
- 原文用途：中心控制子系统启动之前下发但未执行的测试用例集。
- 请求体业务字段：
  - `taskId`
  - `testCaseId`：数组，原文写为“测试集或测试 id”
- 响应体业务字段：
  - `taskId`
  - 其余为通用回执字段
- 最值得和甲方确认的字段：
  `testCaseId`：这里到底是测试集 id、用例 id，还是两者都可能。
  `taskId`：启动动作是否只能作用于已下发的同一任务上下文。

### 3. 中心控制系统文件获取请求

- 接口标题：`中心控制系统文件获取请求`
- method：`ccSysGetFileRequest`
- URL：`https://127.0.0.1:5001/api/subSystem/ccSysGetFileRequest`
- 请求方式：`POST`
- 原文用途：中心控制子系统向二级子系统获取文件；文档同时标注“是否有需求场景待定，暂保留”。
- 请求体业务字段：
  - `taskId`
  - `testCaseId`：可选
  - `fileType`
  - `ftpInfo`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `fileType`：本轮集测是否真的会用到这条取文件链路。
  `testCaseId`：任务级文件和用例级文件如何区分。
  `ftpInfo`：目录和账号由哪一方提供。

### 4. 测试任务执行控制

- 接口标题：`测试任务执行控制`
- method：`controlTestTask`
- URL：`https://127.0.0.1:5001/api/task/controlTestTask`
- 请求方式：`POST`
- 原文用途：对测试任务进行暂停、继续、停止、终止等控制操作。
- 请求体业务字段：
  - `taskId`
  - `action`：原文枚举包括 `pause`、`continue`、`stop`、`abort`
- 响应体业务字段：
  - `taskId`
  - `handleCode`：原文举例有“用例忙”“环境不具备”等
- 最值得和甲方确认的字段：
  `action`：`stop` 和 `abort` 在甲方口径里分别代表什么。
  `handleCode`：完整错误码表是否已有正式定义。
  `taskId`：是否存在并发任务场景，以及控制命令如何精确命中。

## 重要内容

### 1. 测试任务下发是主线入口

原文明确写了“测试任务下发”接口，中心控制子系统通过 `setTestTask` 向二级子系统下发测试任务。

这一节信息量很大，至少包含：
- 任务 id、任务名
- 是否立即执行 `immediate`
- 重复执行次数 `repeatCount`
- 是否任务结束 `isEnd`
- 资源信息 `resources`
- 用例信息 `testCaseInfo`
- 可选的 FTP 信息

### 2. 原文把“任务接收”和“任务启动”拆成了两个动作

在 `setTestTask` 里，`immediate` 可以为 `false`，表示先下发但不立刻执行。

随后又单独定义了“测试用例集启动”接口 `startTestCaseList`，用于启动“之前下发但未执行”的测试用例集。

阅读时要重点感受这条边界：
任务进入系统，不一定等于立刻开跑。

### 3. 任务下发里已经包含了执行上下文

`setTestTask` 不只是告诉二级“有个任务”，它还携带了很多执行上下文，例如：
- 卫星和载荷资源
- 用例所使用设备 id
- 主测/被测语义
- 测试模式、星历模式
- 弧段信息
- 用例输入参数

这说明任务下发在原文里是很“重”的一个动作。

### 4. 控制接口覆盖了暂停、继续、停止、终止几种动作

“测试任务执行控制”给出 `controlTestTask`，动作包括：
- `pause`
- `continue`
- `stop`
- `abort`

并在响应里提供 `handleCode`，用于表达更细的处理原因，例如“用例忙”“环境不具备”。

### 5. “中心控制系统文件获取请求”被放在主链中间，但自身状态并不稳定

原文在这一节标题下直接标注了：
“是否有需求场景待定，暂保留”。

阅读时建议把它标成“主线旁支接口”，不要默认它和下发、启动、控制一样确定。

### 6. 任务主链和结果主链是分开的

这块只讲任务怎么被下发、启动、控制，不讲任务跑完之后怎么报结果、怎么交报告。

后者要去看“结果、文件与报告回传”那一组内容。

## 这一块建议重点精读的内容

### 1. `setTestTask` 的任务结构

这是后续所有执行相关动作的起点，尤其值得注意：
- `immediate`
- `isEnd`
- `testCaseInfo`
- `resources`

### 2. `startTestCaseList` 的启动粒度

原文在这里写“测试集或测试 id”，第一次阅读时很值得单独留心。

### 3. `controlTestTask` 的动作语义和 `handleCode`

如果后续要分析执行控制边界，这节是最直接的入口。

## 读这一块时最容易混淆的地方

### 1. 任务下发 和 测试用例启动

原文显式把它们拆开了，阅读时不要把二者合并成一个动作。

### 2. 任务级控制 和 用例级执行结果

控制是“对正在执行的任务做动作”，不是“报告某个用例执行完了”。

### 3. 文件获取请求 和 结果文件上报

“中心控制系统文件获取请求”是中心侧反向取文件；
“测试数据文件上传完成”是二级侧主动上报文件已传完。两者方向和场景不同。
