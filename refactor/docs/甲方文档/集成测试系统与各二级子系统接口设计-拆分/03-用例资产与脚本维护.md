## 这一块在整份文档里的作用

这一块管理的是“测试资产”，不是“执行中的任务”。

这里的核心对象包括：
- 用例菜单
- 测试用例
- 用例属性信息
- 用例脚本文件

## 本主题包含的具体接口

### 这一组接口共用的请求体外壳

这组接口的请求体通常都会带：
- `method`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`

很多接口还会带 `ftpInfo`，原文中它通常包含：
- `ip`
- `port`
- `username`
- `password`
- `dir`

### 这一组接口共用的响应体外壳

响应体通常包含：
- `method`
- `requestId`
- `subSysType`
- `subSysId`
- `sessionId`
- `statusCode`
- `msg`

个别接口在成功时还会补充返回对象，例如新的 `testCaseId`。

### 1. 用例查询

- 接口标题：`用例查询`
- method：`getTestCaseAll`
- URL：`https://127.0.0.1:5001/api/testCase/getTestCaseAll`
- 请求方式：`POST`
- 原文用途：中心控制子系统查询二级子系统所有测试用例；由于消息可能过大，采用 JSON 文件传输方式。
- 请求体业务字段：
  - `ftpInfo`：可选，文件方式传输时使用
- 响应体业务字段：原文响应消息本身主要是通用回执；真正重要的业务载荷在“测试用例属性信息文件”里。
- 文件体重点：
  - 顶层 `datas`
  - 菜单节点重点：`name`、`id`、`isParent`、`children`
  - 用例节点重点：`name`、`id`、`type`、`runSubSys`、`depSubSys`、`depSubNe`、`relationCaseId`、`priority`、`satelliteCount`、`stationCount`、`durate`、`execSteps`、`checkPoints`、`inputPars`、`preHandle`、`afterHandle`
- 最值得和甲方确认的字段：
  `datas`：甲方到底消费树形菜单还是只关心叶子用例。
  `type`：用例类型枚举是否就是当前文档里的 `land/orbit`，有没有扩展值。
  `depSubSys` / `depSubNe`：甲方是否会直接用它们做协同编排。
  `checkPoints`：这些检查点只是展示，还是会进入甲方验收口径。

### 2. 用例脚本文件上传

- 接口标题：`用例脚本文件上传`
- method：`testCaseFileUpload`
- URL：`https://127.0.0.1:5001/api/testCase/testCaseFileUpload`
- 请求方式：`POST`
- 原文用途：中心控制子系统需要二级子系统上传用例脚本文件时使用。
- 请求体业务字段：
  - `testCaseId`
  - `ftpInfo`
- 响应体业务字段：
  - `testCaseId`
  - 其余为通用回执字段
- 最值得和甲方确认的字段：
  `testCaseId`：脚本文件和用例资产的绑定是否始终一对一。
  `ftpInfo.dir`：目录由谁决定，甲方是否会给固定规则。

### 3. 用例属性及脚本修改

- 接口标题：`用例属性及脚本修改`
- method：`modifyTestCase`
- URL：`https://127.0.0.1:5001/api/testCase/modifyTestCase`
- 请求方式：`POST`
- 原文用途：中心控制子系统更新二级子系统某一用例的属性及脚本文件时使用。
- 请求体业务字段：
  - `testCaseInfo`：用例属性对象
  - `ftpInfo`：可选，若有脚本文件修改则提供
  - `testCaseInfo` 中重点包括：`name`、`id`、`type`、`runSubSys`、`depSubSys`、`depSubNe`、`priority`、`satelliteCount`、`stationCount`、`durate`、`execSteps`、`remark`、`inputPars`、`preHandle`、`afterHandle`
- 响应体业务字段：
  - `testCaseId`
  - 其余为通用回执字段
- 最值得和甲方确认的字段：
  `testCaseInfo`：修改时是全量传还是允许部分字段更新。
  `inputPars`：参数定义是否会直接被甲方页面生成表单。
  `preHandle` / `afterHandle`：这些动作是展示性信息还是执行编排输入。
  `ftpInfo`：为空时是否明确表示“仅改属性，不改脚本”。

### 4. 用例脚本文件下载

- 接口标题：`用例脚本文件下载`
- method：`testCaseFileDownload`
- URL：`https://127.0.0.1:5001/api/testCase/testCaseFileDownload`
- 请求方式：`POST`
- 原文用途：测试用例脚本修改后通知二级子系统下载脚本文件；此接口只下载脚本文件，不更新用例属性。
- 请求体业务字段：
  - `testCaseId`
  - `ftpInfo`
- 响应体业务字段：
  - `testCaseId`
  - 其余为通用回执字段
- 最值得和甲方确认的字段：
  `testCaseId`：是否一定能唯一定位到脚本版本。
  `ftpInfo.dir`：路径中是否包含版本信息或覆盖策略。

### 5. 新增用例菜单

- 接口标题：`新增用例菜单`
- method：`addTestCaseMenu`
- URL：`https://127.0.0.1:5001/api/testCase/addTestCaseMenu`
- 请求方式：`POST`
- 原文用途：中心控制子系统添加二级子系统用例菜单时使用。
- 请求体业务字段：
  - `parentId`
  - `id`
  - `name`
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `parentId`：菜单树是否保证只有单父级结构。
  `id`：菜单 id 是否由甲方统一规划，还是我方可自行生成。

### 6. 新增用例

- 接口标题：`新增用例`
- method：`addTestCase`
- URL：`https://127.0.0.1:5001/api/testCase/addTestCase`
- 请求方式：`POST`
- 原文用途：中心控制子系统新增二级子系统用例时使用。
- 请求体业务字段：
  - `testCaseInfo`
  - `ftpInfo`
  - `testCaseInfo` 中重点包括：`parentId`、`name`、`id`、`type`、`runSubSys`、`depSubSys`、`depSubNe`、`remark`、`inputPars`、`preHandle`、`afterHandle`、`fileHandle`
- 响应体业务字段：
  - `testCaseId`：若二级不支持一级指定的用例 id，可回填二级新生成的 id
  - 其余为通用回执字段
- 最值得和甲方确认的字段：
  `id`：是否允许甲方指定失败后由二级回填新的用例 id。
  `parentId`：新增时菜单层级是否必须预先存在。
  `fileHandle`：这部分只是元数据还是后续执行控制输入。
  `ftpInfo`：新增用例时脚本是否必传，还是允许后补。

### 7. 用例删除

- 接口标题：`用例删除`
- method：`deleteTestCases`
- URL：`https://127.0.0.1:5001/api/testCase/deleteTestCases`
- 请求方式：`POST`
- 原文用途：中心控制子系统删除二级子系统用例时使用。
- 请求体业务字段：
  - `testCaseIds`：支持多个用例 id
- 响应体业务字段：无额外业务字段，主要是通用回执。
- 最值得和甲方确认的字段：
  `testCaseIds`：是否允许批量删除菜单节点和用例节点混传。
  `testCaseIds`：删除是物理删除还是逻辑删除。

## 重要内容

### 1. 用例查询不是一个简单列表接口，而是整个用例资产结构的入口

原文在“用例查询”里说明：
- 中心控制子系统查询二级子系统所有测试用例
- 由于消息可能过大，采用 JSON 文件传输方式
- 文件后缀为 `.json`

更重要的是，这一节后面给出了“测试用例属性信息文件格式定义”，这里其实是后续很多接口的背景结构。

### 2. 用例结构里同时包含菜单层和用例层

原文通过 `isParent`、`children` 等结构，把菜单和具体用例放在同一个树形结构里。

阅读时建议把“菜单”和“用例”当成同一棵资产树上的两种节点，而不是两个完全分离的对象。

### 3. 用例属性里已经包含了执行相关背景信息

用例资产不仅有名称和 id，还包含很多与执行有关的描述，例如：
- 用例类型
- 运行子系统类型
- 配合子系统类型
- 依赖设备类型及数量
- 关联用例信息
- 优先级
- 星数、站数
- 执行时长
- 执行步骤说明
- 检查点及期望结果
- 输入参数
- 前处理、后处理、文件生成控制

这说明用例资产不是静态目录项，而是带执行语义的配置资产。

### 4. 脚本文件相关接口一共有三类动作

原文把脚本相关能力拆成了三块：
- 脚本文件上传
- 用例属性及脚本修改
- 脚本文件下载

阅读时要注意：
- “上传脚本”更偏传文件
- “属性及脚本修改”是同时处理属性与脚本的复合动作
- “下载脚本”是通知二级去取脚本文件，不等于修改属性

### 5. 用例菜单维护和用例维护是两类动作

原文单独定义了：
- 新增用例菜单
- 新增用例
- 用例删除

并特别说明删除菜单也使用“用例删除”接口，由二级自行判断是菜单还是用例。

这说明在这套接口里，菜单和用例虽然结构不同，但在部分维护动作上共用能力入口。

### 6. 新增用例时，脚本文件本身也被纳入新增流程

“新增用例”这一节不是只传用例属性，还允许携带脚本文件信息 `ftpInfo`。说明新增用例在文档里被视为“属性 + 脚本”一体化接入场景。

## 这一块建议重点精读的内容

### 1. 用例属性信息文件格式定义

这是整块内容里最值得反复回看的部分。它相当于告诉你：
原文里一个“用例”到底包含哪些信息。

### 2. 用例脚本上传 / 修改 / 下载三节之间的边界

如果后续要继续拆更细，首先就应该把这三节的动作边界理顺。

### 3. 新增用例时的属性结构

“新增用例”里又把一遍用例属性结构展开了，相当于从维护动作角度再次定义了一遍用例对象。

## 读这一块时最容易混淆的地方

### 1. 用例资产维护 和 测试任务执行

这里的“新增用例”“删除用例”“修改脚本”都不等于“执行一次测试”。

### 2. 脚本文件下载 和 中心侧直接下发脚本内容

原文描述的是通过文件信息让二级子系统下载脚本，不是把脚本正文直接塞进消息。

### 3. 菜单节点 和 用例节点

它们在树里长得很像，但语义不同。阅读时要借助 `isParent` 来区分。
