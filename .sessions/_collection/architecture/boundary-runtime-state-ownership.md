# 运行主状态与状态边界

> 状态：首轮正文
> 阶段：03-architecture
> 作用：回答系统未来的运行主状态应围绕哪些对象组织、哪些状态属于主状态、哪些只能降为派生/展示/记录/缓存状态，以及这些结论将如何约束后续收发主链、任务系统和 SCOE 专题。

## 1. 本文要解决的状态问题边界

### 1.1 当前事实

- `03-architecture` 当前已经先后确认了架构前红线、决策顺序、功能清单压力点，以及桌面能力统一收口方式的当前推荐方向。
- 现有系统最明显的结构问题，不是“状态不够多”，而是“很多不同性质的状态被混写在同一共享状态面里”，并被页面、Store、任务执行器、接收链、显示链和 SCOE 反复回读。
- 后面对接真正压到当前骨架的点，不是单个接口，而是：
  - 中心测试任务上下文
  - 用例执行上下文
  - 生命周期控制语义
  - 统一对外运行状态面
  - 结果归口链路

### 1.2 判断 / 倾向

本文只回答以下问题：

- 未来运行主状态应围绕哪些核心对象组织。
- 哪些状态是运行主状态，哪些只能是派生状态。
- 哪些状态属于展示态、记录态、缓存态。
- 哪些状态必须单点归口，哪些不得继续被多域共享回读当作事实源。

本文当前不回答以下问题：

- 接收主链和发送主链的最终分层稿。
- 任务系统内部引擎实现和对象字段设计。
- SCOE 的最终层级定位。
- 任意 store、class、schema、接口字段的命名与结构终稿。

它在五个架构决策中的位置是：

1. 继承“桌面能力统一收口方式”的边界。
2. 先把运行主状态语言立住。
3. 再用这套语言去约束收发主链、任务系统和 SCOE 三篇正文。

### 1.3 待决问题

- “用例执行上下文”是否在后续文档中保持为一级运行对象，还是收敛为“中心任务上下文”的内层结构，本文先给出倾向，不做字段级拍板。

## 2. 运行主状态的对象中心

### 2.1 当前事实

- 后面对接要求系统围绕“中心下发任务并稳定执行”的主线运转，而不是继续围绕本地发送工作台组织状态。
- 现有发送任务对象、页面工作台状态、接收显示状态、历史记录状态，虽然都在运行中变化，但它们并不处于同一层级。
- 当前最关键的错位点已经明确：
  - “中心测试任务”不是“现有发送任务”的同义词。
  - 现有发送任务更接近本地发送执行能力的任务对象，不等于未来的中心任务主上下文。

### 2.2 判断 / 倾向

未来运行主状态应围绕五类核心对象组织：

1. 子系统运行主体
   - 表示当前子系统是否在线、是否空闲、是否处于执行中、是否异常。
   - 这是对外协同必须稳定提供的统一运行状态面。

2. 中心任务上下文
   - 表示当前承接的是哪一个中心任务、当前任务是否存在、是否已开始、是否已被停止、是否已结束。
   - 它是整个运行主线的一级对象，不能再退化成页面局部任务。

3. 用例执行上下文
   - 表示当前执行到哪个用例、该用例处于何种生命周期、该用例携带的参数和执行结果归属于哪个任务。
   - 它是任务执行过程中的当前运行骨架，而不是展示层附属信息。

4. 生命周期控制状态
   - 表示当前允许的控制语义，如等待触发、等待调度、执行中、停止中、已完成、已停止、异常中止。
   - 这里强调的是控制语义本身，而不是某个页面按钮是否可点。

5. 结果归口状态
   - 表示当前任务和当前用例的结果是否已形成事实、是否已归集、是否已可对外上报或交付。
   - 它服务的是结果闭环，不等于历史记录或展示摘要。

这五类对象之间的关系应理解为：

- 子系统运行主体是最外层统一状态面。
- 中心任务上下文是当前运行主线的一级骨架。
- 用例执行上下文是任务上下文中的当前执行骨架。
- 生命周期控制状态和结果归口状态分别为任务/用例骨架提供“可控”和“可交付”的事实语义。

其中应被视为运行骨架级对象的是：

- 子系统运行主体
- 中心任务上下文
- 用例执行上下文
- 生命周期控制状态
- 结果归口状态

其中不应再被视为运行骨架级对象的是：

- 页面工作台状态
- 现有发送任务监控视图状态
- 接收显示页当前值面板
- 历史记录与测试工具记录
- 任意“为了方便展示而拼出来”的综合状态

### 2.3 待决问题

- 中心任务上下文与本地发送执行上下文之间，是否需要在后续专题中单列一篇边界补充说明；本文当前只先明确二者不是同层对象。

## 3. 状态分类与归口原则

### 3.1 当前事实

- 当前系统中 `receiveFramesStore.groups`、`mappings`、`allReceiveFrameData`、任务对象状态、页面监控状态、显示记录状态、SCOE 状态等，被多个分支同时读取或回写。
- 这导致“接收中的当前值”“工作台显示态”“历史记录”“本地执行对象状态”经常被误拿来充当运行事实源。
- 红线已经明确：
  - 主状态不得多点写入。
  - 展示态、记录态、缓存态不得反向塑造运行主状态语义。
  - 下游能力域不得通过共享状态回读替代明确边界输入。

### 3.2 判断 / 倾向

未来应使用以下分类：

#### 3.2.1 主状态

定义：

- 直接决定系统当前运行事实、控制语义和对外状态面的状态。

至少包括：

- 子系统运行主体状态
- 中心任务上下文
- 当前用例执行上下文
- 生命周期控制状态
- 当前结果归口状态

归口原则：

- 同一语义只能有一个正式写入归口。
- 其他层只能读取、订阅或基于它派生，不得并行维护另一份同义事实。

#### 3.2.2 派生状态

定义：

- 由主状态计算出来，用于监控、聚合、排序、摘要或外部展示适配的状态。

至少包括：

- 当前进度百分比
- “是否可点击开始/停止”的界面态
- 当前任务摘要卡片
- 页面中的任务分类统计
- 统一对外状态面中的简写说明文本

规则：

- 派生状态允许跨域读取。
- 但它不能反向定义主状态，也不能在主状态缺失时充当临时事实源。

#### 3.2.3 展示态

定义：

- 只服务于页面、组件和工作台交互的状态。

至少包括：

- 对话框开关
- 当前选中 tab、筛选项、排序项
- 监控面板展开折叠
- 页面本地工作台中的临时编辑值

规则：

- 展示态不得承担跨页面运行主职责。
- 展示态不得持有中心任务生命周期控制语义。

#### 3.2.4 记录态

定义：

- 用于追溯、审计、历史查看、测试工具展示、报告归档的状态。

至少包括：

- 历史记录
- 运行日志
- 用例完成后的留痕结果
- 任务完成后的报告副本
- SCOE 测试工具收发记录

规则：

- 记录态可以保留已经发生过的事实。
- 但记录态不是当前运行事实本身，不能再被回读来决定“系统现在该怎么编排”。

#### 3.2.5 缓存态

定义：

- 为了解析、匹配、表达式计算、输入复用、性能优化而保留的运行中缓存。

至少包括：

- 接收帧当前值缓存
- 表达式计算缓存
- 映射索引缓存
- 输入侧最近一次解析结果缓存

规则：

- 缓存态可以服务主链处理。
- 但缓存态不是中心任务主状态，不得越级充当跨域编排依据。

### 3.3 单点写入原则

- 子系统运行主体只能由统一运行主状态归口层写入。
- 中心任务上下文只能由中心任务归口层写入。
- 用例执行上下文只能由当前执行归口层推进。
- 生命周期控制状态只能由控制语义归口层推进，页面和组件只能发起控制动作，不能直接改写事实语义。
- 结果归口状态只能由结果归集归口层落定，展示层和记录层只能消费结果，不得先定义结果。

### 3.4 禁止反向塑造主状态语义的规则

以下状态不得继续被反向当作主状态事实源：

- 现有发送任务监控列表中的任务状态
- 页面工作台中的“当前正在执行”显示
- `receiveFramesStore.groups` 这类面向接收配置和显示回填的共享状态
- 历史分析、数据显示、测试工具记录中的最近结果
- 任意从多个共享状态拼出来的“当前系统状态”摘要

### 3.5 待决问题

- 哪些现有结果状态应归为记录态，哪些仍应保留为运行中的事实状态，需要等“任务系统归口方式”正文进一步收紧。

## 4. 当前系统中最需要被降级或剥离的状态类型

### 4.1 当前事实

- 现有系统中最容易被误当成事实源的，是接收共享状态、发送任务本地状态、页面工作台状态和显示/历史记录状态。
- 这些状态之所以危险，不在于它们不重要，而在于它们当前同时承担了“自己这层需要看”和“别的层拿来决定流程怎么走”两种职责。

### 4.2 判断 / 倾向

当前最需要被降级或剥离的状态类型包括：

1. 页面 / 组件持有的运行职责
   - 页面与对话框可以保留配置、触发入口和监控视图。
   - 但不得再长期持有中心任务上下文、生命周期控制语义和运行主清理职责。

2. 现有发送任务对象中的“全局任务”地位
   - 现有发送任务应降为本地执行能力中的执行对象或调度对象。
   - 它可以是未来执行骨架的下层能力，但不能直接等同于中心任务主状态。

3. 接收共享状态的事实源地位
   - `groups`、`mappings`、接收缓存等可以继续存在，但它们应被明确为输入侧缓存、映射或派生视图。
   - 它们不能继续被任务系统、SCOE、显示域反复回读并充当统一运行事实源。

4. 显示态和记录态的回灌作用
   - 数据显示、历史沉淀、任务监控、测试工具记录都不应反向塑造运行语义。
   - 它们只能消费运行事实，不能替运行主状态补定义。

5. 领域状态对总状态面的直接占位
   - SCOE、接收链、发送链各自的领域状态可以存在。
   - 但它们必须通过显式边界输入影响统一运行主状态，而不能各自直接宣布“系统当前就是这个状态”。

如果这些状态不先降级，后续三篇文档会出现的失真是：

- 收发主链会继续把共享状态回读当作主编排前提。
- 任务系统会继续把页面工作台和现有发送任务误当成主骨架。
- SCOE 会继续以领域特例状态直接占据统一运行语义。

### 4.3 待决问题

- 用例执行上下文在当前文档中是否需要进一步从“中心任务上下文”里单列成一级章节，本文先保留当前倾向，留待后续正文视复杂度决定。

## 5. 对后续三篇架构文档的约束接口

### 5.1 当前事实

- 运行主状态如果不先立住，后续三篇都会被现有共享状态和本地工作台语义带偏。
- 尤其是：
  - 接收链当前会显式推进任务触发并被多个分支共享回读。
  - 任务系统当前会把页面、Store、composables 和定时器桥接揉在一起。
  - SCOE 当前既有接收入口特判，也会直接进入统一发送和标准任务生命周期。

### 5.2 判断 / 倾向

#### 5.2.1 对“接收主链与发送主链组织方式”的直接影响

- 接收主链只能提供明确的输入事实、触发信号和处理结果，不得再通过共享状态回读承担中心任务主状态归口。
- 发送主链只能消费明确的任务/用例边界输入，不得通过页面状态、显示状态或接收缓存去补齐自身运行语义。
- 收发主链都可以维护本链路局部运行状态，但这些状态不能越级替代统一运行主状态。

#### 5.2.2 对“任务系统归口方式”的直接影响

- 任务系统正文必须明确：中心任务上下文与本地发送执行对象不是同一个层级。
- 生命周期控制状态和结果归口状态必须在任务归口中有明确正式入口，不能再由页面或监控视图代持。
- 任何暂停、停止、等待触发、等待调度、完成、终止语义，都必须先落在统一运行主状态语言之下，再谈执行器细节。

#### 5.2.3 对“SCOE 的架构位置”的直接影响

- SCOE 可以保留领域特例，但不得直接占有统一运行主状态定义权。
- SCOE 的领域状态、完成条件和执行结果，必须通过显式边界接口影响任务/用例/子系统状态，而不能通过共享状态回读侧向塑造总状态面。
- 是否把 SCOE 视为一级能力域，后续可以再拍板；但无论怎么拍，它都不能绕开本文定义的主状态归口原则。

### 5.3 待决问题

- 哪些状态口径必须等“任务系统归口方式”正文时再进一步收紧，本文先保留为后续继承点，不提前落到实现层。

## 6. 当前结论 / 待决问题 / 下一步建议

### 6.1 当前结论

- 未来运行主状态不能再围绕本地工作台组织，而必须围绕“子系统运行主体 + 中心任务上下文 + 用例执行上下文 + 生命周期控制状态 + 结果归口状态”组织。
- “中心测试任务”已经可以明确视为一级运行主对象；现有发送任务只能作为本地执行能力中的对象，不能再被视为统一任务主状态。
- 页面状态、显示状态、历史记录状态、接收缓存状态和测试工具状态，都不应继续承担运行主语义。
- 主状态必须单点归口；派生状态允许跨域读取，但不得反向塑造主状态；展示态、记录态、缓存态不得再充当事实源。
- 本文拍板的重点不是具体对象字段，而是后续三篇都必须继承的状态语言和边界纪律。

### 6.2 待决问题

- 中心任务上下文与本地发送执行上下文之间，是否需要单独一篇专题补充边界关系。
- 用例执行上下文是否在后续正文中继续保持一级运行对象，还是收敛为任务上下文内部层次。
- 哪些现有结果状态应归为记录态，哪些仍应保留为运行中的事实状态。
- 哪些状态口径必须等“任务系统归口方式”正文时再进一步收紧。

### 6.3 下一步建议

1. 后续先写“接收主链与发送主链组织方式”，但必须直接继承本文的状态分类，不再让接收缓存和页面工作台承担事实源职责。
2. 再写“任务系统归口方式”，重点回答中心任务上下文、用例执行上下文和本地发送执行对象之间的层级关系。
3. 最后在“SCOE 的架构位置”中，处理领域特例如何进入统一运行骨架，而不是反向定义统一运行骨架。
