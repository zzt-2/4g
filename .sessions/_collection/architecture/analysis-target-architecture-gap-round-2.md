# 功能清单与目标架构对比（第二轮）

> 状态：第二轮对比正文
> 阶段：03-architecture
> 作用：在 `03-桌面能力统一收口方式.md` 到 `07-SCOE 的架构位置.md` 已完成首轮闭合的前提下，重新对照[后面对接所需功能清单.md](D:/code/frontend/dongfanghong/后面对接所需功能清单.md)，判断目标架构是否已经足以承接功能清单，并据此决定是否进入 `04-specs`、优先切哪一组 spec。

## 第一部分：目标架构已经提供的稳定落点

### 当前事实

`03` 到 `07` 已经把本轮真正需要的骨架边界立住，当前已经明确的稳定落点主要有五组：

1. 桌面能力统一收口已经明确
   - 渲染层不能再把 Electron 原生能力当作业务正式入口。
   - 桌面能力必须先进入项目级统一收口层，再由主进程承接层处理资源、外设、网络、定时器、文件等能力。
   - 这意味着“谁可以直接接触桌面能力、谁不可以”已经有稳定语言。

2. 统一运行骨架已经明确围绕五类对象组织
   - `子系统运行主体`
   - `中心任务上下文`
   - `用例执行上下文`
   - `生命周期控制状态`
   - `结果归口状态`
   - 这意味着后续功能不必再借页面状态、共享缓存或监控视图来充当事实源。

3. 接收主链与发送主链已经有正式输入 / 输出语言
   - 接收主链只消费明确入站输入，输出标准接收事实、运行输入、记录 / 展示订阅输入或领域接管输入。
   - 发送主链只消费显式发送请求与上下文引用，输出标准发送结果、运行输入、结果归口输入或领域后续处理输入。
   - 这意味着“接收链不是任务系统本身，发送链不是中心任务本身”已经拍板。

4. 任务系统已经明确是独立能力域
   - 它位于统一运行骨架中的任务归口层。
   - 它负责推进中心任务上下文、用例执行上下文、生命周期控制状态和与任务执行相关的结果归口状态。
   - 现有发送任务对象已经被降级为局部执行载体，不再等同于中心任务主骨架。

5. SCOE 的层级与接入方式已经明确
   - SCOE 不是一级骨架能力，而是通过显式边界接入统一骨架的二级领域模块。
   - 它只能通过显式领域入口、显式发送请求、显式任务接入与统一骨架发生关系。
   - 它自己的领域完成条件、反馈语义、测试工具记录保留在领域边界内。

### 判断 / 倾向

当前目标架构已经足以回答下面这些架构级问题：

- 功能清单里的能力应该优先落在统一运行骨架、收发主链、任务系统，还是落在领域模块。
- 哪些能力可以直接进入对象与字段级收紧，哪些能力会因为缺少边界定义而在 spec 阶段失真。
- 哪些地方可以继续把“共享状态回读、页面代持、领域特判内嵌”排除在正式设计之外。

但当前也存在一个没有被 `03` 到 `07` 正式展开的缺口：

- 目标架构已经把“内部运行骨架”立住了。
- 但“中心协同接入与对外交付边界”还没有作为单独主题写清。
- 因此，和中心直接交互的能力虽然已经能看出应落在哪些内层骨架上，但外层协同边界仍缺一个正式落点说明。

### 待决问题

- 是否需要在进入部分 `04-specs` 之前，补一篇只处理“中心协同接入与对外交付边界”的窄范围专题。
- 如果补，该专题应只回答“对外请求 / 响应 / 查询 / 心跳 / 结果交付由哪一层归口”，不重新争论 `03` 到 `07` 已拍板的内部骨架边界。

## 第二部分：功能清单逐项对照目标架构

### 当前事实

从目标架构视角重新看，功能清单中的能力已经不再是同一层问题，而是自然分成四组：

1. 统一运行骨架直接承接的能力
   - 例如任务上下文、用例执行上下文、生命周期、结果归口。
2. 依赖任务系统与收发主链边界的能力
   - 例如接收任务、启动 / 停止任务、用例推进、结果回流。
3. 依赖结果归口与交付闭环的能力
   - 例如用例结果上报、任务汇总、报告生成、运行记录。
4. 依赖“中心协同接入与对外交付边界”的能力
   - 例如状态查询、心跳、基础响应、报告交付，以及所有中心侧请求 / 响应投影。

### 判断 / 倾向

下表给出第二轮对照结论：

| 功能项 | 在目标架构中的主要落点 | 主要依赖的已拍板边界 | 进入 `04-specs` 前还缺什么 | 当前判断 |
| --- | --- | --- | --- | --- |
| 子系统身份信息 | `子系统运行主体` 之上的对外协同投影视图 | `04` 统一运行主状态 | 还缺“对外协同接入边界”来说明身份如何投影给中心 | 有落点但仍需专题收紧 |
| 最小设备信息 | 对外协同投影视图，必要时作为身份信息的附属对象 | `04` 统一运行主状态 | 还缺最小暴露口径与对外投影边界 | 有落点但仍需专题收紧 |
| 接收测试任务 | 对外协同接入层进入 `中心任务上下文` / `用例执行上下文` | `04` 主状态对象、`06` 任务系统归口 | 还缺“中心任务请求如何翻译成任务系统正式输入” | 有落点但仍需专题收紧 |
| 从任务中解析测试用例和参数 | `中心任务上下文` 与 `用例执行上下文` | `04` 主状态对象、`06` 任务系统归口 | 主要缺对象与字段级 spec，不再缺架构落点 | 已有稳定架构落点 |
| 任务上下文管理 | `中心任务上下文` 的正式归口 | `04`、`06` | 主要缺任务对象组 spec | 已有稳定架构落点 |
| 任务启动 | 对外控制输入进入任务系统，推进生命周期控制状态 | `04` 生命周期控制、`06` 任务系统 | 还缺“启动请求”在对外边界的正式入口 | 有落点但仍需专题收紧 |
| 任务停止 | 对外控制输入进入任务系统，推进 `停止中 / 已停止` | `04` 生命周期控制、`06` 任务系统 | 还缺“停止请求 / 停止结果”在对外边界的正式入口 | 有落点但仍需专题收紧 |
| 执行状态维护 | `子系统运行主体`、`中心任务上下文`、`用例执行上下文`、`生命周期控制状态` | `04`、`06` | 主要缺对象级和状态口径级 spec | 已有稳定架构落点 |
| 状态查询响应 | 对外协同查询层，消费统一运行主状态投影 | `04` 统一运行主状态 | 还缺“查询边界与状态投影对象”专题 | 有落点但仍需专题收紧 |
| 心跳 | 对外协同保活层，消费 `子系统运行主体` 的基础投影 | `04` 统一运行主状态 | 还缺“心跳由谁发、带什么基础状态”的边界说明 | 有落点但仍需专题收紧 |
| 用例执行结果上报 | `结果归口状态` 形成用例级结果后，对外交付层负责上报 | `04` 结果归口、`06` 任务系统 | 还缺“上报边界”以及关键结果字段清单 | 有落点但仍需专题收紧 |
| 任务级结果汇总 | 任务系统推进任务结果归口，结果域形成任务级汇总对象 | `04` 结果归口、`06` 任务系统 | 主要缺任务结果对象组 spec | 已有稳定架构落点 |
| JSON 报告生成 | 结果 / 报告域消费任务级汇总对象，生成固定报告对象 | `04` 结果归口、`06` 任务系统 | 还缺报告对象字段与结构 spec | 有落点但仍需专题收紧 |
| 报告交付 | 对外交付层消费报告对象并交付给中心 | `04` 结果归口、`06` 任务系统 | 还缺“交付边界、交付方式、回执语义” | 仍存在较大架构风险 |
| 基础响应与错误反馈 | 对外协同接入层统一承接请求响应与错误归一 | `04` 主状态、`06` 任务系统、`05` 收发主链输入输出纪律 | 还缺“对外响应 / 错误模型”的正式边界 | 有落点但仍需专题收紧 |
| 结果字段整理 | 用例结果对象、任务结果对象、报告对象的字段组 | `04` 结果归口、`06` 任务系统 | 主要缺字段级确认，不再缺主架构落点 | 已有稳定架构落点 |
| 运行记录 | 记录态消费任务系统与结果归口输出 | `04` 状态分类、`06` 任务系统 | 还缺记录对象范围与归档口径 | 有落点但仍需专题收紧 |
| 异常结果归一化 | 任务系统与对外协同层之间的统一错误语义 | `04` 生命周期 / 结果归口、`06` 任务系统 | 还缺错误分类对象和对外投影边界 | 有落点但仍需专题收紧 |

对于“当前不应该先做大的内容”，第二轮对照结论如下：

- 单独参数查询 / 参数配置
  - 当前目标架构里没有必要把它抬成独立骨架能力，应继续作为非主线能力处理。
- 完整用例管理、脚本上传下载全套、全过程步骤上报、升级 / 重启、完整告警体系
  - 当前都不应抢占 `04-specs` 的第一批入口。
  - 它们不是当前“任务进来 -> 执行 -> 用例结果回传 -> 任务报告交付”主线的前置条件。

### 待决问题

- 对外协同边界是否要统一承接“任务接收、状态查询、心跳、结果上报、报告交付、基础响应”这整组能力。
- 报告交付最终是“接口回传”“文件上传”还是“文件交付后通知”，这会影响交付 spec 的对象边界。
- 关键结果字段仍未确认，因此“用例结果上报”和“JSON 报告生成”虽然已有落点，但字段级 spec 不能假装已经闭合。

## 第三部分：哪些能力现在已经可以进入 `04-specs`

### 当前事实

`03` 到 `07` 已经把内部运行骨架立住，因此并不是所有能力都还卡在架构层。真正已经具备架构前提的，是那些主要落在统一运行骨架、任务系统和收发主链边界上的能力。

### 判断 / 倾向

当前已经适合进入 `04-specs` 的能力组，建议明确分成三组：

1. 统一运行骨架对象组
   - `子系统运行主体`
   - `中心任务上下文`
   - `用例执行上下文`
   - `生命周期控制状态`
   - `结果归口状态`
   - 这些对象已经有清晰架构前提，可以开始做对象边界和字段收紧。

2. 任务系统输入 / 推进对象组
   - 任务输入对象
   - 用例与参数承载对象
   - 任务控制输入对象
   - 任务推进结果对象
   - 这组对象直接继承 `04` 和 `06`，已经不需要再等待新的内部架构决策。

3. 主链边界对象组
   - 接收主链输出给任务系统的正式输入对象
   - 任务系统发给发送主链的显式发送请求对象
   - 发送主链返回任务系统的标准执行结果对象
   - 这组对象直接继承 `05` 和 `06`，现在进入 spec 是合适的。

当前不适合直接进入 `04-specs` 的能力组，也应明确分开看：

1. 对外协同接入与查询组
   - 子系统身份信息
   - 最小设备信息
   - 状态查询响应
   - 心跳
   - 基础响应与错误反馈
   - 如果现在硬写，很容易把“中心侧查询 / 响应协议”直接糊成任务系统内部对象。

2. 对外交付组
   - 用例执行结果上报
   - JSON 报告生成
   - 报告交付
   - 其中任务结果对象本身可以进入 spec，但真正的“怎么交给中心”还不宜直接下笔。

3. 关键结果字段清单
   - 架构前提已经足够。
   - 但业务字段事实尚未确认，因此此处不能把“字段未定”误写成“架构未定”。

如果现在硬进 `04-specs`，最容易出现的失真有四类：

1. 把中心协同请求对象直接写成任务系统内部对象
2. 把状态查询和心跳直接写成某个 Store 的读取结果
3. 把报告交付写成“任务结果对象的附带动作”，而不是独立交付边界
4. 把外部事务语义、回执语义、错误语义提前冻结在内部对象命名里

### 待决问题

- 是否接受“先进入内部骨架 spec，再补对外协同边界专题，再写北向协同 spec”这个分段推进方式。
- 关键结果字段如果继续未定，`04-specs` 是否先只写结果骨架对象，不提前拍具体字段全集。

## 第四部分：是否还需要补新的 `03-architecture` 专题

### 当前事实

`03` 到 `07` 已经解决的是：

- 桌面能力从哪里统一进入
- 运行骨架围绕哪些一级对象组织
- 接收主链和发送主链通过什么输入 / 输出语言对接
- 任务系统为什么是独立能力域
- SCOE 为什么是通过显式边界接入的二级领域模块

但功能清单里仍有一整组能力，主要面对的是“中心怎么和我方系统发生协同”，而不是“我方内部运行骨架怎么组织”。

### 判断 / 倾向

当前判断是：**还需要补一个窄范围的新专题，且它应当写在 `03-architecture`。**

建议主题可以收敛成：

- `09-中心协同接入与对外交付边界.md`

这个专题不需要重开总架构，只需要回答四件事：

1. 中心侧请求由哪一层统一承接
2. 统一运行骨架如何向中心侧投影状态、心跳和结果
3. 用例结果上报与任务报告交付属于哪一层的职责
4. 对外成功 / 失败 / 拒绝 / 不可执行等响应语义由哪一层统一归口

之所以不能把这个缺口留到 spec 阶段再补，原因有三点：

1. 这不是字段问题，而是边界问题
   - 不先立住对外协同边界，spec 很容易把“中心请求对象”和“任务系统内部对象”写成同一个东西。

2. 它会同时影响多项功能
   - 子系统身份信息
   - 接收测试任务
   - 任务启动 / 停止
   - 状态查询响应
   - 心跳
   - 用例结果上报
   - 报告交付
   - 基础响应与错误反馈

3. 它决定 `04-specs` 的切分方式
   - 没有这个专题，`04-specs` 很可能被迫按接口标题切，而不是按稳定边界切。

需要特别说明的是：

- 这个缺口不是在推翻 `03` 到 `07`。
- 它只是在内部骨架已经稳定之后，把“北向协同边界”补齐。
- 它的范围应当严格受控，不能扩写成新的总架构正文。

### 待决问题

- 是否接受把这个专题限定为“对外协同边界专题”，而不是把它写成新的任务系统或总线专题。
- 如果不补该专题，则需要明确接受一个后果：`04-specs` 只能先写内部骨架对象组，不能一次性覆盖全部对外能力。

## 第五部分：进入 `04-specs` 的推荐切入顺序

### 当前事实

当前最清晰的是内部骨架，最不清晰的是中心协同边界与报告交付边界。因此，进入 `04-specs` 时不宜按“接口列表”切，而应按“已经稳定的边界”切。

### 判断 / 倾向

推荐切入顺序如下：

1. 先写“统一运行骨架对象组” spec
   - 原因：这是所有功能项的共同事实底座。
   - 先把 `子系统运行主体 / 中心任务上下文 / 用例执行上下文 / 生命周期控制状态 / 结果归口状态` 收紧，后面所有对外能力才不会回退成页面或共享状态拼装。

2. 再写“任务系统输入 / 推进对象组” spec
   - 原因：功能清单主线本质上是“中心任务进入后如何被内部稳定推进”。
   - 这一组直接决定任务输入、用例参数承载、启动 / 停止控制、推进与结束口径。

3. 再写“收发主链边界对象组” spec
   - 原因：任务系统真正落地执行时，必须通过接收主链输入和发送主链请求 / 结果来闭环。
   - 如果这组对象不先收紧，后面用例推进和结果归口仍会被共享状态回读拉偏。

4. 然后补“中心协同接入与对外交付边界”专题
   - 原因：这一步不是写 spec，而是给后续北向协同 spec 立边界。
   - 没有这一步，状态查询、心跳、结果上报、报告交付都会把外部协议对象和内部对象混在一起。

5. 最后写“北向协同对象组” spec
   - 建议包含：
   - 身份 / 设备投影对象
   - 状态查询与心跳对象
   - 任务接入 / 启动 / 停止的对外请求响应对象
   - 用例结果上报对象
   - 任务报告对象与报告交付对象

当前最不该先写的一组 spec 是：

- `报告交付 + 基础响应与错误反馈 + 状态查询 / 心跳` 这一组

原因不是它们不重要，而是：

- 它们最依赖“中心协同接入与对外交付边界”。
- 如果先写，最容易把外部协同语义误压进内部对象。

### 待决问题

- 北向协同对象组是否拆成两批：
  - 先写状态 / 心跳 / 查询
  - 再写结果上报 / 报告交付
- 如果报告交付方式迟迟不定，是否先只写报告对象，不写最终交付对象。

## 第六部分：当前结论 / 待决问题 / 下一步建议

### 当前事实

- `03-architecture` 第一轮已经完成，内部骨架不再处于关键边界悬空状态。
- 功能清单里的多数主线能力，已经能够在目标架构中找到清晰的内层落点。
- 当前真正还没有显式补齐的，不是内部骨架，而是“中心协同接入与对外交付边界”。

### 判断 / 倾向

当前可以得出六个结论：

1. 现在已经可以进入 `04-specs`，但应先进入“内部骨架 spec”，不是一次性全量进入。
2. 已有稳定架构落点的能力，主要集中在：
   - 任务上下文管理
   - 用例与参数承载
   - 执行状态维护
   - 任务级结果汇总
   - 结果字段整理
3. 有落点但仍需继续收紧边界的能力，主要集中在：
   - 接收测试任务
   - 启动 / 停止任务

   - 用例结果上报
   - JSON 报告生成
   - 基础响应与错误反馈
   - 运行记录
   - 异常结果归一化
4. 当前仍存在较大架构风险的能力，最主要是：
   - 报告交付
   - 原因不是报告对象没地方放，而是交付边界、交付方式、回执语义还没有正式归口。
5. 当前最好补一篇新的 `03-architecture` 专题：
   - 范围限定在“中心协同接入与对外交付边界”
   - 不重开 `03` 到 `07` 已拍板的内部架构
