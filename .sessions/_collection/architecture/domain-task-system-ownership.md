# 任务系统归口方式

> 状态：首轮正文
> 阶段：03-architecture
> 作用：在继承运行主状态语言与收发主链边界的前提下，回答任务系统在总架构中的层级、归口职责、生命周期控制语义，以及它与接收主链、发送主链和后续 `07-SCOE 的架构位置.md` 之间的正式关系。

## 第一部分：本文要解决的任务系统问题边界

### 1.1 当前事实

- `00-architecture-pre-red-lines.md` 已经明确：
  - 页面和业务组件不得长期持有跨页面运行主职责。
  - 主状态不得多点写入。
  - 任务生命周期语义不得继续混叠。
  - Store 不得同时长期承担主状态、总编排和多下游副作用分发三类职责。
- `04-运行主状态与状态边界.md` 已经把未来运行骨架收敛到：
  - 子系统运行主体
  - 中心任务上下文
  - 用例执行上下文
  - 生命周期控制状态
  - 结果归口状态
- `05-接收主链与发送主链组织方式.md` 已经明确：
  - 接收主链不是任务系统本身。
  - 发送主链不是中心任务上下文本身。
  - 接收主链只能输出明确的触发候选输入或等价运行输入。
  - 发送主链只能消费显式发送请求和上下文引用。
- 扫描文档已经证实，当前所谓“任务系统”并不是边界清晰的独立域，而是发送页工作台、Store、composables、接收触发和定时器桥接共同拼出来的混合体。
- 当前真正待决的不是“还要不要任务系统”，而是“任务系统到底是不是独立运行骨架，以及它如何与统一运行主状态、接收主链和发送主链对接”。

### 1.2 判断 / 倾向

本文只回答以下问题：

1. 任务系统在总架构中是独立能力域，还是发送工作台的局部能力。
2. 任务生命周期控制、时间推进、调度、等待触发、停止、完成、清理分别归谁负责。
3. 中心任务上下文、用例执行上下文、本地发送执行对象之间是什么层级关系。
4. 任务系统如何消费接收主链输出，如何向发送主链发出显式请求。
5. 哪些能力属于任务系统，哪些能力不再属于任务系统。
6. 本文如何约束后续 `07-SCOE 的架构位置.md`。

本文不回答以下问题：

- 任务引擎内部类图。
- 状态机字段终稿。
- command / event 字段 schema。
- 定时器实现细节。
- 页面改造步骤。
- 迁移波次、实施顺序和代码改造路线。
- SCOE 的最终层级拍板。

本文的目标不是设计任务引擎，而是先拍板一件更基础的事：

- 任务系统在未来总架构里归哪一层。
- 它负责什么事实语义。
- 它不再负责什么。

### 1.3 待决问题

- 用例执行上下文在后续专题里是否继续保持为一级运行对象，还是只在中心任务上下文内部展开，本文先按一级运行对象书写。
- 结果归口状态在后续专题中是否需要继续拆成“运行中归集”和“对外交付”两层，本文先只明确边界，不落拆分终稿。
- `00` 中已经把“暂停”列为必须区分的控制语义之一，但本文先聚焦你本轮要求的主状态语义集合，暂停 / 恢复的精细归口留待后续专题继续收紧。

## 第二部分：任务系统的架构定位

### 2.1 当前事实

- `04` 已经明确：中心任务上下文是一级运行对象，现有发送任务不能直接等同于未来中心任务主上下文。
- `05` 已经明确：接收主链和发送主链都只能通过显式输入 / 输出与统一运行骨架发生关系，不能继续建立在共享状态回读编排之上。
- `01-scan/02-task-system-current-state.md` 已经证实：当前任务系统仍深度依附页面工作台、接收映射、统一发送和定时器桥接，并没有形成稳定的独立边界。
- `02-功能清单与当前架构问题对比.md` 已经证实：真正压到当前骨架的是中心任务上下文、生命周期控制语义、统一状态面和结果交付链，而不是“给现有发送任务再多加几个字段”。

### 2.2 判断 / 倾向

先明确任务系统不是谁：

- 任务系统不是页面工作台。
- 任务系统不是发送主链。
- 任务系统不是接收主链。
- 任务系统不是监控视图。
- 任务系统不是某个领域模块的别名。
- 现有发送任务对象不等于中心任务上下文。

在未来总架构里，任务系统应被视为独立能力域，而不是发送体系内的局部能力。

更准确地说，它应位于“统一运行骨架中的任务归口层”：

- 它不直接等于统一运行主状态。
- 但它是统一运行主状态里“中心任务上下文 / 用例执行上下文 / 生命周期控制状态 / 任务相关结果归口状态”的正式推进者。
- 它位于接收主链和发送主链之间，但不是二者的拼接层，而是消费显式输入、推进运行事实、再输出显式请求的编排归口层。

它与统一运行主状态的关系应理解为：

- 统一运行主状态定义“系统当前事实是什么”。
- 任务系统负责把与任务相关的那部分事实稳定推进出来。
- 页面、Store、composable、定时器桥接都可以作为实现载体存在，但都不再拥有架构级定义权。

它与接收主链、发送主链的连接关系应理解为：

- 接收主链向任务系统输出明确的触发候选输入或等价运行输入。
- 任务系统基于这些输入决定当前任务 / 当前用例是否推进、等待、停止或完成。
- 任务系统向发送主链发出的不是“请你帮我理解任务”，而是显式发送请求和上下文引用。
- 发送主链返回给任务系统的不是“新的任务定义”，而是标准化执行结果和可归口的结果输入。

它不能继续由页面、Store、composable、定时器桥接混合持有的原因有四个：

1. 这会继续违反主状态单点写入原则。
2. 这会继续让生命周期语义混叠在页面动作、局部发送状态和定时器状态里。
3. 这会继续让接收缓存、监控视图、展示态回头定义运行事实。
4. 这会继续让任务系统看起来像“接收触发 + 发送任务”的拼接产物，而不是独立运行骨架。

### 2.3 待决问题

- 任务系统与结果归口状态之间，后续是否需要再拆出单独的“结果交付归口层”，本文当前倾向是：任务系统推进结果事实，报告和交付能力另立边界。
- 手工工作台发送未来是否全部走任务系统统一入口，还是保留一部分显式非任务发送入口，本文先不拍板。

## 第三部分：任务系统负责的核心对象与职责

### 3.1 当前事实

- `04` 已经把中心任务上下文、用例执行上下文、生命周期控制状态、结果归口状态确认为运行骨架对象。
- 扫描文档已经证实，当前现有发送任务对象更接近“本地发送执行任务对象”，其字段天然绑定发送实例、目标、触发条件、定时器和页面配置语义。
- 当前等待触发、等待调度、执行、停止、清理等职责分散在页面工作台、监控组件、Store、执行器、触发监听器和 boot 清理入口上。

### 3.2 判断 / 倾向

任务系统真正负责的核心对象，应按以下层级理解：

#### 3.2.1 中心任务上下文

归口：

- 由任务系统正式负责。

定位：

- 它是一级运行对象。
- 它表示“当前承接的是哪一个中心任务、当前任务是否存在、是否正在推进、是否已停止、是否已结束”。

任务系统对它负责的内容包括：

- 建立当前任务运行事实。
- 推进任务级生命周期语义。
- 决定当前任务是否进入等待触发、等待调度、执行中、停止中、已停止、已完成或异常中止。
- 决定任务级结果是否进入可归集、已归集、可交付等结果归口事实。

#### 3.2.2 用例执行上下文

归口：

- 由任务系统推进。

定位：

- 它是中心任务上下文内的当前执行骨架。
- 它表示“当前执行到哪个用例、该用例基于什么前提进入执行、当前结果属于哪个中心任务”。

任务系统对它负责的内容包括：

- 选择和切换当前用例。
- 推进用例级等待触发、等待调度、执行、停止、完成和异常中止语义。
- 决定何时进入下一用例，何时结束当前任务。

#### 3.2.3 本地发送执行对象

归口：

- 它不再是一级运行对象。
- 它应降为任务系统内部或发送主链前的局部执行载体。

定位：

- 它承载一次或一组本地发送动作所需的实例、目标、批次、局部游标和局部执行状态。
- 它可以由任务系统生成、替换、丢弃，也可以被发送主链消费。
- 但它不拥有中心任务定义权，也不拥有用例生命周期定义权。

因此，三者的层级关系应明确为：

1. 中心任务上下文：一级运行骨架。
2. 用例执行上下文：任务骨架内的当前执行层。
3. 本地发送执行对象：用例执行层之下的局部执行载体。

除对象层级外，任务系统还应正式负责以下职责：

1. 生命周期控制状态推进
   - 任务级和用例级生命周期语义由任务系统归口推进。
   - 页面和组件只能发起控制意图，不能直接改写事实。

2. 时间推进
   - 任务何时开始等待、何时到时、何时超时、何时转入下一步，由任务系统决定。
   - 定时器桥接层只提供时间资源，不拥有时间语义解释权。

3. 调度
   - 任务系统决定当前该推进哪个用例、当前该发出哪一类显式请求、当前是否仍应等待。

4. 等待触发 / 等待调度
   - 这是任务系统对显式输入进行解释后的正式生命周期事实，不再由监听器注册状态或定时器注册状态冒充。

5. 停止
   - 停止请求由任务系统吸收并推进成“停止中 -> 已停止”或“停止中 -> 异常中止 / 完成”的正式事实。

6. 完成
   - 完成必须建立在任务系统确认“当前任务应做的用例已结束，且结果归口状态已落定”之后。

7. 清理
   - 任务系统负责决定何时注销等待条件、何时结束局部执行对象、何时释放调度资源。
   - 页面卸载清理只能是退出场景下的兜底，不再是正常任务生命周期的一部分。

结果归口状态与任务系统的关系应收敛为：

- 结果归口状态不是历史记录，也不是报告页面。
- 它是运行骨架里“当前任务 / 当前用例结果是否已经形成事实、是否已经归集、是否已经可交付”的正式状态。
- 任务系统必须负责推进这部分事实。
- 但报告生成、结果展示、历史沉淀和对外报告交付，不应再混入任务系统内部。

### 3.3 待决问题

- 一个中心任务未来是否只允许一个当前用例执行上下文，还是允许并发用例执行上下文，本文先按“当前执行上下文”书写，不提前拍并发模型。
- 本地发送执行对象里哪些局部重试、局部延时仍可保留为发送域局部状态，哪些必须回收到任务系统调度层，留待后续专题继续收紧。

## 第四部分：生命周期语义、时间推进与清理的归口原则

### 4.1 当前事实

- 当前系统里的 `completed` 同时承接自然完成和手动停止后的结束语义，导致结束原因混叠。
- 当前 `pause / resume` 主要停留在状态切换层，尚未形成稳定控制语义闭环。
- 当前等待触发、等待调度、执行、清理等语义分散在任务对象状态、监听器注册、定时器状态、页面监控和 boot 清理入口上。
- 当前页面工作台和监控组件仍直接参与任务创建、启动、停止和后台继续运行决策。

### 4.2 判断 / 倾向

任务生命周期语义必须分层归口，不能再由局部执行状态、页面状态或监控状态混写。

#### 4.2.1 任务级正式语义

以下语义必须作为任务级正式事实存在，并由任务系统推进：

- `等待触发`
  - 当前任务已经建立，但尚未满足进入当前用例或下一推进步的触发前提。
- `等待调度`
  - 当前任务已经建立，但尚未到达允许推进的时间点或调度点。
- `执行中`
  - 当前任务已经占有运行主线，且存在一个被正式推进的当前用例执行上下文。
- `停止中`
  - 停止请求已被任务系统吸收，正在收束当前用例、监听、调度和清理动作。
- `已停止`
  - 任务已因显式停止而结束，不再继续推进。
- `已完成`
  - 任务要求的用例已经完成，且任务级结果归口状态已落定。
- `异常中止`
  - 任务因异常、边界错误或无法继续推进的故障而结束。

#### 4.2.2 用例级正式语义

以下语义必须作为当前用例执行上下文的正式事实存在，并由任务系统推进：

- `等待触发`
  - 当前用例的进入前提尚未满足。
- `等待调度`
  - 当前用例已被选定，但尚未到达执行时间点或调度点。
- `执行中`
  - 当前用例正在驱动发送、等待响应、判断完成或吸收结果。
- `已停止`
  - 当前用例因任务停止或显式控制而结束。
- `已完成`
  - 当前用例的结果已经落定并可归入任务结果。
- `异常中止`
  - 当前用例在推进中遇到异常而结束。

当前倾向是：

- `停止中` 优先作为任务级正式语义存在。
- 用例级是否需要单列长期 `停止中` 事实，留待后续专题判断；在第一轮架构里，当前用例更适合被理解为“响应任务级停止中而进入收束”。

#### 4.2.3 页面和组件可以做什么，不可以做什么

页面和组件可以做：

- 提供任务创建入口、控制入口和监控视图。
- 展示任务系统输出的派生状态。
- 提交显式控制意图，如开始、停止、手工触发。
- 持有自身对话框、筛选、选中项等展示态。

页面和组件不可以做：

- 长期持有中心任务上下文。
- 直接改写任务级或用例级生命周期事实。
- 通过本地变量、监控视图状态或按钮态定义“系统现在处于什么状态”。
- 持有正常路径下的时间推进、调度、清理职责。

#### 4.2.4 哪些局部执行状态不能再冒充生命周期事实

以下状态都只能降为局部执行状态、展示态或实现细节，不能再冒充正式生命周期事实：

- “监听器已经注册”。
- “定时器已经登记”。
- “当前发送器正在发送”。
- “监控面板某一行显示运行中”。
- “页面对话框关闭后选择后台继续运行”。
- “接收缓存里已经出现某个值”。
- “某个本地发送对象暂时为空闲 / 暂时没有实例”。

时间推进、调度和清理的归口原则应明确为：

- 时间语义归任务系统，不归定时器桥接层。
- 调度语义归任务系统，不归发送主链。
- 等待触发语义归任务系统，不归接收监听器。
- 清理语义归任务系统，不归页面卸载流程。

### 4.3 待决问题

- 暂停 / 恢复在未来是任务级正式语义、用例级正式语义，还是控制语义下的附属状态，本文先不展开，但它仍受 `00` 的红线约束。
- 某些局部重试、局部超时未来是否要提升为用例级正式语义，本文先不提前拍板。

## 第五部分：任务系统与接收主链、发送主链的边界关系

### 5.1 当前事实

- `05` 已经明确：接收主链只能输出触发候选输入或等价运行输入，发送主链只能消费显式发送请求和上下文引用。
- 扫描文档已经证实：当前任务触发仍由接收链显式推进，而条件判断又回读 `receiveFramesStore.mappings`，说明边界输入还没有被正式抽出来。
- 当前发送链与现有发送任务对象、本地工作台和局部执行状态纠缠较深，说明“显式请求 / 标准结果”的边界还没有立住。

### 5.2 判断 / 倾向

任务系统与接收主链、发送主链的关系，应收敛为“显式输入 / 显式输出”。

#### 5.2.1 接收主链能给任务系统什么输入

接收主链可以给任务系统的，只能是明确的边界输入，例如：

1. 触发候选输入
   - 表示某次接收事实已经满足“可供任务系统判断是否触发”的最小条件。
2. 等价运行输入
   - 表示某次接收事实虽不直接等于“触发”，但足以让任务系统推进当前用例或当前任务判断。

这些输入的共同特点必须是：

- 来源明确。
- 含义明确。
- 足以让任务系统独立做生命周期判断。
- 不需要任务系统再回头读接收缓存、接收显示态或历史态来补齐语义。

任务系统不应直接解释以下对象：

- 接收缓存本身。
- 接收显示状态。
- `groups / mappings / allReceiveFrameData` 这类共享状态面。
- 历史记录和最近值面板。

换句话说，任务系统消费的是“接收主链输出”，不是“接收 Store 当前长什么样”。

#### 5.2.2 任务系统如何把显式请求交给发送主链

任务系统向发送主链发出的，应是显式发送请求，而不是“把任务对象整包塞给发送链”。

该显式请求至少应只表达这些架构语义：

- 当前为什么发。
- 当前由哪个中心任务 / 哪个用例上下文发。
- 当前引用哪个目标、哪个载荷或哪个本地发送执行对象。
- 当前是否允许发送，以及发送后结果应归回哪个运行上下文。

这里的关键不是字段多少，而是边界关系：

- 发送主链只消费显式请求和上下文引用。
- 发送主链不负责解释中心任务语义。
- 发送主链不负责补齐用例执行上下文缺失信息。

#### 5.2.3 发送主链返回什么结果给任务系统

发送主链返回给任务系统的，应是标准化执行结果，而不是新的任务事实定义。

它至少应能支撑任务系统判断：

- 这次请求是否被接受。
- 这次发送是否执行、失败或被拒绝。
- 这次结果归属于哪个当前运行上下文。
- 这次结果是否足以推进当前用例完成、继续等待或进入异常中止。

结果归口状态由谁推进，仍应由任务系统决定，而不是由发送链局部状态反向宣布。

#### 5.2.4 为什么不能继续依赖共享状态回读补齐执行语义

如果继续依赖共享状态回读，会出现四种结构性失真：

1. 任务系统的正式输入不可见。
2. 缓存态、展示态、记录态会重新冒充运行事实。
3. 生命周期语义会重新散落到接收缓存、本地发送对象和页面监控状态中。
4. 后续 `07` 会被迫把 SCOE 写成共享状态上的一组领域特判，而不是显式接入任务系统或主链。

### 5.3 待决问题

- 接收主链输出未来是否需要继续拆成“触发候选输入”和“用例推进输入”两类正式对象，留待后续专题和 `04-specs` 收紧。
- 发送主链返回结果未来是否需要继续拆成“执行结果”和“结果归口输入”两层，本文先不落对象终稿。

## 第六部分：当前系统中最需要从任务系统剥离或降级的职责

### 6.1 当前事实

- 当前页面工作台和任务配置对话框不只负责展示和录入，还直接创建任务、启动任务，并决定后台继续运行。
- 当前监控视图直接操作启动、暂停、恢复、停止接口，并在事实意义上参与了生命周期解释。
- 当前现有发送任务对象天然携带“全局任务”色彩，但其实更接近本地发送执行对象。
- 当前触发监听语义依赖接收映射回读，说明任务系统尚未真正拥有独立的正式输入。
- 当前页面卸载清理仍被当作任务清理兜底，说明正常生命周期和退出清理还没有分开。

### 6.2 判断 / 倾向

当前最需要从任务系统剥离或降级的职责至少有五类：

#### 6.2.1 从页面工作台剥离跨页面运行主职责

页面工作台未来应降为：

- 配置入口。
- 控制入口。
- 派生状态展示面。

它不应再继续占有：

- 中心任务上下文定义权。
- 生命周期推进权。
- 正常路径下的时间推进和清理职责。

#### 6.2.2 从监控视图剥离生命周期解释权

监控视图未来应降为：

- 订阅任务系统派生状态的观察面。

它不应再继续承担：

- “任务到底处于什么正式状态”的解释权。
- “当前结束原因是什么”的事实定义权。

#### 6.2.3 把现有发送任务对象降为本地发送执行对象

现有发送任务对象未来更适合被降级为：

- 用例执行上下文之下的局部执行载体。
- 发送主链之前的本地请求承载对象。

它不应再继续承担：

- 中心任务上下文的替身。
- 统一任务生命周期的正式事实源。

#### 6.2.4 从任务系统剥离对接收共享状态面的直接依赖

未来任务系统不应继续直接回读：

- `groups`
- `mappings`
- `allReceiveFrameData`
- 接收显示态和历史态

这些对象仍可存在，但它们应降为：

- 接收主链内部状态。
- 显示 / 记录域消费状态。
- 缓存态或派生态。

#### 6.2.5 把时间资源和清理资源降为基础设施，不再冒充生命周期 owner

主进程定时器、渲染层局部延时和页面退出清理仍然可以存在。

但它们未来的地位应收敛为：

- 任务系统使用的基础设施资源。
- 退出场景下的补充兜底。

它们不应再继续承担：

- “任务现在处于等待调度”这一正式语义。
- “任务现在已经结束”这一正式结论。

### 6.3 待决问题

- 监控视图后续是否需要单独的读模型或投影视图，本文先不展开。
- 手工工作台是否要完全共享任务系统的派生状态面，还是保留一层工作台专属视图，本文先不拍板。

## 第七部分：对 `07-SCOE 的架构位置.md` 的约束接口

### 7.1 当前事实

- 当前 SCOE 已经同时侵入统一接收入口、统一发送链和标准任务生命周期。
- `07` 尚未写，但如果没有前一篇先把任务系统归口立住，SCOE 很容易被误写成“接收特判 + 发送特判 + 发送任务复用”的拼接物。

### 7.2 判断 / 倾向

`07` 必须继承本文的任务系统边界。

如果 SCOE 要复用任务系统，只能以显式边界接入，至少包括以下方式之一：

1. 作为任务系统的上游生产者
   - 以显式任务请求、显式控制请求或显式运行输入进入任务系统。
2. 作为任务系统的下游消费者
   - 消费任务系统输出的生命周期结果、任务结果或结果归口输出。
3. 作为发送主链之前的领域翻译层
   - 先把领域动作翻译成显式发送请求，再进入发送主链。

SCOE 不得这样侵入任务系统：

- 不得直接改写中心任务上下文。
- 不得直接改写用例执行上下文。
- 不得直接改写生命周期控制状态。
- 不得直接改写结果归口状态。
- 不得把固定来源、固定目标、领域完成条件、测试工具状态直接塞成通用任务语义。
- 不得通过接收缓存、显示态、记录态回读来侧向定义任务事实。
- 不得把自己的页面工作台、测试工具或领域局部状态抬升成任务系统的正式状态面。

留到 `07` 再拍板的问题应包括：

- SCOE 最终是一级能力域，还是显式领域模块。
- SCOE 哪些动作可以被翻译成通用任务输入 / 通用发送请求，哪些必须声明为边界例外。
- SCOE 的领域完成条件、反馈语义和结果交付，如何接到任务系统和结果归口状态上。
- SCOE 是否需要自身独立的领域执行上下文，以及它与用例执行上下文之间是什么关系。

如果不先把本文立住，`07` 会出现的失真至少有两类：

1. 把 SCOE 误写成“现有发送任务”的一个领域变体。
2. 把 SCOE 误写成“接收主链内部的一组长期特判”，从而继续侵入统一运行骨架。

### 7.3 待决问题

- SCOE 最终是否长期复用任务系统全部生命周期语义，还是只复用其中一部分运行边界，留待 `07` 结合领域事实继续拍板。

## 第八部分：当前结论 / 待决问题 / 下一步建议

### 8.1 当前结论

- 任务系统在未来架构中应被视为独立能力域，而不是发送工作台的局部能力。
- 它位于统一运行骨架中的任务归口层，负责推进：
  - 中心任务上下文
  - 用例执行上下文
  - 生命周期控制状态
  - 与任务执行相关的结果归口状态
- 中心任务上下文、用例执行上下文、本地发送执行对象的层级关系应明确为：
  - 中心任务上下文 > 用例执行上下文 > 本地发送执行对象
- 任务生命周期控制、时间推进、调度、等待触发、停止、完成、清理都应归任务系统，而不是继续分散在页面工作台、监控视图、发送本地状态和定时器桥接上。
- 接收主链与任务系统之间的正式边界是明确的触发候选输入或等价运行输入。
- 任务系统与发送主链之间的正式边界是显式发送请求和标准化执行结果。
- 页面工作台、监控视图、发送任务本地状态未来都应降级为入口、观察面或局部执行状态，不再占有任务系统定义权。
- 如果不先把任务系统归口立住，后续 `07` 很容易失真为“让 SCOE 去复用现有发送任务和接收特判”，从而把旧混乱重新包装成新架构。

### 8.2 待决问题

- 用例执行上下文的最终粒度和并发模型。
- 结果归口状态与对外交付能力的最终拆分方式。
- 暂停 / 恢复语义在任务级和用例级的正式归口。
- 接收主链输出、发送主链结果输入的正式对象口径。
- SCOE 复用任务系统时，到底复用哪些边界、保留哪些领域例外。

### 8.3 下一步建议

1. 后续写 `07-SCOE 的架构位置.md` 时，不再回头争论“任务系统是不是独立能力域”，而只在本文边界内回答 SCOE 如何接入。
2. 进入后续 `04-specs` 时，再把本文已经拍板的边界关系落成对象口径；不得反过来用字段设计改写本文层级。
3. 本文之后的所有正文，都应默认：
   - 页面工作台不是任务系统本体。
   - 现有发送任务不是中心任务上下文。
   - 共享状态回读不能再作为任务编排前提。
