# 桌面能力统一收口方式

> 状态：首轮决策草案
> 阶段：03-architecture
> 作用：回答渲染层访问桌面能力应如何统一收口，并给出当前阶段的推荐方向、备选方案和后续影响。

## 1. 这份文档要解决什么

当前项目已经确认的事实是：

- 现有系统存在一条主路径：`api/common -> window.electron -> preload -> main/ipc`
- 但这条路径并没有完全收口，仍存在边界例外
- 后续架构又存在一个真实待决问题：
  - 最终继续以 `preload/contextBridge` 为主
  - 还是在受控前提下允许 `nodeIntegration`

因此本文件要解决的不是“Electron 还能不能用”，而是：

- 渲染层访问桌面能力最终采用什么统一收口方式
- 在该方式下，业务层、收口层、承接层分别承担什么职责
- 当前例外资产如何被吸收或淘汰

## 2. 架构目标

这项决策至少要同时满足五个目标：

1. 业务代码不能散落直连桌面原生对象
2. 桌面能力访问路径要足够统一，便于后续重构和替换
3. 不能让 `preload`、业务 API 包装层和主进程承接层职责继续混在一起
4. 后续如果要重写部分骨架，桌面能力收口方式不能成为新的阻碍
5. 能支撑当前项目这种“本地桌面前端 + 高运行态编排”的复杂度，而不是只适合简单 CRUD 页面

## 3. 候选方案

### 3.1 方案 A：继续以 `preload/contextBridge` 为唯一正式收口

定义：

- 渲染层只能通过 `window.electron` 或其上层包装访问桌面能力
- `preload` 作为唯一桥接暴露层
- 主进程通过 `main/ipc` 承接所有桌面能力
- 禁止业务层直接依赖 Electron 原生对象

优点：

- 边界清晰，和现有主路径连续
- 安全边界和职责分层最容易说清
- 对后续 review、测试、替换桥接实现最友好

缺点：

- 会保留一层较重的桥接样板
- 对当前项目这种运行态复杂系统来说，接口面会继续膨胀
- 若后续大量内部能力都要通过桥接层暴露，维护成本偏高

适用前提：

- 团队愿意接受“桥接样板多一些，换边界明确”
- 后续不会频繁把桌面能力与渲染层业务逻辑强耦合

### 3.2 方案 B：受控开启 `nodeIntegration`，但仍保留统一业务收口层

定义：

- 渲染层允许在运行环境上直接获得 Node / Electron 能力
- 但业务代码仍不得散落直接访问 Electron 原生对象
- 必须保留一层项目级统一收口层，作为业务侧唯一正式入口
- 主进程承接逻辑、桌面资源管理和领域外设控制仍需集中归口

优点：

- 可以显著减少大量桥接样板代码
- 对当前这种本地桌面前端项目更贴近实际运行形态
- 在需要快速重建运行骨架时，工程摩擦更小

缺点：

- 如果收口纪律不严，很容易退化成“哪里方便就哪里直接 import Electron”
- 边界失控风险比方案 A 更高
- 需要更强的目录纪律和 review 纪律保证不扩散

适用前提：

- 团队明确接受“运行环境放宽，但业务入口仍统一收口”
- 后续架构文档能先把“哪些层可接触桌面原生能力、哪些层不行”写死

### 3.3 方案 C：双轨长期并存

定义：

- 一部分能力走 `preload/contextBridge`
- 一部分能力允许直接在渲染层访问
- 不强求形成单一路径

优点：

- 短期迁移阻力最小

缺点：

- 会把当前的边界不统一固化成长期结构
- 后续架构、spec、实现和 review 都会失去稳定边界
- 无法支撑后续重建指南写成“拿起来就能干活”的形式

结论：

- 该方案不应作为正式目标架构

## 4. 当前推荐

### 4.1 推荐结论

当前推荐采用 **方案 B：受控开启 `nodeIntegration`，但仍保留统一业务收口层**。

这不是在说“以后业务代码可以随便直连 Electron”，恰恰相反，推荐 B 的前提是：

- 运行环境可以放宽
- 但业务访问路径必须收紧

也就是说，推荐的是：

- 放宽运行时能力取得方式
- 不放宽业务层访问纪律

### 4.2 推荐理由

#### 理由 1：它更符合当前项目的真实定位

当前项目不是一个被严格浏览器沙箱约束驱动出来的前端项目，而是一个本地桌面前端，且运行主线复杂、状态重、外设与链路编排重。

这种项目如果继续把所有桌面能力都严格压在 `preload/contextBridge` 上，未必最适合重建效率。

#### 理由 2：它给后续重建留出更大操作空间

如果后面判断需要局部重写甚至较大范围重建，方案 B 更容易快速搭出新的运行骨架。

方案 A 虽然更纯，但容易让“桥接层样板”本身变成新的负担。

#### 理由 3：真正必须守住的不是桥接技术，而是统一收口纪律

从当前系统问题看，核心问题不是“有没有 preload”，而是：

- 入口不统一
- 例外资产真实存在
- 业务 API 包装层、边界桥接层、主进程承接层职责不清

所以真正要先守住的，是“统一收口纪律”，不是“必须保留某一种桥接技术”。

## 5. 决策后的边界定义

如果采用方案 B，后续必须同步确认以下边界：

### 5.1 业务层正式入口

- 页面、组件、Store、composables 不得直接把 Electron 原生对象当正式业务入口
- 所有桌面能力仍必须通过项目级统一收口层进入

### 5.2 桌面能力收口层职责

- 只负责能力暴露、调用转换、错误归一化、资源访问入口统一
- 不承载业务主逻辑

### 5.3 桌面能力承接层职责

- 主进程资源、外设、网络、串口、定时器、文件能力等仍需集中承接
- 不把主进程承接层写成渲染层业务脚本的延伸

### 5.4 边界例外处理

- 当前已存在的直连资产视为遗留例外
- 遗留例外允许暂存，但不得继续扩张
- 新增代码不得再创造“临时直连”口子

## 6. 这项决策对后续文档的直接影响

### 6.1 对运行主状态与状态边界

- 可以不再被 `preload` 桥接方式本身绑住
- 重点转向“主状态怎么归口、业务入口怎么统一”

### 6.2 对接收主链与发送主链

- 接收、发送、定时器、文件、网络等桌面能力都必须经统一收口层进入业务骨架
- 但不再要求它们都必须长成当前 `window.electron` 的形状

### 6.3 对后续开发/重建指南

后续真正像 `frontend.md` 那样的指南，可以写成：

- 桌面能力统一从哪一层 import
- 哪些目录允许接触桌面能力
- 哪些目录绝对不允许
- 错误、日志、资源释放怎么统一写

这会比“先规定必须保留 preload 主路径”更接近可执行指南。

## 7. 当前不在本文拍板的内容

本文不拍板：

- `nodeIntegration` 的具体安全参数
- 具体目录名和文件落点
- 统一收口层的最终 API 形状
- 具体哪些现有 `preload` 模块保留、哪些下沉、哪些删除
- 每个 IPC 或能力调用的字段级接口

这些内容应留给：

- 后续架构正文
- `04-specs`
- 最终的开发/重建指南

## 8. 当前结论

- 当前项目不应继续把“必须维持 `preload/contextBridge` 主路径”当成先验前提。
- 当前更重要的，是先把“桌面能力统一收口”作为硬边界立住。
- 在这个前提下，当前更推荐：
  - 受控开启 `nodeIntegration`
  - 但保留统一业务收口层
  - 严禁业务代码散落直连 Electron 原生对象
- 若后续你和 Claude 评审后认为方案 B 风险仍过高，再回退到方案 A 也仍然成立；但无论 A 还是 B，都不应接受方案 C 的长期双轨并存。
