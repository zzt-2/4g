## 1. Expression Engine 112 tests — 覆盖与缺口

### 覆盖的能力

来源: S007 第一节第 22-34 行

112 测试分布跨 8 个 spec 文件，当前文件已确认存在：
- `tokenizer.spec.ts` — 词法分析
- `parser.spec.ts` — AST 语法分析
- `compile.spec.ts` — 编译
- `evaluate.spec.ts` — 求值
- `conditional.spec.ts` — 条件编译
- `group.spec.ts` — 分组
- `integration.spec.ts` — 集成
- `performance.spec.ts` — 性能（P1: 10000 次 evaluateGroup < 50ms; P2: 1000 次 compileGroup < 100ms）

验收声明: "112 测试全绿，tsc 零错误，零 Vue/Pinia/Electron 依赖"

### 未覆盖的缺口

**集成到 receive/send/task 时的测试缺口**:

- (a) expression engine 自身的测试全在 `shared/expression/__tests__/` 内，属于**纯函数级单元测试**。它不验证 `receive/core/expression-pass.ts` 如何调用 engine API（如 `compileConditional` / `dependency` 排序），也不验证 `send/core/frame-resolver.ts` 中 `evaluateFieldExpressions` 的调用方式。
- (b) receive Phase 1 新增了 `expression-pass.spec.ts`（23 tests），覆盖了 receive 侧表达式集成，但这些测试仍用 fixture 数据而非真实帧配置文件。
- (c) send-real 新增了 `send-frame-resolver.spec.ts`（19 tests），覆盖了 send 侧表达式调用，但 S007 未说明是否测试了 `compileConditional` 而非 `compileGroup` 的差异行为（S007 第九节第 191 行提到"表达式用 compileConditional"这个设计决策）。
- (d) task-real 仍处于 brainstorm 阶段（S007 第十节），task 的表达式集成测试**完全不存在**。

## 2. Frame-Real "L2 消费者测试通过（receive 9 pass + runtime 28 pass）"

来源: S007 第二节第 54 行

### 含义

"L2 消费者测试"指 frame feature 的公开 API（`index.ts` 导出的 17 处引用）被下游 feature 消费时的兼容性验证。具体:

- **receive 9 pass**: `rewrite/src/features/receive/__tests__/` 下的测试在 frame-real 改动后仍全部通过，共 9 个测试用例。这验证了 receive 对 frame public API 的消费（如 `getFrame(id)` 返回深拷贝、`getSnapshot()` 返回完整快照）在帧字段扩展后未破坏。
- **runtime 28 pass**: `rewrite/src/runtime/__tests__/` 下的测试（当前有 4 个 spec 文件: `rewrite-runtime.spec.ts`, `bootstrap-integration.spec.ts`, `feature-wiring.spec.ts`, `routing-tick.spec.ts`）在 frame-real 改动后全部通过。

### 缺口

- 这是**回归兼容性验证**，不是专门的集成测试。测试的 9+28 数量来自已有的 feature/runtime 单测，frame-real 只验证了"改动后旧测试仍绿"。
- 没有测试验证 frame 变更（如新增 `expressionConfig`/`conditionConfig` 字段）对 receive 的**正向行为影响**——只验证了没破坏旧功能。
- send、task、command-ingress 等其他 frame 消费者未被纳入此 L2 验证范围。

### Phase 4 背压验证方式

来源: S007 第七节第 143-150 行

**验证内容**: routingTick 预算截断——`RoutingTickOptions.maxEventsPerTick` 参数，默认值 50，drain 后用 `slice(0, max)` 截断。

**验证方式**: 3 个新增单测在 `routing-tick.spec.ts` 中。这是**纯逻辑测试**，验证截断行为正确性（输入 N 条事件、设置 max=M、断言输出 <=M 条）。

**缺口**:
- 没有背压的端到端测试（如真实高频数据流入 routingTick 时是否真正触发截断并避免 UI 卡顿）。
- S007 第五节第 113 行提到 receive-real 拆分中 "receive-real-highspeed（推迟，背压压测、高速存储短路）"——背压压测被**显式推迟**。
- D4 决策中提到 "routingTick 预算截断 + 各 feature 独立节流"，但"各 feature 独立节流"部分未见测试证据。

## 4. Platform-Network-Transport AC1-AC3 "手动网络验证"

来源: S007 第四节第 93 行

### 具体含义

S007 第四节（第 76-95 行）描述了 platform-network-transport 的实施，改动涉及 3 层:
1. `src-electron/main/network-handlers.ts` — main 进程网络 handler
2. `src-electron/preload/index.ts` — preload IPC 路由
3. `src/platform/transport.ts` — renderer 侧传输 facade

"9 条 acceptance criteria 全部有证据（AC1-AC3 需手动网络验证）" 表示:
- AC1-AC3 涉及**真实 TCP/UDP 连接建立/断开/数据传输**行为，这些无法用 Vitest 单元测试覆盖（需要真实 Electron 环境中的 main 进程 IPC + 真实网络 socket）。
- 其余 AC4-AC9 可能涉及类型定义、接口对齐、cleanup 逻辑等可通过静态分析或 fixture 测试验证的项目。

### 缺口

- S007 未展开 AC1-AC3 的具体内容（需查看原始 design/checklist 才能确认）。
- "手动网络验证"意味着**从未执行过**——S007 总览表（第 230-240 行）中 platform-network-transport 的测试列写的是 "build pass"，即只有编译通过证据，无运行时测试证据。
- 这 3 条 AC 是集成测试的关键候选——需要真实 Electron + TCP/UDP 环境的端到端验证。

## 5. Connection 580+ tests 和 Send-Real 100 tests 的级别

### Connection 580+ tests

来源: S007 第八节第 170 行

S007 声明 "580+ 测试通过，9/9 项设计-代码对齐，10 条验收场景全部有证据"。

当前 connection 测试文件（6 个 spec 文件）:
- `connection-core.spec.ts` — core 层单元测试
- `connection-fake-adapter.spec.ts` — fake adapter 测试
- `connection-network-adapter.spec.ts` — **网络 adapter 测试**（topic-index 提到该文件"已有 Node `net` 模块先例，可在 Vitest 中直接起 TCP"）
- `connection-reconnect-service.spec.ts` — 重连服务测试
- `connection-reconnect.spec.ts` — 重连逻辑测试
- `connection-state-service-selector.spec.ts` — state/service/selector 测试

**级别判断**: 主要是**单元级 + fake adapter 级**。`connection-network-adapter.spec.ts` 可能包含真实 TCP 级测试（使用 Node `net` 模块），这是最接近集成级的测试。但 S007 同时标注 "CONN-RUNTIME-001/002、CONN-HW-001 待真实 TCP/UDP 环境验证"——说明 580+ tests 中不包含真实 TCP/UDP 环境的集成测试。

### Send-Real 100 tests

来源: S007 第九节第 193 行

当前 send 测试文件（5 个 spec 文件）:
- `send-core.spec.ts` — core 层单元测试
- `send-service-state-selector.spec.ts` — service/state/selector 测试
- `send-checksum-patch.spec.ts`（20 tests） — checksum 补丁专项测试
- `send-frame-resolver.spec.ts`（19 tests） — 帧解析专项测试
- `send-integration.spec.ts`（10 tests） — 名为"集成"但实际验证的是 pipeline 各步骤串联（9 步 pipeline），输入是 fixture 数据

**级别判断**: 全部是**单元级**。`send-integration.spec.ts` 虽然名字含 "integration"，但 S007 描述为验证 pipeline 步骤串联逻辑，输入是 fixture，不涉及真实 transport 写出。send 的 9 步 pipeline（validate -> resolve frame -> resolve field values -> apply factor -> build buffer -> post-build patch -> resolve target -> transport write -> emit result）中，transport write 步骤在测试中使用的是 fake adapter。

### 两者的共同缺口

- Connection 和 Send 的测试都不涉及**跨 feature 交互**（如 send 调用 frame 的 public API、send 通过 connection adapter 发出数据）。
- 不涉及**真实 Electron IPC 链路**。
- 不涉及**真实网络传输**。
