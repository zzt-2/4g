## 问题 2：S005 旧系统三线调研发现 + 新旧覆盖矩阵 P0/P1/P2 缺口

**来源：S005，「三、旧系统调研」+「四、新旧系统覆盖矩阵」段落**

### 三线调研发现

#### SCOE 调研

- **数据流**：接收源 → receiveFramesStore → scoeFrame.ts（帧识别→校验和→参数解析）→ useScoeCommandExecutor.ts（策略模式执行器）→ waitForCompletionConditions（异步轮询 100ms）
- **SCOE 标记方式**：通过 `isSCOEFrame` 标记从全局帧模板筛选，但有独立的帧实例管理 store 和完全独立的完成条件系统
- **需消灭**：跨 store 循环依赖、重复帧实例管理、SCOE 逻辑硬编码在通用接收流程、测试工具耦合核心 store
- **可复用模式**：策略模式执行器映射、统一执行上下文/返回值、分层验证流程
- **集成测试意义**：新系统 command-ingress feature 需验证策略执行器链路完整性，覆盖旧 SCOE 的命令执行→条件匹配→完成通知行为

#### 表达式引擎调研

- **旧系统规模**：1385 行，核心文件 useExpressionCalculator.ts + useFrameExpressionManager.ts + fieldValidation.ts
- **4 种数据源**：CURRENT_FIELD / FRAME_FIELD / GLOBAL_STAT / SCOE_DATA
- **旧系统能力**：已实现 Kahn 算法拓扑排序，缓存策略 configHash + 5 分钟 LRU
- **性能瓶颈**：接收帧字段计算高频路径（每帧 1 次），使用 `new Function` 动态求值
- **旧系统缺失**：无预编译机制、无表达式级缓存
- **新系统状态**：已实现手写 parser + 编译为 JS 闭包，单表达式预编译后求值 < 1us，50 表达式批量拓扑求值 < 20us
- **集成测试意义**：需验证表达式引擎从旧系统 4 种数据源正确获取变量值并批量求值

#### 可视化调研

- **星座图特殊数据管道**：hex → extractValuesFromHex → collectConstellationFieldData → ECharts scatter
- **关键隐患**：星座图无长期数据量上限，高频帧+长刷新间隔=内存累积
- **可复用**：CircularBuffer 循环缓冲区、ECharts 增量更新、时间格式化批量缓存
- **集成测试意义**：display feature 需验证数据量上限和内存控制

#### 交叉发现

- **三套独立的条件匹配系统**（SCOE 完成条件、表达式 condition、任务触发条件）→ 新架构已决策统一到 shared/condition/ 纯函数 + 各 feature 自管触发
- **可复用共用件**：拓扑排序、条件匹配逻辑、校验和计算、循环缓冲区、hex 解析
- **集成测试意义**：需验证 3 个消费方（receive/task/command-ingress）对统一条件匹配的使用一致性

## 问题 5：S014 阶段 7 进度盘点中声称 100% 完成但缺少 runtime 证据的 feature

**来源：S014，「阶段 7：进度收拢与甲方对接」段落 +「后续」段落**

### 事实

S014 阶段 7 声称"确认 10 个 service 全部 100% 完成"。这 10 个 feature 基于 S005 和 S014 上下文为：

| Feature | 完成声称 | Runtime 证据缺口 |
|---------|----------|------------------|
| frame | 100% | selector 浅拷贝未修复（R-3）；factor 字段在 S005 自审发现缺失，后续 design 已更新但实际修复状态未确认 |
| connection | 100% | 无明确 runtime 缺口，但 TCP/UDP transport 在阶段 3 设计完成后真实连接未验证 |
| receive | 100%（注：阶段 3 时 ~40%） | receive-real pipeline 仅 brainstorm + roadmap 拆分完成，设计和实现未启动；"100%"可能是对骨架完成度的声称而非真实接收能力 |
| send | 100% | design 完成，但真实串口/网络发送能力未验证（"帧发送无真实串口"） |
| task | 100% | selector 浅拷贝未修复（R-4）；真实任务执行链路（receive→task→send）未端到端验证 |
| command-ingress | 100% | 设计实现完成，但与旧 SCOE 行为等价性未验证 |
| storage-local-baseline | 100% | S014 后续明确指出"存储完全没接持久化"；"100%"指的是 service 接口完成，而非持久化能力 |
| display | 100% | selector 浅拷贝未修复（R-1）；可视化 UI 缺 ECharts 渲染组件 |
| settings | 100% | 无明确 runtime 缺口记录 |
| status | 100% | 无明确 runtime 缺口记录 |

### 关键判断

**"100% 完成"的含义是 service/core 层 API 设计+实现+单元测试通过，不等于 runtime 可操作性**。S014「后续」段落明确列出的未决项证实了这一点：

1. **持久化层**：所有 feature state 仍为内存闭包，storage "完全没接持久化"
2. **Runtime 真实能力**：serial/TCP 仍为 mock
3. **6 个页面**：UI 骨架存在，数据通路未接真实能力
4. **端到端链路**：fake adapter → routingTick → receive → task → send → result 的完整链路未验证（对应 S005 审计 R-8）

**集成测试关联**：这些"声称完成但缺少 runtime 证据"的 feature 是集成测试的重点目标。每个 feature 的 service API 可能单元测试全过，但在集成场景下（多 feature 协作、真实 adapter、持久化加载）的行为未经检验。
