### Q2: Task-real 间歇性测试失败的根因 / 30-40% 套件间歇性失败是否已解决

来源: S008 第九节（第 185-197 行）、后续（第 284 行）

**根因分析**:

- `runExecutionLoop` 是 fire-and-forget（无 await），多个测试使用真实 `setTimeout` 做延迟。
- 部分测试结束后 async 操作仍在后台运行，泄漏到后续测试，导致测试间状态污染。

**已实施的修复**:

1. `task-service.ts` finally 块从 `abortResolvers.delete(instanceId)` 改为 `lifecycle.abortInstance(instanceId)`，确保正常完成的任务也 resolve signal。
2. 两处 trigger 测试末尾加 `service.stopTask()` 清理。

**修复效果**:

- 单独运行每个测试通过。
- 套件运行仍有约 **30-40% 间歇性失败**。
- 失败模式从 `STACK_TRACE_ERROR` 改善为 `expected 'running' to be 'completed'`——说明部分改善但未根治。

**是否已解决**: **否**。S008 明确记录"未完成"，上下文接近满载交接给新对话。后续建议为 `vi.useFakeTimers()` 或对所有使用 settle 的测试统一加 `afterEach stopTask`。

**后续状态**: 在 S009-S015 中搜索"间歇性"、"30-40%"、"fakeTimers"均无匹配，说明该问题在后续 session note 中**未被记录为已解决**。待确认。

### Q5: Receive-real 最终验收的 globalParams 推迟，对集成测试的影响

来源: S008 第十二节（第 232-253 行）

**推迟原因**: globalParams 收集依赖 settings feature 完善。

**集成测试影响**:

- receive-real 的表达式引擎需要 globalParams 作为运行时变量输入。推迟意味着：
  1. 表达式求值只能消费帧内字段和 receive 自身的匹配输入，**无法消费全局配置参数**。
  2. 依赖 globalParams 的 receive 匹配规则在实际运行中无法测试。
  3. 下游消费者（storage bridge、display bridge）如果依赖 globalParams 影响后的匹配结果，其集成测试也受限。

**S008 记录的影响评估**: 推迟到 settings feature 完善后集成。验收结论为 PASS-WITH-KNOWN-GAPS，认为当前核心逻辑完整。

**对集成测试的具体影响**（推断，待确认）:
- expression-engine 集成测试中 globalParams 路径无法覆盖
- receive → storage/display 桥接测试中全局参数相关的匹配分支无法触发
- 不影响不依赖 globalParams 的匹配和解析路径

### Q6: Storage-real 的 6 个外部依赖 gap 具体是什么

来源: S008 第十三节（第 255-268 行）、storage design 4.11 节（blocked/deferred）、checklist STO-IMPL-005/010/011 和 STO-DEFER-001/002

S008 只提及 3 个类别（"真实文件系统适配器、Electron 平台集成、大数据量性能验证"），未逐条列出 6 个 gap。以下基于 storage design 的 blocked/deferred 段落和 checklist 的 validation 要求推断：

**推断的 6 个外部依赖 gap**:

| # | Gap | 来源 | 阻塞条件 |
|---|-----|------|---------|
| 1 | 真实文件系统适配器（platform file/path/dialog API schema 未定义） | design 4.11: "最终 file/path/dialog platform API schema 未定义"；STO-IMPL-005 validation 含 runtime + package validation | platform facade 就绪后才能替换 fake adapter |
| 2 | Electron 平台集成（preload/main/renderer 文件 API 桥接） | design 4.11: STO-IMPL-005 "adapter 不定义最终 preload/main/renderer API schema"；design 4.6 platform involvement | Electron preload/main 侧 API 未就绪 |
| 3 | 打包态 data path 验证（长期写入、权限、备份、清理） | design 4.11: "打包态 data path、长期写入、备份、清理和权限行为未验证"；STO-IMPL-011 | 需要实际打包后的 Electron 应用验证 |
| 4 | 大数据量性能验证（history 写入吞吐、CSV 大文件导出） | S008 验收段: "大数据量性能验证"；STO-TEST-001: "页面交互、真实文件对话框、打包态路径、高速存储和客户交付不由 Vitest 宣称完成" | 需要 runtime 环境下的性能压测 |
| 5 | 高速存储最终模型 | design 4.11: "高速存储最终模型、short-circuit 口径、高吞吐背压和真实链路验证 deferred"；STO-DEFER-001 | 最终模型设计 deferred，需 runtime/hardware validation |
| 6 | report/northbound delivery 边界 | design 4.11: "northbound HTTP/FTP delivery、完成通知、外部错误码和客户侧接收确认 not touched"；STO-DEFER-002 | 需要客户侧协议确认 |

**注意**: 以上 6 项为基于 design 文档和 checklist 推断的列表。S008 原文未逐条枚举，实际验收对话（ccee53b9）中可能有更精确的 6 项表述。推断结果中的 1-4 项与 S008 提及的 3 个类别高度吻合。第 5、6 项为 deferred 项，可能属于或不属于"6 个已知 gap"的口径。待确认。
