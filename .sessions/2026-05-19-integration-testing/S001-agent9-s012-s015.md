### Q1：LazyPersistence 模式在什么场景下数据会丢？

**事实发现：**

1. **启动中崩溃会丢数据。**
   - 来源：S012 第 59-76 行（启动加载流程描述）；代码 `rewriteRuntime.ts` 第 35-56 行。
   - 机制：`bootstrapRewriteRuntime()` 同步返回 runtime，然后异步执行 `initPersistenceAsync`。如果应用在 `initPersistenceAsync` 完成前崩溃或用户在加载完成前修改数据并触发 save，LazyPersistence 的 delegate 仍是 no-op，所有 save 操作静默丢弃（no-op 实现为空 async 函数）。
   - 影响范围：启动后到 `setDelegate` 完成之间，任何通过 `runtime.persistence.save*()` 的调用都无效。但 `load()` 在 no-op 下返回 `{}`（空对象），不会报错，UI 会显示空状态，用户可能误以为没有数据。

2. **save 时机未设计，运行时修改不会自动持久化。**
   - 来源：S012 第 125 行（后续段）："持久化 save 的触发时机未设计——当前只有手动 save，缺少'何时自动持久化'的策略（如帧变更后自动保存、定时保存、退出前保存等）"。
   - 代码验证：搜索 `saveFrames|saveConnections|saveSettings|saveAll`，仅在 `persistence.ts` 和 `rewriteRuntime.ts` 中定义和代理，无任何业务代码调用这些方法。frame service 的 `replaceFrames`、`addFrame`、`updateFrame`、`removeFrame` 等操作完成后均未触发 save。
   - 结论：当前持久化层有写能力但无触发点。除非有外部代码手动调用 `persistence.saveAll()`（搜索结果为不存在），否则所有运行时修改仅在内存中，关闭应用即丢失。

3. **异步竞争风险。**
   - 代码 `rewriteRuntime.ts` 第 51-53 行：`initPersistenceAsync` 是 fire-and-forget（不 await）。如果用户在应用启动后立刻操作并触发了 save（假设未来加了自动保存），而此时 `initPersistenceAsync` 尚未完成，save 会写入 delegate 尚未初始化的数据路径或直接走 no-op。
   - LazyPersistence 的 `setDelegate` 没有同步保护（无锁、无队列），理论上存在 delegate 在 save 执行过程中被替换的竞争窗口。

**数据丢失场景汇总：**

| 场景 | 是否会丢 | 原因 |
|------|---------|------|
| 启动中崩溃 | 丢 | LazyPersistence delegate 未设置，所有 save 走 no-op |
| 正常运行中修改帧/连接/设置 | 丢 | 无自动 save 触发点 |
| 运行中 `saveAll()` 手动调用 | 不丢（如果 delegate 已设置） | 写入磁盘 |
| 退出前 | 丢 | 无退出前 save 钩子 |
| `initPersistenceAsync` 执行中并发操作 | 丢或行为未定义 | fire-and-forget + 无同步保护 |

### Q2：RealLocalMaterialAdapter 软删除策略是否测试过？

**事实发现：**

1. **无任何测试文件。** 搜索 `*persistence*test*`、`*material*adapter*test*`、`*storage*test*` 均无结果。RealLocalMaterialAdapter 无配套单测。

2. **软删除实现细节（来源：S012 第 24 行；代码 `real-local-material-adapter.ts` 第 65-81 行）：**
   - `deleteMaterial` 先读取原始内容，再写入 `${id}.deleted` 文件（含 `deletedAt` 时间戳和 `original` 数据）。
   - 原始 `.json` 文件未删除，仅创建了一个 `.deleted` 标记文件。
   - ENOENT（文件不存在）视为成功（幂等语义）。

3. **未验证的缺口：**
   - `.deleted` 文件创建后，原始 `.json` 文件仍存在。`readMaterial` 不检查 `.deleted` 标记，可能读回已删除数据。
   - 无清理逻辑。S012 第 122 行确认："实际文件清理逻辑未实现"。
   - `listMaterials` 依赖 `_index.json`，delete 操作不更新 `_index.json`，可能导致已删除条目仍在列表中出现。
   - 打包态路径未验证。S012 第 123 行："存储路径 `${userData}/dongfanghong/state/` 未在打包态下验证（runtime blocker）"。

**结论：软删除策略未测试，且存在 read/delete/list 语义不一致的设计缺口。**

### Q3：connections 和 settings 的恢复状态

**事实发现：**

1. **只实现了 frames 的启动加载。** 来源：S012 第 121 行；代码 `rewriteRuntime.ts` 第 72-77 行。
   - `initPersistenceAsync` 中只读取 `frames.json` 并调用 `frameService.replaceFrames()`。

2. **connections 和 settings 的现状：**
   - `FeaturePersistence.load()` 方法确实会并发读取三个文件（`persistence.ts` 第 57-67 行），返回包含 `frames`、`connectionConfigs`、`settings` 的结构。
   - 但 `initPersistenceAsync` 只对 frames 做了恢复处理（第 72-77 行），对 `load()` 返回的 `connectionConfigs` 和 `settings` 完全忽略。
   - 搜索 `replaceConnections` 和 `replaceSettings`：无任何匹配结果。connection service 和 settings service 不存在批量替换 API。
   - S012 第 121 行确认："持久化层当前只实现了 frames 的启动加载（`replaceFrames`），connections 和 settings 的启动恢复尚未接入，需在后续补全"。

3. **FeaturePersistence.save*() 方法已实现写入，但无调用方。** `saveConnections()` 和 `saveSettings()` 在 `persistence.ts` 中完整实现（第 75-82 行），但搜索整个 `rewrite/src/` 无业务代码调用它们。

**结论：connections 和 settings 的持久化 write 代码存在但无触发点；load 代码在 `load()` 中读取但在启动流程中被丢弃，无恢复逻辑。两端都是空转状态。**

### Q4：S015 的 10 个集测维度评估

S015 列出的 10 个维度：

| # | 维度 | 评估 | 意见 |
|---|------|------|------|
| 1 | TCP 环路 | 必要 | send → TCP → receive 是核心数据通路，必须覆盖 |
| 2 | UDP 环路 | 必要（如果 UDP 是正式支持的传输） | 待确认：UDP adapter 是否已实现。如果没有 UDP 实现，此项应降为 manual checklist |
| 3 | 多连接并发 | 必要 | composite adapter 是新增关键路径，S015 确认 H002 刚修了 3 个硬阻断 |
| 4 | connection 生命周期 | 必要 | 连接创建/断开/重连是用户高频操作 |
| 5 | receive 解析链路 | 必要 | 字节流→帧匹配→字段解析→表达式求值是核心解析链 |
| 6 | send 构帧链路 | 必要 | 帧定义→构造字节→目标路由→transport write |
| 7 | task 执行链路 | 必要 | step 执行→send→wait-condition→完成 是任务引擎核心 |
| 8 | command-ingress 链路 | 必要 | 外部命令→解析→任务触发，SCOE/甲方对接入口 |
| 9 | 持久化 | 必要 | 但当前实现缺口大（见 Q1-Q3），需决定测什么范围 |
| 10 | 旧 JSON 迁移 | 必要（如果迁移是 P0 功能） | frame service 已有 `importLegacyFrames` 方法 |

**可能的遗漏：**

1. **错误路径 / 异常恢复**：网络断开、写入失败、JSON 损坏、帧解析异常等。10 个维度全走 happy path，缺少 error path 覆盖。来源：S015 未提及任何错误场景维度。

2. **settings 变更链路**：settings 影响全局行为（如采样率、超时），但未单独列出。可合并到持久化维度，但如果 settings 有运行时生效逻辑，应独立覆盖。

3. **表达式引擎集成**：S015 第 5 维度提到"表达式求值"，但表达式引擎是 shared/ 独立模块，其跨 feature 集成（receive 条件匹配 + task wait-condition + send 字段计算）可能需要独立测试矩阵。

4. **selector 不变性**：CLAUDE.md 有"Selector 不可变约束"硬规则，但集测维度中无 selector 边界测试（验证通过 selector 拿到的对象修改后不影响源 store）。

**可能的多余：**

- 第 2 维度（UDP 环路）：如果 UDP adapter 未实现或不在 MVP 范围，此项应为 deferred。待确认。
