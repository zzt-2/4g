### Q1：S001 的 16 条质量红线（R1-R16）中，哪些直接要求集成测试级别的验证？

**来源：S001 第 5 节"质量红线制定"，表列 R1-R16 概要**

从 S001 session note 列出的 R1-R16 概要表中，以下规则直接要求跨模块/跨 feature 级别的验证，单 feature 内部单测不足以证明合规：

| 规则 | 要点 | 为什么需要集成测试 |
|------|------|--------------------|
| **R2** | Feature 归口显式，禁止跨 feature 内部写入 | 需验证 feature A 的实现代码不 import feature B 的内部路径；单元测试只覆盖单个 feature 内部逻辑，跨 feature 边界违规只能通过集成级 import/依赖扫描或实际 cross-feature 调用路径测试发现 |
| **R5** | Electron 能力边界必须窄 | 需验证 renderer 侧代码不直接访问 Node/Electron API、不通过裸 IPC 调用 main；这需要跨 renderer-preload-main 三层的集成级验证 |
| **R8** | Receive/Send 主链保持显式输入输出 | 需验证 receive 输出确实能作为 send 输入的正确来源、数据流经各 feature 边界时类型和语义不变；单 feature 内部无法验证端到端数据流 |
| **R10** | Northbound 必须是边界层，不是 feature 快捷方式 | 需验证 northbound 不直接调用 send/receive/result 内部方法，只通过 public API 交互；跨 feature 边界合规性验证 |
| **R11** | Result/Report/Delivery 三者必须分开 | 需验证 result 不直接生成报告文件、report 不直接做 HTTP 交付、delivery 不重新计算结果；三层分离的合规性只能跨层验证 |

以下规则主要在单 feature 内部可验证，但在集成层面也有价值：

| 规则 | 要点 | 集成测试价值 |
|------|------|-------------|
| **R3** | Core 逻辑必须可测试 TypeScript，不依赖 Vue/Electron | 集成级验证：runtime 装配后，feature core 仍不拉入 Vue/Electron 依赖 |
| **R7** | 运行主状态单一 owner | 集成级验证：跨 feature 交互时，feature A 不会写 feature B 的主状态 |
| **R9** | SCOE 必须作为声明的领域例外 | 集成级验证：SCOE 命令确实通过 task 执行引擎执行，SCOE adapter 不绕过 task 直接操作 send |
| **R12** | 旧 JSON 兼容不得污染新模型 | 集成级验证：导入旧 JSON 后，新 frame definition 的字段类型/校验规则保持新模型语义 |
| **R13** | 统计不得写回源模型 | 集成级验证：receive 统计、send 统计等 read model 不修改 frame definition 或 connection config |

R1（保留行为替换结构）、R4（UI 不是业务流程 owner）、R6（未在概要中展开）主要为设计原则，难以用自动化集成测试直接验证，需要人工审查或端到端手工测试。

### Q2：S001 第 10 节"Northbound 独立定位"中的"不得"项，哪些需要集成测试证明合规？

**来源：S001 第 10 节，逐条提取**

S001 第 10 节列出以下关键决策，其中隐含的"不得"项及其集成测试需求：

| "不得"项 | 原文依据 | 需要集成测试 | 说明 |
|----------|----------|-------------|------|
| deviceId 不得等同串口/网口 target | "deviceId 是业务设备身份，不是串口/网口 target" | **是** | 需验证 northbound 的 deviceId 映射路径确实经过转换层到达 connection target，而不是直接传递 serial port name 或 TCP address |
| setTestTask 不得等同旧本地 send task | "setTestTask 是甲方外部任务下发，不是旧本地 send task" | **是** | 需验证 northbound 的外部任务下发路径与本地 send task 的执行路径分离，两者不共享任务状态 |
| result/report/delivery 三层不得混淆 | "result/report/delivery 三者必须分开" | **是** | 同 Q1 的 R11 分析，需跨三层验证分离 |
| 旧 history/CSV 不得等同 TestReport 或 file delivery 闭环 | "旧 history/CSV/本地文件导出只能作为报告素材或 oracle 材料" | **是** | 需验证 report 的素材来源确实经过 result 层聚合，不直接消费旧 CSV 或旧 history |
| northbound 不得是 feature 快捷方式 | "Northbound 必须是边界层，不是 feature 快捷方式" | **是** | 同 Q1 的 R10 分析 |

全部 5 项均需要集成测试证明合规。原因：这些"不得"项都是跨模块边界约束，单 feature 单测无法验证"不做什么"——只有集成级依赖扫描、跨 feature 调用路径测试、或端到端数据流验证才能证明合规。

**但注意**：S001 记录时（2026-04-23 ~ 04-28），northbound 尚未实现。这些是架构级约束声明，不是可立即测试的验收标准。实际集成测试需要等 northbound 实现完成后才能执行。

### Q5：Send/Task/SCOE/Result 的 design 完成后，验收标准具体是什么？

**来源：S002 各节分别记录**

#### Send Feature

**验收标准**（S002 第 3 节）：
- Design 产出：`rewrite-send-design.md` + `rewrite-send-checklist.yaml`（10 steps + 16 checks）
- 实现 Verdict：pass-with-known-gaps
- 测试规模：20 suites / 249 tests（+96 新 send tests）
- 验证证据：lint 通过，边界检查通过
- 结构合规：core（types/encode/checksum/validation/clone）+ adapters（ports/fake）+ fixtures + state + selectors + services + index.ts
- 核心边界约束：send owns 单帧发送生命周期（构帧 -> target 解析 -> transport write -> 结果输出），send 不了解 task 存在

**集成测试层面的验收缺口**：
- "send 不了解 task 存在"——需集成测试验证 send 模块不 import task 的任何 export
- "单帧发送全链路"——需端到端测试验证从 buildFrame 到 transport write 的完整路径
- "target 解析"——需验证 send 的 target 解析与 connection 的 target 路由正确对接

#### Task Feature

**验收标准**（S002 第 4 节）：
- Design 产出：`rewrite-task-design.md` + `rewrite-task-checklist.yaml`
- Core 实现 Verdict：pass-with-known-gaps
- 测试规模：21 suites / 330 tests（+81 新 task core tests）
- 验证证据：lint 通过，边界检查通过
- 关键约束：stop != completed（northbound 要求 stopped 状态），error policy（retry/skip/stop/pause），condition matcher 使用自有 ConditionMatchInput 不依赖 receive 内部

**集成测试层面的验收缺口**：
- "condition matcher 不依赖 receive 内部"——需集成验证 task 的 condition matcher 确实不 import receive 内部路径
- task service/state/selectors 实现是 S002 结束时的"下一步最大缺口"
- task 与 send 的协作（task 调用 send public API）尚未实现

#### SCOE Feature

**验收标准**（S002 第 6 节）：
- Design 产出：`rewrite-scoe-design.md` + `rewrite-scoe-checklist.yaml`
- Verdict：仅 design 完成，无实现
- 核心决策：SCOE 是"外部指令适配器"模式的第一个实现，翻译命令为 TaskDefinition 通过 task 执行

**集成测试层面的验收缺口**：
- SCOE 未实现，无测试证据
- "翻译命令为 TaskDefinition"的端到端验证完全缺失
- SCOE -> task -> send 的三 feature 协作链路无测试

#### Result Feature

**验收标准**（S002 第 7 节）：
- Design 产出：`rewrite-result-design.md` + `rewrite-result-checklist.yaml`
- Verdict：仅 design 完成，无实现
- 核心决策：result owns 内部结果事实，所有素材经 TaskInstanceCompletion + TaskStepResult 统一进入，Result / Report / Northbound 三层分离

**集成测试层面的验收缺口**：
- Result 未实现，无测试证据
- "所有素材经 TaskInstanceCompletion 统一进入"——需端到端验证 task 完成后确实产生 TaskInstanceCompletion 并传入 result
- "三层分离"——同 Q1 的 R11 分析，需跨三层验证
