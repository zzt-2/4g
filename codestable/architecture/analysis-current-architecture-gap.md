# 功能清单与当前架构问题对比

> 状态：首轮正文
> 阶段：03-architecture
> 作用：把“后面对接所需功能清单”与当前已扫描出的系统结构问题对齐，判断哪些能力可包裹式承接，哪些能力会直接压到现有主链和运行骨架。

## 1. 本文要回答什么

本文不回答“现在就要不要重写”。

本文只回答三件事：

1. [后面对接所需功能清单.md](D:/code/frontend/dongfanghong/后面对接所需功能清单.md) 会把压力压到当前系统的哪些结构点
2. 哪些点更像外围对接能力，可以包裹式增加
3. 哪些点一旦真正落地，很可能逼出更大范围的重构或重写判断

## 2. 对比口径

本次对比统一采用三类判断：

- `可包裹承接`
  - 现有系统外侧加一层上下文、接口或交付能力，短期不必先打穿核心骨架
- `会压到现有骨架`
  - 功能能做，但会直接撞上当前任务系统、状态边界或收发主链问题
- `高概率逼出重写判断`
  - 一旦要做稳、做长期、做可联调，这项能力很难建立在当前骨架上继续叠补丁

## 3. 功能清单对比

### 3.1 子系统身份信息、状态查询、心跳

对应功能：

- 子系统身份信息
- 状态查询响应
- 心跳
- 最小设备信息

判断：

- `可包裹承接`

原因：

- 这组功能更像对外接入层能力。
- 需要统一的“我是谁、我现在什么状态、我是否在线”表达，但不必立刻改写接收主链和任务执行骨架。
- 真正会受影响的是状态口径统一，而不是底层运行机制本身。

当前主要压力点：

- 当前系统缺少面向“外部任务协同”的统一身份与运行态表达。
- 现有状态更多服务于本地工作台，不是服务于外部中心视角。

### 3.2 接收测试任务、解析用例与参数、任务上下文管理

对应功能：

- 接收测试任务
- 从任务中解析出测试用例和参数
- 保存当前任务上下文

判断：

- `会压到现有骨架`

原因：

- 当前系统有“发送任务”概念，但它是发送页工作台内的任务，不是“中心下发测试任务”的统一执行上下文。
- 现在缺的不是一个 JSON 解析器，而是一套外部任务上下文归口。
- 一旦任务来自中心，任务 ID、sessionId、用例列表、参数、当前执行指针、停止语义、结果汇总都需要统一承载。

当前主要压力点：

- 任务系统边界混合化
- 主状态没有围绕“外部测试任务上下文”统一设计
- 页面工作台仍深度参与任务生命周期

### 3.3 启动任务、停止任务、执行状态维护

对应功能：

- 任务启动
- 任务停止
- 执行状态维护

判断：

- `高概率逼出重写判断`

原因：

- 功能清单里的任务控制语义是“中心任务执行器”的控制语义。
- 当前扫描结论里，任务系统的 `pause / resume / stop / complete` 已经存在语义混叠，且任务生命周期依赖页面和对话框。
- 这意味着：如果后面真的要让中心稳定地下发、启动、停止、查询任务，现有任务系统不是不能复用，而是复用成本极高，且容易把旧混乱直接暴露给新链路。

当前主要压力点：

- 任务生命周期语义不完整
- 页面与任务运行控制边界混合
- 定时器职责双轨并存

### 3.4 每个用例完成后上报结果

对应功能：

- 用例执行结果上报
- 基础成功/失败/异常响应
- 异常结果归一化

判断：

- `会压到现有骨架`

原因：

- 用例级上报要求系统内部先有稳定的“用例完成”语义。
- 当前系统更接近“发送动作 / 接收副作用 / 局部任务执行”的混合形态，还没有天然的“标准测试用例结束事件”。
- 因此这项能力不是简单发个接口，而是要求内部先能稳定地界定：谁是用例、何时完成、失败怎么算、停止怎么算。

当前主要压力点：

- 任务控制语义不完整
- 状态归口与结果归口尚未统一
- 领域动作与通用执行状态尚未分层

### 3.5 任务级结果汇总、JSON 报告生成、报告交付

对应功能：

- 任务级结果汇总
- JSON 报告生成
- 报告交付
- 运行记录

判断：

- `可包裹承接` 到 `会压到现有骨架` 之间，取决于前面是否先把任务上下文立住

原因：

- 如果前面没有统一任务上下文，这三项都会变成“从各种局部状态里回捞结果拼 JSON”。
- 如果前面已有稳定任务上下文与用例结果对象，这三项反而比较适合做成外层交付能力。

当前主要压力点：

- 当前系统缺少“对外任务交付视角”的统一结果对象
- 运行记录目前更偏局部行为记录，未必能直接支撑对外报告链

### 3.6 结果字段整理

对应功能：

- 用例结果字段整理
- 任务报告字段整理

判断：

- `可包裹承接`

原因：

- 这项更多是模型设计和对接契约整理问题。
- 它会依赖前面的架构决策，但本身不直接决定是否重写核心运行骨架。

当前主要压力点：

- 结果口径还没定
- 当前系统内部很多结果是“本地运行语义”，不是“中心协同语义”

## 4. 目前最值得警惕的三个撞点

### 4.1 “中心测试任务”与“现有发送任务”不是同一个概念

这是当前最关键的错位点。

现有系统里的任务，更多是发送页工作台中的发送执行任务。  
功能清单里的任务，是“中心下发的测试任务执行上下文”。

这两个概念如果不先拆开，后续很容易误以为“已有任务系统，直接接一下接口就行”。  
实际更可能出现的情况是：旧任务系统只能复用局部执行能力，无法直接充当新的中心任务骨架。

### 4.2 当前系统缺少面向对外协同的统一状态视图

功能清单要求的很多能力，本质上都依赖一个统一状态视图：

- 当前是否在线
- 当前是否空闲
- 当前任务是谁
- 当前执行到哪个用例
- 当前是否异常
- 当前结果如何

而当前系统的状态更多是围绕：

- 本地工作台
- 接收配置与显示
- 发送实例与发送任务
- SCOE 特例状态

也就是说，后面对接真正要的不是“更多状态”，而是“统一对外状态面”。

### 4.3 如果现在就做对接实现，很容易把旧混乱冻结成新接口契约

这是为什么本文不建议现在就拍板“直接接上去”的原因。

如果在 `03-architecture` 关键决策之前直接开做，最可能发生的是：

- 外部中心任务直接映射到现有发送任务
- 页面和工作台继续持有运行控制语义
- 共享状态回读继续作为结果生成依据
- 报告和上报靠拼接局部状态临时凑出来

这样短期可能能跑，但会把当前系统最不稳定的部分变成对外契约，后面再调整代价会更高。

## 5. 对“什么时候判断要不要重写”的建议

### 5.1 现在就能做的判断

现在已经可以先下一个阶段判断：

- 后面对接不会只是补几个接口。
- 它至少会压到任务上下文、任务控制语义、状态统一口径和结果交付链。
- 因此它一定是架构级问题，不是单纯功能补丁。

### 5.2 现在还不该下的判断

现在还不适合直接下这些结论：

- 一定全量重写
- 一定不重写
- 一定只重写任务系统
- 一定可以原地演进

因为这些判断取决于 `03-architecture` 至少先回答：

1. 桌面能力怎么统一收口
2. 主状态怎么分层
3. 中心任务上下文怎么落位
4. 现有发送任务能力在新骨架里是“核心骨架”还是“可复用执行器”
5. SCOE 与通用执行链怎么共存

### 5.3 最稳的时机

最稳的做法不是等 `03` 全写完，而是：

1. 现在先完成这份“功能清单 vs 当前问题”的对比
2. 接着在 `03` 里把关键决策先拍到第一版
3. 然后立刻做第二轮对比：
   - 功能清单 vs 目标架构
4. 第二轮对比之后，再判断：
   - 不重写
   - 局部重写
   - 以中心任务执行骨架为核心重写

## 6. 当前结论

- [后面对接所需功能清单.md](D:/code/frontend/dongfanghong/后面对接所需功能清单.md) 对当前系统施加的最大压力，不在接口数量，而在“任务执行闭环”。
- 能较平滑承接的部分主要是：
  - 身份信息
  - 状态查询
  - 心跳
  - 结果字段整理
- 会直接压到当前运行骨架的部分主要是：
  - 外部测试任务上下文
  - 启动/停止控制语义
  - 用例完成语义
  - 任务级结果汇总与交付链
- 其中最可能逼出重写判断的核心点，是：
  - “中心测试任务”与“现有发送任务”并不是同一层概念
  - 当前系统缺少统一的对外任务执行状态面
  - 若现在直接接实现，极容易把旧混乱冻结成新接口契约
- 因此，关于“要不要重写”的最稳时机是：
  - 不等 `03` 全部写完
  - 但至少等 `03` 的第一轮关键架构决策完成之后，再做第二轮判断
