# SCOE 的架构位置

> 状态：首轮正文
> 阶段：03-architecture
> 作用：在继承运行主状态、收发主链和任务系统边界的前提下，回答 SCOE 在未来总架构中的层级定位、接入方式、可复用范围、必须保留的领域语义，以及哪些点必须被显式声明为边界例外。

## 第一部分：本文要解决的 SCOE 架构问题边界

### 1.1 当前事实

- `00-architecture-pre-red-lines.md` 已明确：
  - 领域特例不得无声明侵入通用链路。
  - 下游能力域不得通过共享状态回读替代明确边界输入。
  - 通用发送链和通用接收链不得长期承载越来越多的领域特例判断。
- `04-运行主状态与状态边界.md` 已明确：
  - 运行骨架围绕 `子系统运行主体 / 中心任务上下文 / 用例执行上下文 / 生命周期控制状态 / 结果归口状态` 组织。
  - SCOE 不得绕开主状态归口原则。
- `05-接收主链与发送主链组织方式.md` 已明确：
  - 领域特例允许进入主链，但必须通过显式领域入口或显式边界例外进入。
  - 固定来源、固定目标、领域完成条件、领域记录回写，不属于通用主链规则。
- `06-任务系统归口方式.md` 已明确：
  - 任务系统是独立能力域。
  - SCOE 如果要复用任务系统，只能显式接入。
  - SCOE 不得直接改写中心任务上下文、用例执行上下文、生命周期控制状态和结果归口状态。
- `01-scan/05-scoe-current-state.md` 已证实：
  - SCOE 当前同时侵入接收入口、统一发送、任务生命周期和测试工具记录。
  - 它既不是普通下游消费者，也不是边界干净的独立子系统。

### 1.2 判断 / 倾向

本文只回答以下问题：

1. SCOE 在未来总架构中应被视为一级能力域、二级领域模块，还是通用链路中的显式例外模块。
2. SCOE 与接收主链、发送主链、任务系统、统一运行主状态之间如何连接。
3. SCOE 哪些动作可以翻译为通用输入，哪些必须保留为领域特例。
4. SCOE 的领域完成条件、领域反馈语义、测试工具记录和工作台视图，应落在哪一层边界之外。
5. 哪些点必须被显式声明为边界例外，不能再隐式埋在通用主链内部。

本文不回答以下问题：

- 协议全集。
- 指令参数字段细节。
- API、schema、class、store 的终稿命名。
- 代码迁移顺序。
- 具体测试方案和实现步骤。

因此，本文不是 SCOE 重构方案，也不是 `04-specs` 的前置字段稿，而是先拍板一件更基础的事：

- SCOE 在未来总架构里到底站哪一层。
- 它靠什么边界进入统一运行骨架。
- 它有哪些语义必须被保留在领域边界内。

### 1.3 待决问题

- SCOE 后续是否需要在 `04-specs` 中单列一个“领域接入对象组”，还是分别并入接收入口、发送请求、任务请求和结果输出对象中，本文先不落对象拆分终稿。
- SCOE 内部是否还需要独立的领域执行上下文命名，本文先只立边界关系，不提前拍板对象名称。

## 第二部分：SCOE 的架构定位判断

### 2.1 当前事实

- SCOE 当前不是单纯页面工作台，它已有独立配置、独立状态、独立命令执行链、独立测试工具记录和独立网络连接策略。
- SCOE 当前也不是纯外围工具。它在接收链上拥有 `scoe-tcp-server` 前置分流，在发送链上复用统一发送并侵入领域目标特判，在任务侧还能直接创建并启动标准发送任务。
- SCOE 当前还不是独立运行骨架。它的完成条件依赖统一接收缓存，它的发送依赖统一发送链，它的长生命周期动作会借入任务系统。
- `04`、`05`、`06` 三篇已经把统一运行骨架、收发主链和任务系统的主边界立住，SCOE 现在真正待决的是“如何接入”，而不是“要不要存在”。

### 2.2 判断 / 倾向

#### 2.2.1 SCOE 不是谁

SCOE 不是：

- 通用接收主链本身。
- 通用发送主链本身。
- 任务系统本身。
- 统一运行主状态本身。
- 单纯的外围测试工具 UI。
- 现有“发送任务 + 接收特判 + 测试记录”的拼接体在未来架构中的延续形态。

这组否定性判断的意义是：

- SCOE 不能继续作为通用主链内部的一组长期特判存在。
- SCOE 也不能被草率排除成一个只消费结果、不参与运行事实的外围工具。

#### 2.2.2 三种定位里的当前推荐结论

当前更适合的定位是：

- SCOE 应被视为二级领域模块。
- 它通过显式领域入口、显式发送请求、显式任务接入与统一运行骨架发生关系。
- 它内部允许保留少量必须声明的边界例外，但这些例外不改变它作为领域模块的总体层级。

更具体地说：

- 一级能力域是接收主链、发送主链、任务系统、统一运行主状态这类为多个领域提供底层运行语义的骨架级能力。
- SCOE 更像建立在这些骨架能力之上的领域模块。它有自己的领域目标、领域完成条件、领域反馈语义和测试工具视图，但它不拥有总骨架定义权。

#### 2.2.3 为什么不应把 SCOE 定位为一级能力域

把 SCOE 直接定位成一级能力域，不合适的原因有四个：

1. SCOE 不提供普适运行骨架。
   - 它的指令识别、固定来源、固定目标、反馈语义和完成条件都依赖 SCOE 领域知识。
   - 这些规则不能抽象为其他领域可复用的一般能力。

2. SCOE 的关键运行仍建立在既有一级能力域之上。
   - 接收入口来自接收主链。
   - 发送落地来自发送主链。
   - 长生命周期动作若需调度、等待、停止和结果归口，仍依赖任务系统和统一运行主状态。

3. 如果把 SCOE 升格成一级能力域，容易把领域成功语义误写成系统成功语义。
   - `04` 已明确统一运行主状态的骨架语言。
   - SCOE 只能消费这套语言，不能重新定义它。

4. 这会弱化“任务系统是独立能力域”的既有结论。
   - 一旦把 SCOE 也写成一级能力域，很容易又回退到“每个复杂领域都自己带一套任务和生命周期解释”的老问题。

#### 2.2.4 为什么也不应把 SCOE 仅定位为通用链路中的显式例外模块

把 SCOE 只写成一组显式例外模块，也不合适，原因有四个：

1. SCOE 已经拥有独立领域对象和独立工作流。
   - 它不只是某个入口上的 if/else，也不只是某个目标 ID 特判。

2. SCOE 需要承载自己的领域状态和领域结果语义。
   - 例如领域完成条件、领域反馈、领域记录、测试工具视图。
   - 这些内容需要稳定边界，而不是散落在各处例外判断里。

3. 如果只把它写成例外模块，会默认通用主链长期承载其主工作流。
   - 这会直接违反 `AR-08` 和 `AR-09`。

4. 这会让后续 `04-specs` 失真。
   - 字段级设计会退化成“在哪里再补一个特判字段”，而不是“哪个边界输入属于 SCOE 域”。

#### 2.2.5 SCOE 与通用链路、任务系统、统一运行主状态的正式关系

SCOE 与通用链路的关系应理解为：

- 它是通用接收主链和通用发送主链的领域消费者与领域翻译层。
- 它可以消费通用链路输出，也可以把领域动作翻译成通用链路可消费的输入。
- 但它不是通用链路内部的规则承载层。

SCOE 与任务系统的关系应理解为：

- 它不是任务系统本身。
- 当某类 SCOE 动作需要统一生命周期、调度、等待和结果归口时，它可以作为任务系统的上游生产者或领域翻译层显式接入。
- 它也可以消费任务系统的生命周期结果与结果归口输出。

SCOE 与统一运行主状态的关系应理解为：

- 它不能直接定义 `子系统运行主体 / 中心任务上下文 / 用例执行上下文 / 生命周期控制状态 / 结果归口状态`。
- 它只能通过显式边界输入影响这些状态，或消费这些状态的正式输出。
- 它自己的领域状态必须建立在统一运行骨架输出之上，而不是反向塑造统一运行骨架。

### 2.3 待决问题

- SCOE 在更高层的总模块图中，最终挂在“领域模块组”还是“专项集成域”之下，本文先不处理命名层级，只拍板它不是一级骨架能力。
- 后续是否要把“边界例外”作为 SCOE 文档中的独立附录列清单，本文先给规则，不先列实现载体。

## 第三部分：SCOE 与接收主链的关系

### 3.1 当前事实

- 当前 SCOE 通过 `scoe-tcp-server` 在统一接收入口中前置短路，这已经被 `01-scan/03` 和 `01-scan/05` 共同证实。
- 当前 SCOE 的识别、参数提取、完成条件等待都带有明显领域知识，且完成条件等待还会回读统一接收缓存。
- `05-接收主链与发送主链组织方式.md` 已经明确：领域特例可以进入接收主链，但必须通过显式领域入口进入；接收主链不能继续长期承载来源特判和领域状态回读。

### 3.2 判断 / 倾向

#### 3.2.1 SCOE 如何从接收主链进入

未来 SCOE 不应再以“通用接收入口中的隐式前置短路”进入，而应改为：

- 接收主链先完成输入承接与基础归一。
- 当命中已声明的 SCOE 领域入口条件时，输出一个明确的“SCOE 领域接管输入”。
- 之后由 SCOE 领域模块接管其后的识别、执行、等待、反馈和记录语义。

这里的关键不是“晚一点特判”，而是“把特判从通用主链内部逻辑，提升为被声明的边界入口”。

#### 3.2.2 什么属于显式领域入口

显式领域入口至少应同时满足两点：

1. 入口条件被架构上明确声明。
   - 例如固定来源类型、固定来源标识、命中 SCOE 协议识别条件。

2. 入口之后的领域接管边界是清晰可见的。
   - 通用接收主链负责把命中结果输出为领域接管输入。
   - SCOE 负责解释此后才成立的领域知识。

因此，未来可被接受的入口语义不是：

- “如果这里刚好是 `scoe-tcp-server`，就在通用主链里直接展开 SCOE 流程”。

而应是：

- “接收主链识别到来源与基础特征命中 SCOE 入口，输出 SCOE 领域输入，由 SCOE 模块接管”。

#### 3.2.3 哪些接收来源或条件不能继续留在通用接收链里

一旦某项判断依赖以下任一类 SCOE 领域知识，就不能继续留在通用接收链里：

- 固定来源 ID 与 SCOE 协议识别的组合判断。
- 固定功能码、固定信源 / 信宿、型号 ID、卫星 ID 等领域识别条件。
- 当前是否已加载卫星帧这类领域状态。
- 指令参数解析规则与校验规则。
- 领域完成条件、领域成功条件、领域反馈时机。
- 任何必须读取 SCOE 自身领域状态才能判断是否继续处理的逻辑。

通用接收链只应负责：

- 输入承接。
- 基础归一。
- 领域入口命中结果输出。

它不应继续负责：

- SCOE 指令识别后的完整执行。
- SCOE 领域完成判断。
- SCOE 测试工具记录写入。

#### 3.2.4 SCOE 可以消费接收主链的什么输出

SCOE 可以消费接收主链的正式输出，应限于：

- 标准化后的入站事实。
- 来源标识与接收时序信息。
- 接收解析成功、失败或未识别等基础结果。
- 已命中 SCOE 领域入口的接管输入。

SCOE 不应再直接消费：

- `receiveFramesStore.groups` 这样的共享展示 / 配置状态面。
- `receiveFramesStore.mappings` 这样的接收配置共享面。
- `allReceiveFrameData` 这类全局接收缓存，作为自己运行语义的事实源。

如果 SCOE 的后续完成判断需要依赖接收结果，也应改成：

- 消费接收主链向其输出的明确结果输入。

而不是：

- 自己回到统一接收 Store 里扫描共享缓存。

#### 3.2.5 SCOE 不能再通过什么共享状态回读定义自己的运行语义

以下方式不应再作为 SCOE 运行语义的长期前提：

- 回读 `receiveFramesStore.groups` 定义链路状态或完成条件。
- 回读 `receiveFramesStore.mappings` 补齐自己命令识别或条件判断语义。
- 回读 `receiveFramesStore.allReceiveFrameData` 直接充当领域完成事实源。
- 回读显示态、历史态或最近值面板来决定“命令是否完成”。

一旦 SCOE 还需要这些数据，它应通过显式边界输入获得，而不是通过共享状态面自取。

### 3.3 待决问题

- “SCOE 领域接管输入”在 `04-specs` 中是否需要再拆成“领域入口命中输入”和“领域推进输入”两类对象，本文先不落对象终稿。
- 某些接收结果如果同时能被通用链路和 SCOE 域消费，后续是否采用一份结果多订阅，还是拆成两类输出对象，留待 `04-specs` 继续收紧。

## 第四部分：SCOE 与发送主链的关系

### 4.1 当前事实

- 当前 SCOE 的主动发帧、指令触发发送、成功反馈发送和测试工具手工发送，最终都复用了统一发送。
- 当前统一发送链内部已存在对 `network:scoe-udp:scoe-udp-remote` 的领域特例处理，这说明通用发送链已经被 SCOE 反向侵入。
- `05-接收主链与发送主链组织方式.md` 已明确：领域动作应先翻译为显式发送请求再进入通用发送链；如果不能翻译，就必须被声明为边界例外。

### 4.2 判断 / 倾向

#### 4.2.1 哪些 SCOE 动作可以先翻译成显式发送请求，再进入通用发送链

以下 SCOE 动作原则上都可以先翻译成显式发送请求：

- 主动发送某个已确定的领域帧实例。
- 指令触发后的单次领域发送动作。
- 领域成功后需要下发的反馈帧发送动作。
- 测试工具中用户显式触发的手工发送动作。

这些动作虽然属于 SCOE 域，但它们在进入发送主链前，都可以先被翻译成：

- 为什么发。
- 由谁发。
- 要发往哪个明确目标引用。
- 使用哪个明确载荷或实例引用。
- 返回结果应归给哪个领域上下文或运行上下文。

换句话说，领域动作可以保留在 SCOE 域，真正进入发送主链的应始终是显式发送请求。

#### 4.2.2 哪些动作如果不能翻译，就必须被标注为边界例外

以下内容如果仍然需要出现在发送侧，就不能伪装成通用发送规则，而必须被声明为边界例外：

- 固定领域目标才成立的发送判断。
- 发送前必须依赖领域加载态、领域确认态才能成立的规则。
- 发送后还要由领域成功条件决定是否算完成的规则。
- 发送结果需要反写测试工具记录、领域记录或领域反馈语义的规则。
- 发送动作本身其实是在启动一个领域专属长流程，而不是一次显式发送请求。

特别是：

- 如果某个动作本质上不是“一次发送”，而是“围绕发送展开的一段可调度、可等待、可停止的领域执行过程”，它更接近任务系统接入点，而不是发送主链内部逻辑。

#### 4.2.3 固定目标、固定反馈、领域成功条件、领域工具记录为什么不能继续沉积在通用发送链内部

这些规则不能继续沉积在通用发送链内部，原因有四个：

1. 它们依赖 SCOE 领域知识，不能为其他领域复用。
2. 它们会把通用发送链从“发送请求执行层”扭曲成“领域语义解释层”。
3. 它们会让发送结果不再是标准结果，而变成夹带领域副作用的混合结果。
4. 这会直接违反 `05` 已立住的“固定目标、固定反馈、领域记录回写不属于通用主链规则”。

因此：

- 固定目标解析应在 SCOE 领域边界内完成，再把结果作为明确目标引用送入发送主链。
- 领域成功条件应留在 SCOE 域或任务系统消费层判断，不应塞回发送链。
- 测试工具记录应由 SCOE 领域记录层消费标准发送结果，不应写进通用发送层。

#### 4.2.4 SCOE 和发送主链之间的正式输入 / 输出关系

SCOE 到发送主链的正式输入关系应描述为：

- SCOE 输出显式发送请求给发送主链。
- 如果某些动作无法表达为显式发送请求，则输出的是显式边界例外声明，而不是在发送主链里补特判。

发送主链回给 SCOE 的正式输出关系应描述为：

- 返回标准发送执行结果。
- 返回拒绝、失败、取消等标准结果口径。
- 返回可被任务系统或 SCOE 域继续消费的标准结果输入。

发送主链不应再直接回给 SCOE：

- 已经带着领域记录副作用的结果。
- 已经替 SCOE 判完成功条件的结果。
- 已经替 SCOE 回写测试工具的结果。

### 4.3 待决问题

- 成功反馈帧在 `04-specs` 中是否定义为“领域动作翻译后的一类普通发送请求”，还是单列为“领域反馈请求”，本文先不拍字段层终稿。
- 某些 SCOE 周期性主动发送动作，后续究竟直接作为领域发送请求进入发送主链，还是先进入任务系统调度，再由任务系统发起发送请求，留待 `04-specs` 结合用例语义继续拍板。

## 第五部分：SCOE 与任务系统、统一运行主状态的关系

### 5.1 当前事实

- `06-任务系统归口方式.md` 已明确：任务系统是独立能力域，负责推进中心任务上下文、用例执行上下文、生命周期控制状态和与任务执行相关的结果归口状态。
- 当前 SCOE 中的 `READ_FILE_AND_SEND` 已证实会直接创建并启动标准发送任务，说明 SCOE 与任务系统不是弱关系。
- `04` 已明确统一运行主状态的五类骨架对象，SCOE 不得绕开这套归口语言。

### 5.2 判断 / 倾向

#### 5.2.1 SCOE 是否复用任务系统，以及复用到哪一层

SCOE 可以复用任务系统，但只能复用到“显式接入的任务边界层”，而不是复用任务系统的定义权。

更具体地说，SCOE 可复用的内容是：

- 任务请求入口。
- 生命周期控制能力。
- 调度、等待、停止、完成和结果归口能力。
- 任务系统对发送主链的显式请求能力。

SCOE 不可复用的内容是：

- 直接改写中心任务上下文。
- 直接改写用例执行上下文。
- 直接改写生命周期控制状态。
- 直接把领域状态当作任务系统正式状态。

因此，SCOE 与任务系统的关系不是“内嵌在任务系统里”，而是“需要时显式接入任务系统”。

#### 5.2.2 SCOE 是任务系统的上游生产者、下游消费者，还是领域翻译层

当前倾向是：

- SCOE 首先是任务系统的上游生产者与领域翻译层。
- 当某类 SCOE 动作需要进入统一生命周期时，SCOE 负责把它翻译成显式任务请求或等价运行输入。
- 在任务执行结束后，SCOE 也可以作为任务系统结果的下游消费者，继续形成领域反馈、领域记录和工具视图。

因此，SCOE 与任务系统的关系不是单向的“复用一下”，而是：

- 上游把领域动作翻译成任务输入。
- 下游把任务结果翻译回领域结果。

但这仍然不等于：

- SCOE 拥有任务系统的正式状态面。

#### 5.2.3 哪些 SCOE 动作可以进入中心任务上下文 / 用例执行上下文

只有满足以下条件的 SCOE 动作，才应进入中心任务上下文或用例执行上下文：

- 动作需要统一调度、等待、停止、恢复或完成语义。
- 动作不是一次性发送，而是具有多步推进过程。
- 动作需要接入统一结果归口，而不是只在领域内留下临时记录。
- 动作需要被统一运行主状态看见，并对外表达为正式运行事实。

典型地说：

- 类似“读取文件后按一定节拍组织多次发送”的动作，更适合翻译为任务系统输入。
- 需要等待统一生命周期控制和统一结果归口的 SCOE 长流程，也应进入任务系统边界。

反过来，以下动作不应直接进入中心任务上下文或用例执行上下文：

- 纯领域局部状态更新。
- 纯测试工具手工动作。
- 只服务于领域视图的记录更新。
- 不需要统一调度与生命周期管理的一次性局部动作。

#### 5.2.4 哪些领域状态绝不能直接抬升成任务系统正式状态

以下 SCOE 领域状态不应直接抬升成任务系统正式状态：

- 当前已加载卫星 ID。
- 健康检查结论、链路自检结论。
- 某条指令最近一次是否成功。
- 领域成功反馈是否已经发出。
- 测试工具收发记录、高亮配置、停止开关。
- 任何只对 SCOE 领域工作台有意义的中间状态。

这些状态可以：

- 作为 SCOE 领域内部状态。
- 作为 SCOE 消费任务系统结果后的派生视图。

但它们不能：

- 直接占据中心任务上下文。
- 直接替代用例执行上下文。
- 直接冒充生命周期控制状态或结果归口状态。

#### 5.2.5 SCOE 与统一运行主状态的正式关系

SCOE 与统一运行主状态的正式关系应收敛为：

- SCOE 可以输出显式运行输入，由主状态归口层或任务系统决定是否推进运行事实。
- SCOE 可以消费统一运行主状态输出，用于更新自己的领域视图和领域执行判断。
- SCOE 不能直接宣布“系统当前正在执行什么任务”“当前用例是否已完成”“当前结果是否已归口”。

这意味着：

- SCOE 的领域状态必须建立在主状态输出之上。
- 统一运行主状态不是 SCOE 的派生视图。
- SCOE 也不是统一运行主状态的一种领域别名。

### 5.3 待决问题

- 后续是否需要在 `04-specs` 中为 SCOE 单列“任务接入请求”与“任务结果适配输出”两组对象，本文先不落对象名称。
- 某些 SCOE 周期性动作究竟属于任务系统正式调度对象，还是保留为 SCOE 域内的局部运行对象，留待 `04-specs` 结合生命周期粒度继续收紧。

## 第六部分：SCOE 应保留的领域边界与必须声明的例外

### 6.1 当前事实

- 当前 SCOE 的成功语义、反馈发送、记录回写和测试工具视图仍交错在接收链、发送链和共享状态回读之中。
- `04` 已明确：记录态、展示态、缓存态不得反向塑造运行主状态语义。
- `05` 已明确：固定来源、固定目标、领域完成条件、领域记录回写不属于通用主链规则。
- `06` 已明确：SCOE 不得直接改写统一任务事实。

### 6.2 判断 / 倾向

#### 6.2.1 SCOE 自身应保留哪些领域边界

以下内容应明确保留在 SCOE 领域边界内：

- 领域指令识别规则。
- 领域参数解析规则与校验规则。
- 领域完成条件与成功条件。
- 领域反馈语义与反馈帧构造规则。
- 领域记录、测试工具记录和测试工具视图。
- 领域工作台所需的配置与局部运行状态。

这些内容之所以必须留在 SCOE 域内，是因为：

- 它们都依赖 SCOE 领域知识。
- 它们不具备通用复用性。
- 它们一旦外溢到通用主链，就会把通用规则重新拖回领域特判。

#### 6.2.2 这些领域边界应落在哪一层边界之外

SCOE 的领域完成条件、领域反馈语义、领域记录和测试工具视图，应明确落在以下边界之外：

- 统一运行主状态之外。
- 通用接收主链之外。
- 通用发送主链之外。
- 任务系统正式状态面之外。

更准确地说，它们应位于：

- SCOE 领域内部逻辑层。
- SCOE 领域结果适配与领域观察层。

这两层可以消费统一骨架输出，但不应反向定义统一骨架。

#### 6.2.3 它们如何消费统一运行骨架输出，而不是反向定义统一运行骨架

正确方向应是：

1. 接收主链输出标准接收事实或领域接管输入。
2. 发送主链输出标准发送结果。
3. 任务系统输出生命周期结果与结果归口结果。
4. 统一运行主状态输出正式运行事实。
5. SCOE 在这些正式输出之上形成自己的领域完成判断、领域反馈和领域记录。

错误方向则是：

- 先由 SCOE 的领域记录或测试工具结果反推任务是否完成。
- 先由 SCOE 的局部状态反推主状态是否已进入某生命周期。
- 先由 SCOE 的反馈发送结果反推发送链或任务系统的正式结果。

#### 6.2.4 什么应保留为 SCOE 领域内部逻辑，什么应抽成显式边界

应保留为 SCOE 领域内部逻辑的内容：

- 领域识别、领域参数、领域完成条件。
- 领域反馈和领域记录语义。
- 领域工作台和测试工具视图。
- 只对 SCOE 自身有意义的局部状态。

应抽成显式边界的内容：

- SCOE 从接收主链接入的领域入口。
- SCOE 向发送主链发起的显式发送请求。
- SCOE 向任务系统发起的显式任务请求或控制请求。
- SCOE 消费任务系统和统一运行主状态结果的适配输出。
- 无法翻译为通用输入 / 输出时必须声明的边界例外。

#### 6.2.5 哪些地方必须显式声明为边界例外

以下点必须显式声明为边界例外，不能再隐式埋在代码路径里：

1. 固定来源进入接收主链时的 SCOE 领域入口声明。
2. 固定目标进入发送侧时的领域目标例外声明。
3. 领域完成条件不能被通用主链表达时的领域完成例外声明。
4. 领域反馈发送与领域记录回写之间的边界例外声明。
5. 某些 SCOE 长流程若既不能完全表达为任务请求、也不能表达为一次发送请求时的领域执行例外声明。

这里的重点不是“例外越少越好”，而是：

- 例外必须被看见。
- 例外必须被命名。
- 例外不能假装自己是通用链路规则。

### 6.3 待决问题

- “领域内部逻辑层”和“领域结果适配与观察层”在 `04-specs` 中是否需要继续拆成更细的对象层次，本文先不展开。
- 某些当前依赖接收结果推进的领域完成条件，后续究竟沉淀为哪一类正式输入对象，留待 `04-specs` 继续细化。

## 第七部分：当前结论 / 待决问题 / 下一步建议

### 7.1 当前结论

经过 `03-architecture` 的 `03` 到 `07` 这几篇，当前阶段已经拍板了以下内容：

1. 桌面能力访问必须统一收口，不能再由领域代码散落直连。
2. 统一运行骨架围绕 `子系统运行主体 / 中心任务上下文 / 用例执行上下文 / 生命周期控制状态 / 结果归口状态` 组织。
3. 接收主链和发送主链都只处理显式输入和显式输出，通用链路与领域特例的边界已经立住。
4. 任务系统是独立能力域，不能再等同于发送工作台或本地发送任务。
5. SCOE 在未来总架构中更适合被视为二级领域模块，而不是一级能力域，也不是通用主链内部的一组长期特判。
6. SCOE 进入统一运行骨架的方式已经有了原则结论：
   - 从接收主链通过显式领域入口进入。
   - 从发送侧通过显式发送请求进入。
   - 需要统一生命周期时，通过显式任务接入进入任务系统。
   - 自身领域完成条件、反馈语义和测试工具记录保留在领域边界内。
7. 需要例外的地方已经有了规则结论：
   - 例外允许存在。
   - 但必须显式声明，不能继续伪装成通用主链规则。

### 7.2 待决问题

留给 `04-specs` 的问题主要包括：

- 接收主链到 SCOE 的领域入口对象如何定义。
- SCOE 到发送主链的显式发送请求对象如何定义。
- SCOE 到任务系统的显式任务请求或控制请求如何定义。
- 任务系统与 SCOE 之间的结果适配对象如何定义。
- 领域完成条件、领域反馈结果、领域记录输入在对象层如何表达。
- 哪些边界例外需要形成正式清单，以及它们各自的输入 / 输出口径。

这些问题都属于：

- 字段级设计。
- capability 级设计。
- 对象口径设计。

它们不应反过来改写本文已经拍板的层级和边界。

### 7.3 下一步建议

当前已经可以进入“功能清单 vs 目标架构”的第二轮对比。

原因是：

- `03` 到 `07` 已经把桌面能力、主状态、收发主链、任务系统和 SCOE 的架构位置依次拍板。
- 现在再对照功能清单时，已经有稳定的边界语言可以判断“某项能力应该落在哪一层”，而不会再退化成现状代码映射。
- 如果不先把 SCOE 的位置拍板，`04-specs` 会失真为：
  - 把 SCOE 继续拆成接收特判字段、发送特判字段和任务复用字段。
  - 把领域完成条件误写成通用生命周期字段。
  - 把测试工具记录误抬成运行主状态或任务正式状态。

因此，当前阶段的第一轮闭合标准可以明确为：

- `03-architecture` 已经足以回答“未来总架构里谁是一级骨架能力、谁是领域模块、谁通过什么边界接入、哪些点必须保留为领域例外”。
- 在这个前提下，后续 `04-specs` 可以开始做对象和字段层收紧，但不得回退重争本文已立住的边界结论。
