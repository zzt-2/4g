# 接收主链与发送主链组织方式

> 状态：首轮正文
> 阶段：03-architecture
> 作用：在继承桌面能力统一收口方式与运行主状态语言的前提下，先立住接收主链、发送主链的层次边界、正式输入输出、通用链路与领域特例的进入规则，并给后续 `06-任务系统归口方式.md` 与 `07-SCOE 的架构位置.md` 提供必须继承的前提。

## 第一部分：本文要解决的主链问题边界

### 1.1 当前事实

- 当前项目已经先后立住：
  - 架构前红线
  - 决策顺序
  - 桌面能力统一收口方式的当前推荐方向
  - 运行主状态与状态边界
- 当前接收链不只是解析链，而是同时影响显示、历史沉淀、任务触发、SCOE 和统计更新的多分支主耦合轴。
- 当前发送链不只是一次发送动作，而是和发送任务、本地工作台、连接目标、统一发送、部分领域目标特例深度缠绕。
- 当前系统里最明显的失真来源，不是“链路不够多”，而是“共享状态被多个分支反复回读，并被反向当成主链编排事实源”。

### 1.2 判断 / 倾向

本文只回答六件事：

1. 接收主链由哪些层构成。
2. 发送主链由哪些层构成。
3. 两条主链的正式输入、正式输出分别是什么。
4. 哪些职责属于通用链路，哪些属于领域特例。
5. 领域特例允许以什么方式进入主链。
6. 两条主链与统一运行主状态之间是什么关系。

本文先明确三条否定性边界：

1. 接收主链不是统一运行主状态本身。
2. 发送主链不是中心任务上下文本身。
3. 接收缓存、显示态、历史记录、页面工作台状态，都不能再充当主链编排事实源。

因此，主链只能做两件事：

- 消费明确边界输入。
- 产出明确边界结果。

它不能再依赖“当前共享状态里似乎已经有这些信息”来补齐自己的语义。

当前共享状态回读造成的最大失真是：

- 主链的真实输入不再可见。
- 运行事实和展示事实被混写。
- 下游能力通过回读共享状态，反向决定上游主链该如何理解当前系统。
- 最终会把“接收链 = 任务系统的一部分”“发送链 = 中心任务主骨架本身”这两种错误印象结构化下来。

本文不回答的内容包括：

- `06-任务系统归口方式.md` 的终稿。
- `07-SCOE 的架构位置.md` 的终稿。
- 解析引擎内部实现。
- 命令对象、结果对象、状态对象的字段 schema。
- 任意 store、class、module 的命名终稿。

### 1.3 待决问题

- 接收主链与发送主链各自的正式结果对象，后续是否需要在 `04-specs` 中拆成多类结果口径，本文先不落字段级定义。
- 发送主链的正式起点是否要在后续继续区分“中心任务驱动发送”“手工工作台发送”“领域模块驱动发送”三类入口，本文先统一按“显式发送请求”处理。

## 第二部分：接收主链的层次与职责

### 2.1 当前事实

- 当前系统中的串口和网络入站已经统一收敛到同一接收入口。
- 当前接收链在解析之后没有自然结束，而是继续扩散到表达式、显示 / 历史、任务触发、SCOE、统计等多个方向。
- 当前接收链里的多方向扩散，大量依赖 `groups`、`mappings`、接收缓存等共享状态面回读完成。
- 当前 SCOE 已经在统一接收入口中以来源特判方式提前短路通用解析链。

### 2.2 判断 / 倾向

未来接收主链应分成四层。

#### 2.2.1 输入承接层

职责：

- 承接统一桌面能力收口层送入的明确输入。
- 把入站数据整理成可进入业务链的标准输入包。
- 处理仅属于本链路的排队、串行化、去重、来源标识完整性检查。

正式输入：

- 入站来源类型。
- 入站来源标识。
- 原始字节流或等价原始载荷。
- 接收时刻或等价顺序信息。

允许持有的局部状态：

- 本链路队列。
- 串行化锁。
- 当前批次处理上下文。

不允许持有的状态：

- 中心任务上下文。
- 生命周期控制状态。
- 结果归口状态。
- 任意展示态或历史态的事实定义权。

#### 2.2.2 解析 / 归一层

职责：

- 把原始输入转换为标准化的接收结果。
- 产出“已识别”“未识别”“解析失败”“领域入口命中”等明确口径。
- 把接收事实从来源细节中剥离出来，形成可被下游消费的统一结果。

正式输出至少应包含的语义口径：

- 标准接收结果。
- 标准接收失败结果。
- 领域入口命中结果。

允许持有的局部状态：

- 解析过程临时上下文。
- 仅为本次接收处理服务的局部缓存。

不允许做的事：

- 越级写入统一运行主状态。
- 直接改写页面显示态。
- 直接沉淀历史记录。
- 直接触发任务执行。

#### 2.2.3 领域分发层

职责：

- 只根据显式声明的领域入口条件，把结果分发给通用接收输出或领域模块。
- 负责判断“这是不是通用接收链继续处理的对象”，而不是在这里展开领域业务。

这里的关键不是“多分支副作用”，而是“边界分发”。

允许做的事：

- 把标准接收结果送往通用结果输出。
- 把领域入口命中结果送往对应领域模块。

不允许做的事：

- 直接触发显示、历史、任务、测试工具等多方向副作用。
- 在主链内部长期堆积来源特判、领域 if/else 和领域状态回读。

#### 2.2.4 结果输出层

职责：

- 把接收主链真正能对外宣告的结果，变成明确边界输出。

接收主链的正式输出应理解为以下几类之一：

1. 通用接收事实输出。
2. 统一运行主状态可消费的输入。
3. 记录 / 展示域可订阅的结果输入。
4. 领域模块接管输入。

这里刻意不再使用“接收链直接触发某模块”的语言，而改用“接收链输出什么，谁来消费”的语言。

### 2.3 接收主链与统一运行主状态的关系

接收主链与统一运行主状态的关系应收敛为：

- 接收主链不是主状态。
- 接收主链只能产出可被主状态归口层消费的输入。
- 是否改变 `子系统运行主体 / 中心任务上下文 / 用例执行上下文 / 生命周期控制状态 / 结果归口状态`，必须由主状态归口层决定，而不是由接收缓存或接收副作用链路反向宣布。

### 2.4 接收链哪些事情可以做，哪些不该再做

接收链可以做：

- 统一承接输入。
- 标准化解析和归一。
- 明确领域入口分发。
- 产出标准接收结果和主状态输入。

接收链不该再做：

- 直接把显示、历史、任务触发、测试工具记录混成同一条主链尾部。
- 依赖共享状态回读补齐任务语义。
- 让领域来源特判长期内嵌在通用接收主链内部。
- 让接收缓存、显示态、记录态回头定义“系统现在处于什么运行事实”。

### 2.5 待决问题

- “触发候选信号”最终是否作为接收主链的独立正式输出类型，还是被收敛进更上层的运行输入口径，留待 `06-任务系统归口方式.md` 继承本文边界后再收紧。
- 领域入口命中后，是全部改为“领域模块先接管”，还是保留少量“通用解析后再转领域”的路径，留待 `07-SCOE 的架构位置.md` 在不突破本文边界的前提下拍板。

## 第三部分：发送主链的层次与职责

### 3.1 当前事实

- 当前发送链已经不是单次发送动作，而是与发送页工作台、任务对象、统一发送器、连接目标和领域目标特例交织在一起。
- 当前任务系统中的发送任务对象，天然携带发送实例、目标、触发条件、时间调度和页面语义。
- 当前统一发送层已经出现领域目标特判，这说明“通用发送链”与“领域特例”还没有真正分开。
- `04-运行主状态与状态边界.md` 已经明确：中心任务上下文是一级运行对象，现有发送任务不能直接等同于中心任务主上下文。

### 3.2 判断 / 倾向

未来发送主链应分成四层。

#### 3.2.1 发送请求承接层

职责：

- 承接显式发送请求。
- 检查发送请求是否带着明确的边界输入进入主链。

发送主链的正式输入，应至少包含以下语义：

- 发送意图。
- 触发原因。
- 执行上下文引用。
- 目标引用。
- 载荷或实例引用。

这里的执行上下文引用，可能来自：

- 中心任务上下文。
- 用例执行上下文。
- 手工工作台动作。
- 领域模块的显式调用。

但无论来源是谁，都必须先变成统一的“显式发送请求”，才能进入发送主链。

#### 3.2.2 通用编排 / 归一层

职责：

- 把发送请求和只读的运行上下文拼成通用发送计划。
- 只做真正通用的判断，不承担领域判断。

属于通用发送链的判断包括：

- 当前是否存在合法的执行上下文引用。
- 当前生命周期控制状态是否允许发送。
- 目标引用是否可解析。
- 载荷或实例是否可被通用发送能力消费。
- 发送顺序、批次、局部重试、局部失败结果如何归一。

不属于通用发送链的判断包括：

- 某个领域目标是否因领域状态而允许发送。
- 某个领域成功条件是否已满足。
- 某个领域指令是否需要回读领域缓存才能决定发送。
- 某个领域动作是否应直接创建任务或进入领域测试工具记录。

#### 3.2.3 目标落地层

职责：

- 把已经完成通用编排的发送计划，交给统一桌面能力收口层或统一发送能力落地执行。
- 这里处理的是通用目标能力差异，而不是领域语义差异。

这一层允许处理的通用差异包括：

- 串口目标与网络目标的通用路由差异。
- 通用发送错误归一。
- 通用发送回执归一。

这一层不允许长期承载：

- 固定领域目标 ID 特判。
- 领域测试工具记录回写。
- 领域成功反馈帧拼装。
- 领域加载状态判断。

#### 3.2.4 执行结果输出层

职责：

- 产出发送主链真正能对外宣告的结果。

发送主链的正式输出应理解为以下几类之一：

1. 标准发送执行结果。
2. 统一运行主状态可消费的结果输入。
3. 结果归口状态可消费的结果输入。
4. 领域模块可继续处理的发送结果。

这里同样不再使用“发送链直接维护中心任务事实”的语言，而改用“发送链产出结果，由主状态和任务归口层消费”的语言。

### 3.3 发送主链与统一运行主状态的关系

发送主链与统一运行主状态的关系应收敛为：

- 发送主链不是中心任务上下文本身。
- 发送主链只能读取中心任务上下文、用例执行上下文和生命周期控制状态，不能占有它们的定义权。
- 发送主链可以持有本链路局部执行状态，但不能持有系统级运行语义。

发送主链允许持有的局部执行状态包括：

- 当前请求队列。
- 当前批次游标。
- 当前发送中状态。
- 局部失败信息。
- 局部重试计数。

发送主链不能持有的全局运行语义包括：

- 当前中心任务是否存在。
- 当前用例是否已完成。
- 当前子系统整体是否进入完成、停止、暂停、等待调度。
- 当前结果是否已经最终归口。

### 3.4 发送链哪些判断属于通用能力，哪些必须下沉到领域模块

属于通用能力的判断：

- 是否具备明确发送请求。
- 是否具备合法上下文引用。
- 生命周期控制是否允许进入发送。
- 目标是否可解析到通用发送能力。
- 发送结果如何统一归一为成功、失败、取消、拒绝等口径。

必须下沉到领域模块的判断：

- 领域固定目标、固定来源、固定反馈对象。
- 领域加载态、领域完成条件、领域确认语义。
- 领域测试工具记录。
- 领域特有的“发送之后还要再回读某共享状态才能算完成”的规则。
- 领域动作直接生成任务、复用任务生命周期或改写任务解释的规则。

### 3.5 待决问题

- 定时调度、条件触发、后台继续运行这些能力，后续究竟全部归任务系统，还是部分仍保留为发送主链外侧的运行编排能力，需要 `06-任务系统归口方式.md` 在继承本文前提后再拍板。
- 发送主链是否需要在后续再细分“通用目标适配”和“统一发送执行”两个层次，本文先不细拆实现级分层。

## 第四部分：通用链路与领域特例的边界规则

### 4.1 当前事实

- 当前接收主链里已经存在 `scoe-tcp-server` 这类来源特判与提前短路。
- 当前统一发送链里已经存在对领域目标的特判与领域测试记录回写。
- 当前部分任务触发与完成条件判断，仍然依赖共享状态回读，而不是依赖显式边界输入。

### 4.2 判断 / 倾向

本文先给出五条边界规则。

#### 规则 1：什么叫通用链路

通用链路是指：

- 不依赖某个特定领域的固定来源、固定目标、固定状态语义。
- 能被多个领域复用。
- 只消费通用边界输入，只产出通用边界结果。

换句话说，通用链路描述的是：

- 收到什么类型的输入。
- 如何把它变成标准结果。
- 如何把结果交给明确的下游边界。

它不描述某个领域自己的成功条件、加载态、测试工具记录和例外目标。

#### 规则 2：什么叫领域特例

领域特例是指：

- 判断是否成立，必须依赖某一领域自己的来源、目标、状态、协议语义或完成语义。
- 无法在不引入该领域知识的前提下，被其他领域复用。

典型领域特例包括：

- 某领域的固定来源 ID 或固定目标 ID。
- 某领域的完成条件等待。
- 某领域的成功反馈帧或领域记录写回。
- 某领域直接借用任务系统解释自己的执行语义。

#### 规则 3：领域特例允许进入主链，但必须以显式边界入口进入

允许进入，不等于允许散落在通用主链内部。

允许的进入方式只能是以下两种之一：

1. 在接收主链中，作为显式声明的领域入口条件，被领域模块接管。
2. 在发送主链中，先由领域模块把领域动作翻译成显式发送请求，再进入通用发送链；如果无法翻译，则必须声明为边界例外，而不是塞进通用发送层内部特判。

#### 规则 4：什么情况必须显式标注为边界例外

出现以下任一情况，就必须显式标注为边界例外：

- 需要绕过通用主链的正常起点。
- 需要在通用主链中提前短路。
- 需要读取领域状态才能判断是否继续主链。
- 需要在通用输出之外附带领域记录、副作用或领域反馈。
- 需要使用固定领域来源 / 目标 ID 才能成立。

#### 规则 5：哪类特判绝不能继续沉积在通用主链内部

以下特判绝不能继续沉积在通用主链内部：

- 领域固定来源 / 固定目标判断。
- 领域成功条件与完成条件判断。
- 领域测试工具记录与领域展示回写。
- 领域动作直接创建任务、接管生命周期、重写结果解释的逻辑。
- 依赖共享状态回读才能成立的领域规则。

### 4.3 待决问题

- 领域例外最终采用什么声明载体、什么注册形式、由哪一篇文档负责列清单，本文不提前拍实现形式。
- 某些“看起来是协议差异、实际上已经是领域语义”的边界案例，后续需要在 `07-SCOE 的架构位置.md` 中按本文规则逐项归类。

## 第五部分：当前系统中最需要从主链剥离的职责

### 5.1 当前事实

- 当前接收链显式推动显示、历史、任务触发、SCOE 等多个方向。
- 当前发送链把发送任务、本地工作台、领域目标、统一发送、部分定时与触发语义揉在一起。
- 当前共享状态回读已经让“谁是输入、谁是结果、谁只是缓存或展示”变得模糊。

### 5.2 判断 / 倾向

当前最需要从主链剥离的职责至少有五类。

#### 5.2.1 从接收主链剥离显示与历史编排职责

- 显示域、历史域可以继续消费接收结果。
- 但它们不应再作为接收主链的同步尾部职责存在。
- 尤其不应再通过共享状态周期回读，反向参与接收主链事实定义。

#### 5.2.2 从接收主链剥离任务触发语义本体

- 接收主链可以输出触发候选输入。
- 但不应再直接承担“任务是否进入等待触发、是否满足触发条件、是否立即开始执行”的完整解释权。

#### 5.2.3 从接收主链剥离领域短路本体

- 领域入口可以存在。
- 但领域入口之后的完整识别、执行、完成条件判断，不应继续长期嵌在通用接收主链内部。

#### 5.2.4 从发送主链剥离页面工作台与发送任务语义

- 页面工作台可以继续提供手工入口和监控视图。
- 发送任务可以继续作为某类运行编排能力存在。
- 但二者都不应继续等同于发送主链本身，更不应继续占有中心任务主骨架的解释权。

#### 5.2.5 从通用发送链剥离领域目标特判

- 统一发送层只能承接通用目标差异。
- 领域固定目标、领域记录回写、领域反馈帧、领域任务复用，都应移出通用发送主链。

当前共享状态回读造成的最大失真，也最应该先从主链里剥离：

- 主链本应只基于显式输入决定自己做什么。
- 现在却变成多个分支回看同一共享状态面，再反过来决定主链语义。
- 这会让缓存态、记录态、展示态看起来像主状态，让领域规则看起来像通用规则，让页面工作台看起来像运行主骨架。

### 5.3 待决问题

- 记录态和展示态是否需要在后续文档中进一步拆成“可订阅结果输入”和“纯界面态”两层，本文先不继续细分。
- 某些局部延时、局部重试、局部采样是否算主链局部执行状态，还是应全部外提到任务系统或领域模块，留待 `06`、`07` 在继承本文边界后继续收紧。

## 第六部分：对后续两篇架构文档的约束接口

### 6.1 当前事实

- 如果不先立住收发主链边界，后续 `06` 很容易把任务系统写成“接收触发 + 发送任务”的延长线。
- 如果不先立住收发主链边界，后续 `07` 很容易把 SCOE 写成“直接占据主链内部的长期特判”，或者反过来把它误写成与主链无关的外围工具。

### 6.2 判断 / 倾向

#### 6.2.1 本文如何约束 `06-任务系统归口方式.md`

后续 `06` 必须继承以下前提：

1. 任务系统不是接收主链，也不是发送主链本身。
2. 接收主链只能向任务系统输出明确的触发候选输入或等价运行输入，不能继续直接承担任务编排事实源。
3. 发送主链只能消费任务系统提供的显式发送请求与上下文引用，不能把任务对象本身当成发送主链全部语义。
4. 生命周期控制状态、结果归口状态仍属于统一运行主状态语言，不能被任务本地状态、页面工作台状态或发送局部执行状态替代。
5. 后续若定义任务系统内部状态机，必须建立在本文的“显式输入 / 显式输出”前提上，不能回退到共享状态回读编排。

如果不继承这些前提，`06` 会出现的失真是：

- 把中心任务上下文误收缩成现有发送任务对象。
- 把接收链继续当作任务系统的事实主入口。
- 把发送页工作台和任务监控继续误当作任务系统主骨架。

#### 6.2.2 本文如何约束 `07-SCOE 的架构位置.md`

后续 `07` 必须继承以下前提：

1. SCOE 可以是领域特例，但不能无声明侵入通用接收链和通用发送链。
2. SCOE 的来源特判、固定目标、完成条件、成功反馈、测试工具记录，都首先属于领域规则，不属于通用主链规则。
3. SCOE 若要进入接收主链，必须以显式领域入口方式进入；若要进入发送主链，必须先翻译成显式发送请求，或被标注为边界例外。
4. SCOE 不得通过共享状态回读，反向占有统一运行主状态定义权。
5. SCOE 是否升为一级能力域，可以后续再拍板；但无论层级如何，都不得突破本文给出的主链边界。

如果不继承这些前提，`07` 会出现的失真是：

- 把 SCOE 误写成通用主链内部的一组长期特判。
- 把通用发送链继续写成带有领域目标意识的半通用层。
- 把 SCOE 的领域完成语义误当成统一运行主状态的一般规则。

### 6.3 待决问题

- `06` 最终是否需要在任务系统内部再定义“触发输入层 / 调度层 / 执行层”的分层，本文不提前展开。
- `07` 最终是否把 SCOE 定位为一级能力域还是显式领域模块，本文只给边界约束，不给层级终稿。

## 第七部分：当前结论 / 待决问题 / 下一步建议

### 7.1 当前结论

- 接收主链未来应分成：输入承接层、解析 / 归一层、领域分发层、结果输出层。
- 发送主链未来应分成：发送请求承接层、通用编排 / 归一层、目标落地层、执行结果输出层。
- 接收主链的正式输入是明确入站输入包，正式输出是标准接收结果、运行输入、记录 / 展示订阅输入和领域接管输入。
- 发送主链的正式输入是显式发送请求，正式输出是标准发送结果、运行输入、结果归口输入和领域后续处理输入。
- 接收主链不是统一运行主状态，发送主链不是中心任务上下文本身；二者都只能与统一运行主状态通过显式输入 / 输出发生关系。
- 通用链路与领域特例的真正边界，不在于“能不能复用代码”，而在于“是否必须依赖某一领域自己的来源、目标、状态或完成语义才能成立”。
- 当前最需要先剥离的，是共享状态回读对主链语义的反向塑造，以及领域特例对通用主链的长期沉积。

### 7.2 待决问题

- 接收主链的触发候选输出、发送主链的运行结果输出，后续是否需要在 `04-specs` 中进一步拆出更细的对象类型。
- 任务系统内部到底承接多少调度、时间推进、后台继续运行语义，留待 `06` 继续拍板。
- SCOE 最终是一级能力域还是显式领域模块，留待 `07` 在不突破本文边界的前提下继续拍板。

### 7.3 下一步建议

1. 后续写 `06-任务系统归口方式.md` 时，应直接使用本文的输入 / 输出语言，先回答任务系统如何消费接收主链输出、如何向发送主链发出显式请求。
2. 后续写 `07-SCOE 的架构位置.md` 时，应直接使用本文的“显式领域入口 / 边界例外”语言，回答 SCOE 如何进入收发主链，而不是继续把领域特判糊在通用主链内部。
3. 在进入 `04-specs` 前，应先默认本文已经拍板的不是字段，而是边界；任何字段级设计都不得反向改写这里的主链层次和职责划分。
