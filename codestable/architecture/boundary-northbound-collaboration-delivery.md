# 中心协同接入与对外交付边界

> 状态：首轮正文
> 阶段：03-architecture
> 作用：在继承 `04-运行主状态与状态边界.md` 到 `08-功能清单与目标架构对比（第二轮）.md` 已拍板结论的前提下，只补“中心如何与内部统一运行骨架发生协同”的北向边界，明确中心侧请求由谁承接、内部事实如何投影为对外状态 / 心跳 / 结果 / 报告，以及对外响应与交付语义由谁统一归口。

## 第一部分：本文要解决的北向协同边界问题

### 当前事实

- `04-运行主状态与状态边界.md` 已经拍板：未来内部运行骨架围绕 `子系统运行主体`、`中心任务上下文`、`用例执行上下文`、`生命周期控制状态`、`结果归口状态` 组织。
- `05-接收主链与发送主链组织方式.md` 已经拍板：接收主链和发送主链都只处理显式输入 / 输出，不再通过共享状态回读补齐自身语义。
- `06-任务系统归口方式.md` 已经拍板：任务系统是独立能力域，负责推进任务和用例的正式运行事实，但它不是对外协同边界本身。
- `07-SCOE 的架构位置.md` 已经拍板：领域模块只能通过显式边界接入统一骨架，不能绕过统一运行骨架和任务系统直接定义总状态面。
- `08-功能清单与目标架构对比（第二轮）.md` 已经明确：当前真正还缺的，不是内部骨架，而是“中心协同接入与对外交付边界”。
- [后面对接所需功能清单.md](D:/code/frontend/dongfanghong/后面对接所需功能清单.md) 中真正面对中心侧协同的能力，至少包括：
  - 子系统身份信息
  - 最小设备信息
  - 接收测试任务
  - 启动 / 停止任务
  - 状态查询响应
  - 心跳
  - 用例结果上报
  - JSON 报告生成后的交付
  - 基础响应与错误反馈

### 判断 / 倾向

本文只回答以下问题：

- 中心侧请求 / 查询 / 控制 / 保活 / 结果接收，由哪一层统一承接。
- 内部统一运行骨架如何被投影成对外可见状态、心跳、用例结果和任务报告。
- 用例结果上报与任务报告交付分别属于哪一层职责。
- 对外成功 / 失败 / 拒绝 / 不可执行 / 异常等响应语义由哪一层统一归口。
- 外部协同对象与内部骨架对象之间是什么关系。

本文不回答以下问题：

- 接口字段 schema。
- 心跳报文结构。
- 报告 JSON 字段全集。
- 上传方式、传输协议、重试实现细节。
- 具体接口命名。

先明确：北向协同边界不是谁。

- 不是任务系统本身。
- 不是统一运行主状态本身。
- 不是接收主链或发送主链本身。
- 不是页面工作台。
- 不是某个协议字段模型的别名。

它在当前架构中的作用应收敛为三件事：

- 承接中心侧协同请求。
- 把内部骨架事实投影成对外对象。
- 统一处理对外响应与交付语义。

如果不先补这篇，`04-specs` 最容易出现的失真是：

- 把中心侧请求对象直接写成任务系统内部对象。
- 把状态查询和心跳直接写成某个 Store 的读取结果。
- 把报告交付写成“结果归口状态的附带动作”。
- 把外部响应语义直接冻结在内部生命周期或内部错误对象命名里。

### 待决问题

- “保活”在后续 spec 中最终表现为中心侧轮询、租约刷新，还是我方向中心主动心跳加回执，本文先只拍边界归属，不拍交互形式。
- `子系统身份信息` 与 `最小设备信息` 在后续 spec 中是拆成两组投影对象，还是收敛成一组对外身份视图，本文先不落对象终稿。

## 第二部分：中心协同接入层与对外交付层的定位

### 当前事实

- `04` 到 `07` 已经把内部运行骨架、收发主链、任务系统和领域接入方式依次立住。
- `08` 已经明确：当前悬空的不是内部对象，而是“中心如何与内部骨架发生协同”的外层边界。
- 功能清单中的 `接收测试任务`、`状态查询响应`、`心跳`、`用例执行结果上报`、`报告交付`、`基础响应与错误反馈`，都不是同一个内层对象可以直接吞掉的问题。

### 判断 / 倾向

当前更合适的定位是：

- 应把“北向协同边界”视为一个独立边界层。
- 这个边界层内部再分成两个紧邻子层：
  - `中心协同接入层`
  - `对外交付层`

这两个子层的职责应区分为：

`中心协同接入层`

- 统一承接中心侧进入我方系统的任务请求、控制请求、查询请求和保活协同请求。
- 把这些外部协同动作翻译成内部骨架能够消费的显式输入。
- 统一给出对外同步响应或等价接入回执语义。

`对外交付层`

- 统一消费内部骨架已经落定的运行事实、结果事实和报告对象。
- 把这些内部事实投影成中心侧可见的状态、心跳、用例结果上报和任务报告交付对象。
- 统一给出对外交付回执、交付失败和不可交付语义。

它在总架构中的位置应理解为：

- 位于统一运行骨架之外。
- 位于任务系统、接收主链、发送主链之上，而不是它们之内。
- 它是内部统一骨架与外部中心协同之间的正式边界层。

它与统一运行骨架的关系应理解为：

- 不定义内部运行事实。
- 只向内部骨架输入显式协同输入，或从内部骨架读取已落定事实做投影。

它与任务系统的关系应理解为：

- 接入层负责把“接收任务 / 启动 / 停止”翻译成任务系统正式输入。
- 交付层负责消费任务系统和结果归口侧已经形成的正式输出。
- 任务系统负责内部推进，不负责直接持有外部协同契约。

它与接收主链、发送主链的关系应理解为：

- 接收主链和发送主链仍是内部通用链路。
- 北向协同边界不直接占有接收主链或发送主链的定义权。
- 它只通过任务系统、统一运行骨架或显式交付动作与这两条主链发生关系。

它不能简单并入任务系统或结果归口状态本身，原因至少有四个：

1. 任务系统负责内部运行推进，不负责外部事务语义、外部回执语义和外部拒绝语义。
2. 结果归口状态负责“事实是否已形成、是否已归集、是否已可交付”，不负责真正的对外交付动作。
3. 外部协同对象携带的是中心侧事务语义，内部骨架对象携带的是内部运行语义，二者稳定性来源不同。
4. 如果把两者并在一起，外部协议变化会反向塑造内部运行骨架，重新把 `04` 到 `06` 已拍板的边界打散。

### 待决问题

- 后续 `04-specs` 中，北向协同边界对象组是先按“接入对象组 / 交付对象组”拆写，还是先按“查询 / 控制 / 结果 / 报告”拆写，本文当前倾向是按边界职责拆，不按接口标题拆。
- `中心协同接入层` 是否需要在后续对象组中单列“基础响应语义对象”，本文先只拍归属，不拍对象清单。

## 第三部分：中心侧请求如何进入内部骨架

### 当前事实

- 功能清单已明确：中心侧至少会发生 `接收测试任务`、`启动任务`、`停止任务`、`状态查询`、`心跳 / 保活协同` 这些动作。
- `05` 已经拍板：接收主链和发送主链都只处理显式输入 / 输出，不应再直接承接中心侧协同对象。
- `06` 已经拍板：任务系统才是推进 `中心任务上下文`、`用例执行上下文` 和 `生命周期控制状态` 的正式归口层。

### 判断 / 倾向

中心侧动作应通过接入层中的不同入口统一承接，而不是直接落到某个内部对象上。

建议至少区分四类边界入口：

1. `任务接入入口`
   - 统一承接中心下发的测试任务。
   - 负责把外部任务请求翻译成任务系统可消费的正式任务输入。
   - 任务系统再据此建立或更新 `中心任务上下文` 和 `用例执行上下文`。

2. `控制入口`
   - 统一承接启动任务、停止任务这类控制动作。
   - 负责把外部控制请求翻译成任务系统可消费的控制输入。
   - 任务系统再推进 `生命周期控制状态`。

3. `查询入口`
   - 统一承接状态查询、身份查询、最小设备信息查询这类读取动作。
   - 这类入口不直接推进任务系统。
   - 它们只读取统一运行骨架的对外投影视图。

4. `保活协同入口`
   - 如果中心侧存在注册确认、租约刷新、被动保活检查等动作，应统一由此入口承接。
   - 这类动作原则上不直接等同于任务控制输入。
   - 它们最多影响 `子系统运行主体` 的协同可见性或接入可用性判断。

在这四类入口中，真正直接进入任务系统的请求应是：

- 接收测试任务。
- 启动任务。
- 停止任务。

真正只读取统一运行主状态投影的请求应是：

- 状态查询。
- 子系统身份信息查询。
- 最小设备信息查询。

需要特别说明的保活动作应这样处理：

- 如果是“我方向中心发送心跳”，那不是接入层入口，而是交付层动作。
- 如果是“中心侧发起一次保活协同请求”，才属于接入层保活入口。

这些入口翻译成内部骨架显式输入的关系，应理解为：

- `任务接入入口` 翻译成任务系统正式任务输入。
- `控制入口` 翻译成任务系统正式控制输入。
- `查询入口` 不翻译成运行推进输入，只读取投影视图。
- `保活协同入口` 翻译成北向协同可见性的边界输入，而不是把心跳对象直接塞进任务系统。

不能让中心侧请求对象直接等同于任务系统内部对象，原因至少有五个：

1. 外部请求对象带有 `请求 / 回执 / 拒绝 / 不可执行` 这类外部事务语义，内部任务对象不应被这些语义绑死。
2. 外部请求对象的字段稳定性受中心协议约束，内部任务对象的稳定性受运行骨架约束，两者变化节奏不同。
3. 外部请求可能是部分信息、补充信息或控制信息，内部任务对象必须是可稳定推进的运行事实。
4. 如果直接等同，中心协议命名会反向污染 `中心任务上下文` 和 `用例执行上下文` 的内部语义。
5. 这会让 `04-specs` 把“外部事务对象”和“内部运行对象”写成一组混合对象，后续极难收敛。

### 待决问题

- `接收测试任务` 与 `启动任务` 在后续 spec 中最终是两个独立外部请求，还是允许“任务接收即带启动语义”的组合形式，本文先不拍外部事务组合规则。
- 保活协同入口是否需要在后续 spec 中单列“注册 / 保活 / 续约”对象组，本文先只拍归属，不拍交互细分。

## 第四部分：内部骨架如何向中心侧投影状态、结果与报告

### 当前事实

- `04` 已经拍板：真正稳定的内部事实来源，只能是 `子系统运行主体`、`中心任务上下文`、`用例执行上下文`、`生命周期控制状态`、`结果归口状态`。
- 功能清单要求中心侧看到的，不是这些内部对象原样，而是：
  - 子系统身份信息
  - 最小设备信息
  - 状态查询响应
  - 心跳
  - 用例执行结果
  - 任务级结果汇总
  - JSON 报告

### 判断 / 倾向

北向交付对象应统一理解为“内部事实的投影视图”，而不是“内部对象原样外露”。

建议采用以下投影关系：

`子系统身份信息`

- 主要从 `子系统运行主体` 上的稳定身份事实投影出来。
- 它表达的是“我是谁”，不是“当前任务是什么”。
- 它不应从页面工作台、测试工具状态或任务系统局部对象中拼出来。

`最小设备信息`

- 应从与 `子系统运行主体` 相邻的最小设备事实投影出来。
- 它属于对外身份视图的附属投影，而不是任务系统内部对象的一部分。
- 它不应被扩大成完整设备管理对象。

`状态查询响应`

- 应从 `子系统运行主体`、`中心任务上下文` 摘要、`用例执行上下文` 摘要和 `生命周期控制状态` 投影出来。
- 查询对象看到的是对外状态面，而不是内部完整上下文。
- 它可以压缩、归纳内部状态，但不能发明新的内部生命周期语义。

`心跳`

- 应优先从 `子系统运行主体` 的最小运行事实投影出来。
- 必要时附带最小的当前执行摘要。
- 它不应等于完整任务状态对象的周期性直出。

`用例执行结果`

- 应从 `用例执行上下文` 已落定的结果事实和 `结果归口状态` 的用例级归集结果投影出来。
- 换句话说，对外上报的是“已归口的用例结果”，不是运行中临时态。

`任务级结果汇总`

- 应从 `中心任务上下文` 与 `结果归口状态` 的任务级聚合事实投影出来。
- 它回答的是“整个任务最终形成了什么结果”，不是“内部曾经经历过哪些局部过程状态”。

`报告对象`

- 应从任务级结果汇总事实进一步形成。
- 它是内部结果事实对外可交付的一种稳定表达。
- 它不是交付动作本身。

投影层与内部对象之间的关系，应明确为：

- 投影对象是内部事实的单向映射。
- 投影对象可以裁剪、重命名、归纳、压缩内部事实。
- 投影对象不拥有反向定义内部骨架的权力。

为什么投影对象不能反向塑造内部对象语义，原因至少有四个：

1. 中心侧看到的是协同视角，内部骨架维护的是运行视角，两者粒度并不天然一致。
2. 对外状态和对外结果通常需要压缩、摘要和稳定命名，内部对象则需要保留足够的运行语义。
3. 如果由投影对象反向塑造内部对象，未来任何协议命名变化都会倒逼内部骨架重命名或改层级。
4. 这会让“外部可见对象”重新变成“内部事实源”，直接违反 `04` 已拍板的主状态纪律。

### 待决问题

- 报告对象在后续 spec 中是单列一个稳定对象组，还是建立在“任务级结果汇总对象 + 对外报告视图”两层之上，本文先不拍对象层次。
- 心跳是否只带 `子系统运行主体` 的最小摘要，还是允许附带最小任务摘要，本文先只拍投影来源，不拍投影粒度。

## 第五部分：对外响应与错误语义的统一归口

### 当前事实

- 功能清单已经明确：中心协同链路必须统一处理成功、失败、不可执行、参数不完整、当前状态不允许等基础反馈。
- `06` 已经拍板：内部生命周期控制和结果归口应由任务系统推进，而不是由页面或局部执行对象解释。
- `08` 已经指出：如果不先补对外协同边界，`04-specs` 很容易把外部响应 / 错误对象直接糊进内部对象命名。

### 判断 / 倾向

对外成功 / 失败 / 拒绝 / 不可执行 / 异常等语义，应统一归口在北向协同边界层，而不是分别散落在任务系统、发送主链或报告生成侧。

更细一点说：

- `中心协同接入层` 统一归口接入请求的响应语义。
- `对外交付层` 统一归口用例结果上报、心跳、报告交付等交付动作的回执语义。

内部事实向对外语义的转换关系，应按下面的原则理解：

`任务拒绝`

- 这是任务系统或接入前置校验给出的内部判断。
- 对外应统一转换成“拒绝”或“不可执行”一类语义。
- 不应直接把内部校验对象原样暴露出去。

`生命周期失败 / 执行异常`

- 这是任务系统在推进过程中形成的内部失败事实。
- 对外应统一转换成“失败”或“异常”语义。
- 不应把内部状态机细枝末节直接暴露给中心侧。

`发送失败`

- 这是发送主链或交付动作中的内部失败事实。
- 如果它影响任务或用例结果，应先回到任务系统和结果归口侧落定结果，再由边界层转换成对外语义。
- 如果它只影响某次交付动作，则应由交付层统一转换成“交付失败”或“不可交付”语义。

`报告不可交付`

- 这不是“报告对象不存在”，而是“报告对象已形成，但交付动作未成功”。
- 对外应归口为交付层的交付失败语义，而不是反向改写内部结果对象。

不能直接把内部错误对象原样暴露成对外错误对象，原因至少有五个：

1. 内部错误对象服务于排错和运行推进，天然带有实现细节。
2. 内部错误分类往往比对外错误语义更细，原样外露会让外部契约不稳定。
3. 多个内部错误可能应归并成同一种对外语义，反过来也成立。
4. 原样暴露会泄露内部边界、内部模块名和局部执行细节。
5. 这会让外部协同契约依赖内部实现结构，后续很难演进。

### 待决问题

- “拒绝”与“不可执行”在后续 spec 中最终是否拆成两个稳定对外语义类别，本文当前倾向是对外保留二者区分，但先不拍字段级差异。
- 交付层是否需要单列“回执对象”和“交付失败对象”两类外部对象，本文先只拍语义归口，不拍对象形态。

## 第六部分：用例结果上报与任务报告交付的边界划分

### 当前事实

- 功能清单已明确：每个用例完成后需要上报结果，整个任务完成后需要形成 JSON 报告并交付给中心。
- `04` 已拍板：`结果归口状态` 是运行骨架对象，用于表达结果是否已经形成、归集、可交付。
- `06` 已拍板：任务系统负责推进任务和用例结果事实，但不负责直接占有对外交付边界。

### 判断 / 倾向

用例结果上报、任务级汇总、报告生成、报告交付，不应被写成同一个边界动作，而应明确拆开。

`用例结果上报属于哪一层`

- 属于 `对外交付层`。
- 它消费的是 `结果归口状态` 中已经落定的用例级结果事实。
- 它不是任务系统内部推进动作本身。

`任务级结果汇总和报告生成属于哪一层`

- 属于内部结果归口侧的稳定事实形成过程。
- 更准确地说，它建立在 `中心任务上下文` 和 `结果归口状态` 之上。
- 它回答的是“任务最终形成了什么可交付结果”和“报告对象是什么”，而不是“如何把它送出去”。

`报告交付属于哪一层`

- 属于 `对外交付层`。
- 它消费已经形成的报告对象，执行真正的对外交付动作。
- 它处理的是交付成功、交付失败、不可交付、回执未达等交付语义，而不是重新定义报告内容。

应把“内部结果归口事实”和“对外交付动作”明确区分为两组：

内部结果归口事实：

- 用例结果已落定。
- 任务结果已汇总。
- 报告对象已形成。

对外交付动作：

- 用例结果已上报或上报失败。
- 心跳已送达或送达失败。
- 报告已交付、交付失败或暂不可交付。

为什么“报告对象”与“报告交付”不能混成一个边界，原因至少有四个：

1. 报告对象是稳定事实，报告交付是外部副作用。
2. 报告对象可能已形成，但交付动作可以失败、延后或重试。
3. 报告对象需要服务内部归档、追溯和重新交付，不能跟某次交付动作绑定成一次性对象。
4. 如果混在一起，`04-specs` 会把“报告是什么”与“报告怎么送”写成一类对象，边界会再次失稳。

为什么不能把对外交付简单当成结果归口状态的附带动作，原因至少有三点：

1. `结果归口状态` 负责结果事实闭合，不负责实际网络交付、回执和交付失败语义。
2. 一次结果事实可能对应多次交付尝试，附带动作写法会把事实状态和副作用历史混在一起。
3. 交付层还承担心跳、查询响应、基础回执等职责，本来就不是 `结果归口状态` 的附属模块。

### 待决问题

- 报告对象是否只在“任务自然完成”后形成，还是在“已停止 / 异常中止”时也形成受限报告对象，本文先不拍结论，只先拍边界。
- 用例结果上报与报告交付后续是否复用同一套交付基础设施，本文先不拍实现组织方式。

## 第七部分：补完本文后，`04-specs` 应如何切入

### 当前事实

- `08-功能清单与目标架构对比（第二轮）.md` 已经建议：在内部骨架稳定后，应先补一个窄范围的“中心协同接入与对外交付边界”专题，再进入北向协同 spec。
- 当前这篇补完后，`03-architecture` 中：
  - 内部统一运行骨架已拍板
  - 收发主链边界已拍板
  - 任务系统归口已拍板
  - 领域模块接入方式已拍板
  - 北向协同接入与对外交付边界已补齐

### 判断 / 倾向

补完本文后，当前判断是：`03-architecture` 已经足以进入 `04-specs`。

但进入方式应按稳定边界切，而不是按接口标题乱切。

当前更稳定的切法应至少分成五组：

1. `统一运行骨架对象组`
   - `子系统运行主体`
   - `中心任务上下文`
   - `用例执行上下文`
   - `生命周期控制状态`
   - `结果归口状态`

2. `任务系统输入 / 推进对象组`
   - 任务接入输入
   - 任务控制输入
   - 用例与参数承载
   - 任务推进结果

3. `收发主链边界对象组`
   - 接收主链输出给任务系统的正式输入对象
   - 任务系统发给发送主链的显式发送请求对象
   - 发送主链返回任务系统或结果归口侧的标准结果对象

4. `北向协同接入对象组`
   - 任务接入请求 / 响应对象
   - 启动 / 停止控制请求 / 响应对象
   - 查询请求 / 响应对象
   - 基础响应与错误语义对象

5. `北向投影与对外交付对象组`
   - 身份 / 最小设备信息投影对象
   - 状态查询投影对象
   - 心跳投影对象
   - 用例结果上报对象
   - 任务级结果汇总对象
   - 报告对象
   - 报告交付对象

第一批 spec 最适合先写的，仍然应是前三组：

- `统一运行骨架对象组`
- `任务系统输入 / 推进对象组`
- `收发主链边界对象组`

原因不是北向对象不重要，而是：

- 北向对象组现在已经有稳定边界，但它们仍建立在前三组对象之上。
- 先把前三组收紧，北向对象组就不会回退成“拿内部对象直接外露”。

补完本文后，第四组和第五组也已经可以稳定进入 `04-specs`，但不应先写以下内容：

- 心跳报文字段细节。
- 报告 JSON 全字段结构。
- 上传协议、传输协议、重试流程。
- 具体接口命名。
- 详细错误码字典。

如果不补本文就直接进入 `04-specs`，会出现的失真至少包括：

1. 把中心侧请求对象直接写成任务系统内部对象。
2. 把状态查询和心跳对象直接写成共享状态或页面状态的对外别名。
3. 把用例结果上报和报告交付写成“结果对象顺手附带一个动作”。
4. 把对外成功 / 失败 / 拒绝 / 不可执行语义提前冻结在内部生命周期命名里。

### 待决问题

- 北向对象组在具体落 spec 时，是先写“接入对象组”还是先写“投影与交付对象组”，本文当前倾向是先写接入对象组，因为它更直接约束外部请求对象与内部对象的解耦。
- 如果报告交付方式迟迟未定，后续 spec 是否先只写“报告对象”，暂缓“报告交付对象”，本文当前倾向是允许分批推进。

## 第八部分：当前结论 / 待决问题 / 下一步建议

### 当前事实

- 内部骨架已经拍板，当前悬空点只剩北向协同边界。
- 功能清单中的身份、查询、保活、结果上报、报告交付和基础响应，都依赖这层边界。
- 当前最大的架构风险，仍然是把外部协同对象和内部骨架对象直接写成同一个对象。

### 判断 / 倾向

当前可以收敛出八个结论：

1. 中心协同请求应由独立的 `北向协同边界层` 统一承接，而不是直接落到任务系统、统一运行主状态或收发主链上。
2. `北向协同边界层` 内部应分成 `中心协同接入层` 和 `对外交付层` 两个紧邻子层。
3. 接收测试任务、启动任务、停止任务这三类请求，应由接入层翻译成任务系统正式输入；状态查询和身份 / 设备查询，应只读取统一运行骨架投影。
4. 内部统一运行骨架应通过投影层向中心侧表达身份、最小设备信息、状态、心跳、用例结果、任务结果和报告对象，而不是把内部对象原样外露。
5. 对外成功 / 失败 / 拒绝 / 不可执行 / 异常等语义，应由北向协同边界层统一归口；内部错误对象不能原样外露。
6. 用例结果上报属于 `对外交付层`；任务级结果汇总和报告对象形成属于内部结果归口侧；报告交付属于 `对外交付层`。
7. “报告对象”与“报告交付”必须分开，“结果归口事实”与“对外交付动作”也必须分开；不能再把对外交付简单当成结果归口状态的附带动作。
8. 补完本文后，`04-specs` 已经可以按稳定边界切入；如果不补本文，spec 会在接入对象、投影对象、交付对象和响应语义上持续失真。

### 待决问题

- 卡在 `接入层` 的问题：
  - 保活协同最终采用什么交互形式。
  - 启动任务与接收任务是否允许在一个外部事务中合并表达。
  - 基础响应语义是否拆成独立对象组。

- 卡在 `交付层` 的问题：
  - 报告最终采用哪种交付方式。
  - 报告交付失败后的回执口径如何表达。
  - 用例结果上报与报告交付是否共享同一交付基础设施。

- 卡在 `投影层` 的问题：
  - 身份信息与最小设备信息的对象拆分方式。
  - 状态查询与心跳的最小投影粒度。

### 下一步建议

1. 接受本文作为 `03-architecture` 的补口专题，不再回头重开 `04` 到 `08` 的内部争论。
2. 进入 `04-specs` 时，先写 `统一运行骨架对象组`、`任务系统输入 / 推进对象组`、`收发主链边界对象组`，随后再写 `北向协同接入对象组` 与 `北向投影与对外交付对象组`。
3. 在进入北向对象 spec 时，始终坚持一条边界纪律：
   - 外部协同对象不等于内部骨架对象。
   - 投影对象不反向定义内部对象。
   - 交付动作不混入结果归口事实。
