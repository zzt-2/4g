# D### Decision Record Template

每个专题的 `decisions.md` 中，每条 D### 条目使用以下格式。

## 标准格式

```markdown
## D###: [决策标题]

> status: active
> date: YYYY-MM-DD
> 取代：无 / D###
> 被取代：无 / D###

### 决策

[一句话]

### 理由

[为什么做这个决策]

### 排除的替代方案

[考虑过但没选的，以及为什么没选]

### 影响范围

[影响哪些文件、步骤、契约、其他专题]

### 来源

[S### / 用户纠正 / 验证数据]
```

## 字段说明

- **status**: 默认 `active`。当被后续决策替代时改为 `superseded`。永远不删除。
- **取代**: 本决策替代了哪个旧 D###。填写时同步更新旧 D### 的"被取代"字段。
- **被取代**: 当新决策在"取代"中引用本决策时，自动填写。
- **排除的替代方案**: 每条决策必须记录没选什么、为什么没选。防止后续重复探索已否决的方向。
- **来源**: 追踪决策从哪来。"S###"表示 session 推导，"用户纠正"表示用户纠正，"验证数据"表示数据驱动。

## 失败路线记录

当一条路线被验证为失败时，使用以下扩展格式：

```markdown
## D###: [路线名称]失败记录

> status: active
> date: YYYY-MM-DD
> 取代：无
> 被取代：无

### 决策

[路线名称]验证失败，否决该方向。

### 核心失败机制

[根因，不只是"没跑通"。例如："LLM 自由输出文本时会压缩/省略/编造细节"]

### 否决了什么

[哪些做法因此被排除。例如："否决了'LLM 直接输出改后文本'路线"]

### 可复用部分

[这条路线中哪些部分可以复用。例如："章节拆分逻辑可复用"]

### 具体数据

[关键失败数据点。例如：V1 丢了 25 个数字、V3 丢失 11 个数字和 6/13 引用]

### 影响范围

[影响哪些后续步骤和方案选择]

### 来源

[S### / V###]
```

## 血缘链维护

当新决策替代旧决策时：

1. 新 D### 的"取代"字段填写旧 D### 编号
2. 旧 D### 的"被取代"字段填写新 D### 编号
3. 旧 D### 的 status 改为 `superseded`
4. 不要删除旧 D###，保留可追溯

当新方案重复使用已被否决的做法时：
- 治理 skill 应检测到与失败路线 D### 的冲突
- **阻断**并要求显式论证"为什么这次不同"
